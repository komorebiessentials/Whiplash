API Design

  

    
POST   /api/v1/pastes
// Create a new paste
Body: {
  content:    "def hello(): print('Hello World')",
  language:   "python",       // for syntax highlighting
  expiresAt:  "2026-07-21",   // optional
  password:   "secret123",    // optional
  isPublic:   true
}
→ 201 Created { pasteId: "xK3mP9", url: "https://paste.io/xK3mP9" }

  

  

    
GET    /api/v1/pastes/{pasteId}
// Returns metadata; content URL points to CDN/S3
→ { pasteId, title, language, contentUrl, size, viewCount,
    createdAt, expiresAt, isPublic, hasPassword }

GET    /api/v1/pastes/{pasteId}/raw       // Raw text content
DELETE /api/v1/pastes/{pasteId}           // Requires owner JWT
GET    /api/v1/pastes?userId=&page=       // User's paste history

  







  
Section 06

  
Database Design

  

    
pastes (PostgreSQL) — metadata only

    

      paste_id     VARCHAR(8) PRIMARY KEY — Base62 short code

      owner_id     UUID — nullable (anonymous pastes)

      title        TEXT

      language     TEXT — "python", "java", "text"

      content_url  TEXT — S3 key: "pastes/xK3mP9"

      size_bytes   INTEGER

      password_hash TEXT — bcrypt, nullable

      is_public    BOOLEAN

      view_count   INTEGER

      created_at   TIMESTAMP

      expires_at   TIMESTAMP — nullable
    

    
Content is NOT stored in DB — only the S3 key. DB stays lean (<1 KB per row). 10M pastes = <10 GB for metadata table.

  

  

    
Content Storage (S3 Object Store)

    

      Key: pastes/{paste_id}

      Value: raw paste content (bytes)

      Content-Type: text/plain; charset=utf-8

      Metadata: language, created_at (in S3 object metadata)
    

    
Object store scales to petabytes. S3 standard: 99.999999999% (11 nines) durability. No size limit per object (up to 5 TB; our limit is 10 MB). Immutable after creation.

  

  

    
users (PostgreSQL)

    

      user_id      UUID PRIMARY KEY

      username     TEXT UNIQUE

      email        TEXT UNIQUE

      password_hash TEXT

      created_at   TIMESTAMP
    

  







  
Section 07

  
Key Design Decisions

  

    
Unique ID Generation — Three Options

    

      

        

          
Option 1: Random Base62

          
Generate random 6-char Base62 string. Check DB for collision. ~115 writes/sec makes collision probability negligible (birthday problem at 56B space). Simple.

        

        

          
Option 2: Hash-Based

          
MD5(content + userId + timestamp) → first 6 chars → Base62. Deterministic: same content from same user → same ID (natural dedup). Risk: collisions on different content.

        

        

          
Option 3: Pre-Generated Pool

          
Background job pre-generates IDs in batches, stores in available_ids table. Service pops one on create. Zero collision risk. Slightly more complex.

        

      

      
Recommendation: Option 1 (random) for simplicity — at 115 writes/sec with 62^6 space, collision is astronomically rare. Retry on collision (once is enough).

    

  

  

    
Content in S3, Metadata in PostgreSQL

    

      The most important architectural decision: store paste content in S3 (object store), not in the database. The DB stores only the S3 key as a pointer.


      Why: text content is variable-size (up to 10 MB), stored forever, rarely updated. Databases are optimized for structured queries, not large blob storage. S3 scales to exabytes; PostgreSQL with 10 MB rows does not.


      Result: DB stays lean (metadata queries stay fast), S3 provides 11-9s durability for the actual content, CDN can serve directly from S3 edge without touching the app server.
    

  

  

    
Expiration Handling

    

      Two approaches:
      
    
        
  • Lazy deletion: On every read, check expires_at <= NOW(). If expired, return 404 and mark for cleanup. Zero background job needed, but expired data lingers in storage.
    • 
        
  • Proactive cron: Daily job runs SELECT paste_id FROM pastes WHERE expires_at <= NOW(), deletes from S3 and DB in batches. Frees storage promptly.
    • 
      

      Use both: lazy check on read for correctness, cron job for storage reclamation. Cron uses expires_at index for efficient query.
    

  

  

    
View Count — Don't Block on Write

    

      Naive approach: UPDATE pastes SET view_count = view_count + 1 on every read → DB write on the hot read path (bad).


      Better: increment a Redis counter INCR view:{pasteId} on read. A background job periodically flushes Redis counters to PostgreSQL in bulk. Eventual consistency is fine for view counts.

Section 09

  
Self-Check: Interview Questions

  

    

      Why store content in S3 instead of PostgreSQL? Storage
      +
    

    
PostgreSQL is a relational database optimized for structured data, joins, and transactions on fixed-schema rows. A paste can be up to 10 MB — storing 10 MB TEXT blobs in PostgreSQL rows creates several problems: (1) table rows become massive, slowing down scans and index reads; (2) PostgreSQL buffers the blob in memory on read, crowding out index data; (3) replication lag increases because 10 MB rows flow through WAL (write-ahead log). Object stores (S3, GCS) are designed exactly for large immutable blobs: they stream content directly to the client, have 11-9s durability via erasure coding, and scale to exabytes at low cost (~$0.023/GB/month). The DB stores only a tiny S3 key pointer, staying lean and fast for metadata queries.

  

  

    

      How do you handle a paste going viral (1M views in 1 hour)? Scale
      +
    

    
A viral paste is exactly the CDN use case. Paste content is immutable — the same bytes are served to all readers. Public pastes get Cache-Control: public, max-age=86400 (or longer for "never expires"). CDN serves all 1M reads from edge nodes nearest to users. The S3 origin and app servers see essentially zero traffic after the CDN warms up.

For metadata (title, view count, language): Redis caches paste metadata with TTL=1 hour. Cache hit rate for a viral paste is ~100% — all 1M readers hit Redis, not PostgreSQL. View count: INCR view:{pasteId} in Redis (atomic). Background job flushes to DB hourly. The eventual view count drift is acceptable — nobody cares if it shows 999,843 vs 1,000,000 in real-time.

  

  

    

      What's the collision probability with random Base62 IDs? ID Generation
      +
    

    
Key space: 62^6 = 56,800,235,584 (~56 billion). With 10M DAU creating 1 paste each = 10M pastes/day. The birthday problem: probability of at least one collision with n items in space N is approximately n²/(2N). For n=10M (daily creates) and N=56B: P ≈ (10^7)² / (2 × 5.68×10^10) ≈ 10^14 / 1.14×10^11 ≈ 877. Wait — that would be >1, meaning collisions are guaranteed! But the space gets consumed across all time, not just one day. After 5 years at 10M/day: 18B pastes (assuming no expiry). At 18B/56B = 32% space used. Then daily collision probability: (10M × 18B) / (2 × 56B) ≈ 1.6 per day. So yes, collisions do happen occasionally — the DB unique constraint catches them and we retry with a new random ID. Just retry once.

  

  

    

      How do you implement password-protected pastes? Security
      +
    

    
At create time: bcrypt-hash the password with cost factor 12. Store password_hash in DB. Do NOT send raw password to S3 — the S3 object stores only plain content.

At read time: if password_hash IS NOT NULL in metadata, return HTTP 401 with {requiresPassword: true}. Client shows password prompt. POST /pastes/{id}/unlock {password} → server bcrypt.compare(password, password_hash) → if match, return a short-lived signed token (JWT with pasteId + exp=10min). Subsequent content requests include this token.

CDN caching: set Cache-Control: private, no-store for password-protected pastes. CDN must not cache them — each user must go through the unlock flow. This slightly increases load for protected pastes, but volume is low enough to not matter.

  

  

    

      How do anonymous pastes work — what if the creator wants to edit later? Auth
      +
    

    
Anonymous pastes are extremely common (no login required). The system issues an edit token — a random 128-bit value — returned to the client in the creation response. The client stores this in browser localStorage. The DB stores edit_token_hash = bcrypt(editToken) with the paste row.

To delete or edit: client sends DELETE /pastes/{id} with X-Edit-Token: {editToken}. Server looks up the paste, bcrypt.compare(editToken, edit_token_hash) → if match, allow the operation. This is essentially a capability-based auth: whoever has the edit token can modify the paste. Risk: if user clears localStorage, they lose the ability to edit — no recovery. This is the actual behavior of Pastebin: "Save this link if you want to edit later."

  







  
Section 10

  
Scaling & Bottlenecks

  

    

      
CDN for Reads

      
Paste content is immutable after creation. Public pastes can be cached at CDN with infinite TTL (until deleted). A viral code snippet shared on Twitter handles 100K reads/sec from CDN edge.

    

    

      
Direct S3 Upload

      
For large pastes: issue a presigned S3 PUT URL to client → client uploads directly to S3 (bypasses app server). App server only stores the S3 key in DB after confirmation.

    

    

      
Read Replicas

      
Paste reads are 10x more frequent than writes. PostgreSQL read replicas handle metadata reads. Primary only for creates, updates (view count flush), deletes.

    

    

      
Redis Metadata Cache

      
Cache paste metadata in Redis: paste:{id} → JSON with TTL 1 hour. Cache miss → hit PostgreSQL → populate cache. 80% of reads hit hot pastes.

    

    

      
Soft Delete Pattern

      
Mark deleted in DB immediately (user sees success). Async cleanup job removes from S3. Decouples fast user-facing response from slower storage deletion.

    

    

      
Password-Protected Pastes

      
Check password_hash on read. Return 403 if wrong password. CDN cannot cache password-protected pastes (vary by auth). Set Cache-Control: private for these.

    

  







  
Section 11

  
Interview Tips

  
📦
Object store is the key insight: When the interviewer asks where you store the content, say S3 (object store), not a DB column. Explain why: DB is for structured queries, object store is for blobs. This single decision prevents the biggest scalability mistake.

  
🔗
Pastebin = URL Shortener + File Storage: Recognize the overlap with URL shortener (short ID generation is identical). If you've discussed URL shortener first, reference it — interviewers like system thinking that connects prior designs.

  
🌐
CDN is decisive: A paste shared on social media can get millions of views. Without CDN, your app servers and S3 take the full hit. With CDN, the content is served from edge — ~0ms latency, ~$0 origin cost per view.

  
⏰
Expiration — lazy vs proactive: Interviewers often ask "how do you handle expired pastes?" Have both answers: lazy check on read (correctness) and cron cleanup (storage reclamation). Explain why you need both.

  
📊
Anonymous vs authenticated pastes: Many pastes are created without logging in. owner_id is nullable. Anonymous paste creators get an edit token in the response — stored in their browser localStorage. This is how Pastebin actually works.
