What Is It

Event-Driven Architecture (EDA) is a design paradigm where components communicate by producing and consuming events — immutable facts about things that happened — rather than making direct synchronous calls. Three patterns build on this foundation:

Event Sourcing

Store all changes as a sequence of events rather than current state. Rebuild state by replaying the event log.

CQRS

Separate the write model (commands → events) from the read model (optimised query projections).

Saga Pattern

Manage distributed transactions across microservices using compensating transactions instead of 2PC.

Outbox Pattern

Atomically write to a DB and publish an event by using an outbox table and a relay process.

The Problem It Solves

Tight coupling: In a monolith, when you complete an order you might call inventory.reserve(), payment.charge(), notification.send() synchronously. If any one fails, the whole operation is at risk and all services are coupled together.

Distributed transaction impossibility: Spanning a business transaction across multiple microservices with ACID guarantees requires 2PC, which blocks all participants and is a single point of failure.

Audit and temporal queries: Traditional state-based storage overwrites data. You can't answer "what was the account balance 3 months ago?" without separate audit tables.

EDA solves these by making communication asynchronous, decoupled, and durable through an event log.

How It Works

The core flow: a Command arrives → the Command Handler validates it and emits one or more Domain Events to the Event Store → events are published to the Event Bus → multiple subscribers react independently: a Projection updates the read model, a Saga orchestrates downstream steps, a Notification service sends emails. No subscriber blocks the others.

Patterns in Depth

Event Sourcing

Instead of storing the current state of an entity (e.g., balance = 100), store the sequence of events that produced it: [Deposited(200), Withdrew(100)]. To read current state, replay the events. Advantages: full audit trail, time-travel queries (replay up to timestamp T), ability to create new projections from history. Disadvantages: querying current state requires projections; event schema evolution is painful.

CQRS (Command Query Responsibility Segregation)

The write model handles commands and emits events to the event store. The read model maintains separate, denormalised views (projections) optimised for specific queries — updated asynchronously by consuming events. This separates write consistency (the command side) from read performance (the query side). Can be used without Event Sourcing, but they complement each other well.

Saga Pattern

When a business transaction spans multiple services, avoid 2PC (which locks all participants and is fragile). Instead, each step emits an event; if a step fails, compensating transactions are run in reverse order.

Choreography Saga

Each service listens for events and reacts. No central coordinator. Decoupled but hard to visualise the overall flow and diagnose failures.

Orchestration Saga

A central Saga Orchestrator sends commands, awaits replies, and decides next steps or compensations. Easier to understand and debug; single coordinator is a design-time dependency.

Example: Order saga — CreateOrder → ReserveInventory → ChargePayment → ShipOrder. If ChargePayment fails → ReleaseInventory → CancelOrder.

Outbox Pattern

The dual-write problem: you can't atomically write to a DB and publish to Kafka in the same operation (they're separate systems). Solution: write the event to an outbox table in the same DB transaction as your domain write → a relay process (or Debezium CDC) reads the outbox and publishes to Kafka → marks the row as sent. Guarantees at-least-once event delivery without 2PC.

// Event Sourcing — BankAccount
sealed interface AccountEvent {}
record MoneyDeposited(long amount) implements AccountEvent {}
record MoneyWithdrawn(long amount) implements AccountEvent {}
record AccountOpened(String owner) implements AccountEvent {}

class BankAccount {
    private List<AccountEvent> events = new ArrayList<>();
    private long balance = 0;

    void deposit(long amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        apply(new MoneyDeposited(amount));
    }

    void withdraw(long amount) {
        if (amount > balance) throw new InsufficientFundsException();
        apply(new MoneyWithdrawn(amount));
    }

    private void apply(AccountEvent e) {
        events.add(e); // append to event store
        switch (e) {
            case MoneyDeposited d  -> balance += d.amount();
            case MoneyWithdrawn w  -> balance -= w.amount();
            case AccountOpened  o  -> {} // no state change on open
        }
    }

    // Rebuild from stored events (replay)
    static BankAccount rebuild(List<AccountEvent> history) {
        var account = new BankAccount();
        history.forEach(account::apply);
        return account;
    }
}
// CQRS — Separate write model and read model

// WRITE side: command handler emits events
class OrderCommandService {
    private final EventStore eventStore;

    OrderCreated createOrder(CreateOrderCommand cmd) {
        var event = new OrderCreated(cmd.orderId(), cmd.items(), Instant.now());
        eventStore.append("order-" + cmd.orderId(), event);
        return event;
    }
}

// READ side: query denormalised projection
class OrderQueryService {
    private final OrderReadRepository readRepo; // separate DB/table

    OrderSummary getOrder(String orderId) {
        return readRepo.findById(orderId); // fast denormalised read
    }
}

// Projection updater: subscribes to events, updates read DB
class OrderProjectionUpdater {
    @KafkaListener(topics = "order-events")
    void on(OrderCreated event) {
        var summary = new OrderSummary(
            event.orderId(), event.items(), "PENDING", event.createdAt()
        );
        readRepo.save(summary); // update read model asynchronously
    }
}
// Saga Orchestrator — Order fulfillment
class OrderSaga {
    private SagaState state = SagaState.STARTED;

    void handle(OrderCreated event) {
        state = SagaState.RESERVING_INVENTORY;
        inventoryService.reserve(event.items()); // send command
    }

    void handle(InventoryReserved event) {
        state = SagaState.CHARGING_PAYMENT;
        paymentService.charge(event.orderId(), event.totalAmount());
    }

    void handle(PaymentCharged event) {
        state = SagaState.SHIPPING;
        shippingService.ship(event.orderId());
    }

    // Compensation: payment failed → release inventory → cancel order
    void handle(PaymentFailed event) {
        state = SagaState.COMPENSATING;
        inventoryService.release(event.orderId()); // compensating txn
        orderService.cancel(event.orderId());
        state = SagaState.CANCELLED;
    }
}
// Outbox Pattern — atomic DB write + event publish
@Transactional
void createOrder(Order order) {
    orderRepository.save(order); // domain write

    // Write event to SAME transaction
    var outboxEvent = OutboxEvent.builder()
        .id(UUID.randomUUID())
        .aggregateType("Order")
        .aggregateId(order.getId())
        .eventType("OrderCreated")
        .payload(toJson(order))
        .status(OutboxStatus.PENDING)
        .build();
    outboxRepository.save(outboxEvent); // same DB, same txn
} // COMMIT — both order and outbox row committed atomically

// Relay runs separately (or Debezium CDC reads outbox table)
@Scheduled(fixedDelay = 1000)
void relayEvents() {
    var pending = outboxRepository.findByStatus(OutboxStatus.PENDING);
    for (var event : pending) {
        kafkaProducer.send(event.getEventType(), event.getPayload());
        outboxRepository.markSent(event.getId()); // at-least-once
    }
}

Real-World Usage

Axon Framework

Java framework that implements Event Sourcing + CQRS + Saga out of the box. Used by enterprises with complex domain models.

Temporal.io

Workflow orchestration engine built on Event Sourcing principles. Used by Netflix, Stripe, Airbnb for reliable distributed workflows (replaces hand-rolled sagas).

Debezium

CDC (Change Data Capture) tool that reads the DB write-ahead log and publishes changes to Kafka — the production implementation of the Outbox pattern.

Kafka

The event store / event bus for most production EDA systems. Immutable append-only log, replay capability, consumer groups for independent projections.

EventStoreDB

Purpose-built event store database. Supports subscriptions, projections, and optimistic concurrency for event streams.

Martin Fowler's Patterns

CQRS and Event Sourcing patterns were popularised by Martin Fowler and Greg Young. The original articles are required reading for senior engineers.

Interview Tips

"When would you use Event Sourcing?"

Audit-critical domains (banking, legal), systems needing temporal queries, complex domains with many event types. NOT for simple CRUD — the overhead is not worth it.

"What's the difference between Saga and 2PC?"

2PC blocks all participants during the protocol and has a coordinator SPOF. Saga is eventual consistency via compensation — no locking, each step is a local transaction.

"How do you handle event schema evolution?"

Version your events (v1, v2). Use upcasters to transform old schema to new on read. Never delete fields — add new optional fields only.

"What is the dual-write problem?"

Writing to DB and Kafka are two separate operations — if the app crashes between them, you get inconsistency. Solution: Outbox pattern or Kafka transactions with exactly-once semantics.

"How do you handle failed Saga steps?"

Each step must be idempotent (safe to retry) and have a compensating transaction. The orchestrator tracks state — on failure, runs compensations in reverse order.

"Can you use CQRS without Event Sourcing?"

Yes. You can have a standard write model that updates a DB and also updates a separate read-optimised projection (e.g., Redis or Elasticsearch) asynchronously. ES is not required.
