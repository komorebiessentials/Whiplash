1. Requirements

Functional

  • Limit requests per client (by user ID, IP, or API key)
  • Return HTTP 429 with Retry-After header when limit exceeded
  • Support different limits per endpoint and per plan tier
  • Rate limit rules configurable without restarting services
  • Expose metrics: current usage, violations, top abusers

Non-Functional

  • Add <1ms overhead to every request (p99)
  • 99.999% availability — fail open if limiter is down
  • Support 1M+ concurrent clients, 100K QPS
  • Horizontally scalable — add nodes without reconfiguration
  • Consistent behavior across multiple API server instances

5. Core Components

Rate Limiter Middleware
Runs inside API Gateway. For each request, calls Redis to check and increment the counter. Decision in <1ms. If Redis is unreachable, fail open (let request through) and log the miss.
Redis Cluster
Stores counters per (clientId, endpoint, window). Cluster mode shards keys using consistent hashing. Master-replica pairs ensure HA. Lua scripts run atomically — no race conditions.
Rules Service
CRUD API for rate limit rules (plan tier → endpoint → limit). Rules stored in PostgreSQL, cached in Redis with 30s TTL. Middleware polls Redis for rules — hot reload without restart.
Local Cache (In-Process)
Each API Gateway node maintains a small in-memory cache of recent decisions (last 100ms window). Reduces Redis calls by ~60% for repeated requests from same client.
Violations Log
Async log of 429 events to Kafka → ClickHouse. Powers dashboards: top abusers by IP/user, violation trends, per-endpoint violation rates. Used for alerting and abuse detection.

7. Key Design Decisions

Algorithm Choice: Sliding Window Counter
Token bucket allows bursts (good), but state is harder to shard. Sliding window counter (prev_count × overlap_ratio + curr_count) is chosen: O(1) memory, no boundary burst, easily distributable. Two Redis keys per client per window. For burst-tolerant APIs, use token bucket per endpoint via algorithm config field.
Atomic Check-and-Increment via Lua
Without atomicity, two concurrent requests could both read count=99 (limit=100), both increment to 100, and both be allowed — double-counting. Redis Lua script runs atomically on the server: GET → compare → INCR → EXPIRE, all in one operation. No race condition possible.
Placement: Gateway vs. Sidecar vs. Service
API Gateway: single enforcement point, easy to update, possible single point of failure. Sidecar per service: resilient but N×Redis calls for same client hitting different services. Dedicated service: adds network hop. Recommendation: API Gateway with Redis HA — best latency/simplicity tradeoff.
Fail Open vs. Fail Closed
If Redis cluster goes down: Fail closed (deny all) = 100% outage. Fail open (allow all) = temporarily unprotected but functional. Choose fail open with immediate alerting. Log every fail-open event. Accept slight over-serving during Redis downtime rather than a customer-visible outage.

Token Bucket
A bucket holds tokens that refill at a steady rate (e.g., 10/sec, max capacity 100). Each request consumes 1 token. If the bucket is empty, the request is denied. Allows short bursts up to bucket capacity, then throttles to the refill rate. Perfect for APIs where brief traffic spikes are acceptable — burst capacity is configurable independently of the sustained rate.

9. Self-Check

Fixed window vs sliding window vs token bucket — when to use each? Algorithms +

Fixed window: divide time into fixed buckets (1-min windows). Counter resets at :00. Simplest. Problem: burst at boundary — 100 requests at :59 + 100 at :01 = 200 in 2 seconds while both windows are 'valid'. Use when simplicity > accuracy.

Token bucket: bucket fills at a steady rate (10 tokens/sec, max 100). Each request consumes 1 token. Empty bucket = deny. Allows bursting up to capacity, then throttles to fill rate. Best for: APIs where short bursts are permitted.

Sliding window log: every request timestamp stored in a sorted set. Count in last N seconds — exact. Memory: O(requests) per client. Best for strict accuracy. Sliding window counter: approximation using two fixed windows weighted by position. O(1) memory, ~95% accurate. Best production balance.

How do you rate limit across multiple API server instances? Distributed +

Local counters: fast (in-memory) but with 10 servers a user gets 10× the limit. Fails for distributed deployment. Centralized Redis: all instances share one Redis. Atomic Lua scripts prevent race conditions. Single source of truth. +0.5-1ms latency per request. This is the standard approach.

Redis Cluster: shard rate limit keys across multiple Redis nodes for horizontal scale. All keys for one client hash to one shard — Lua scripts run on the correct shard via CLUSTER KEYSLOT. Cell-based limiting: each server maintains a local token bucket, periodically syncs with Redis to reconcile. Reduces Redis traffic; allows small bounded over-limit. Acceptable for non-security-critical APIs.

A DDoS sends 1M req/sec. How does the rate limiter cope? Scale +

At 1M req/sec with sliding window: 1M ZADD operations/second to Redis sorted sets. Redis handles ~100K ops/sec per instance — this is the bottleneck. Mitigations: (1) Fixed window with INCR (O(1), Redis handles 1M+ INCR/sec) instead of sorted set. Sacrifice accuracy for scale. (2) Local token bucket in application memory — zero Redis calls for most requests, sync every 100ms. Bounded over-limit.

(3) WAF/CDN-level blocking (AWS WAF, Cloudflare) before the rate limiter — blocks known bad IPs and bot patterns at the network edge, never reaches your API. For true DDoS, network-level mitigation (Cloudflare Magic Transit, AWS Shield) is required — no application-layer rate limiter sustains 1M req/sec.

How do you implement different rate limits per tier (free/pro/enterprise)? Multi-tier +

Store rules in Config Service per (clientTier, endpoint): free = 100/hour for /search, pro = 10K/hour, enterprise = custom. API key carries a tier claim (or middleware looks it up). Same algorithm, different parameters.

Rule evaluation: fetch (api_key_tier, endpoint) rule from Config cache (30s TTL). Update rules: write to Config DB → propagates to middleware cache within 30s. No restart needed. Multi-level: some systems implement both per-endpoint AND total-per-client limits simultaneously — a user might have 100 req/hour for /search and 500 req/hour total. Check both counters per request.

Redis is down. Should you fail open or fail closed? Resilience +

For most production APIs: fail open (allow requests). Rationale: a rate limiter outage should not take down your entire API — the harm from allowing extra requests during an outage is less than a full API outage visible to all users. Implementation: try Redis with 50ms timeout. On timeout/error: allow the request, log the error, increment a metric. Alert on-call if error rate > 1%.

Circuit breaker: if Redis errors exceed threshold, open the circuit — allow all requests directly without trying Redis. Close when Redis recovers. Fallback local limiting: on Redis failure, use in-process counter per instance. Allows up to N×limit but bounds damage. For security-critical limits (login brute force, password reset): fail closed — return 503 rather than allowing unlimited attempts.

10. Interview Tips

🎯
Clarify: per-user or per-IP? Per-user is better for authenticated APIs (prevents sharing). Per-IP is needed for unauthenticated endpoints. Mention both dimensions and ask which to prioritize.
⚛️
Lua script atomicity is the key insight — without it, concurrent requests race on the counter. Mention this explicitly: "I'd use a Redis Lua script to atomically check and increment — Lua runs as a single atomic operation on Redis."
🔓
Always address fail-open — if you don't mention it, the interviewer will ask. Justify: a brief window of unprotected traffic is better than customer-visible 429s caused by your rate limiter itself being down.
📡
Return the right headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, and on 429: Retry-After. This shows product thinking — good clients will back off instead of hammering.
🌐
Multi-region is a follow-up trap — interviewers often push "what if you have 3 regions?" Answer: regional limits (simple) vs. centralized Redis with replication (accurate but latency + single point). Most production systems use regional limits.
