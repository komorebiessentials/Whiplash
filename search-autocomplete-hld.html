Section 06

  
Database Design

  

    
In-Memory Trie (on Autocomplete Servers)

    

      Each node: children Map<char, TrieNode>

      Each node: topK List<{query, score}> — cached top-5

      Lookup: O(prefix_length) — traverse prefix, return node.topK

      Memory: ~500 MB for 10M queries (fits in RAM)
    

    
Loaded from S3 snapshot on startup. Hot-reloaded every 10min when new snapshot is available. Zero-downtime: load into new trie object, swap reference atomically.

  

  

    
ac:{prefix} (Redis Cache)

    

      Key: ac:{prefix}

      Value: JSON array of top-10 suggestions

      TTL: 10 minutes

      

      Covers the long tail of prefixes not cached by CDN
    

    
Sub-ms lookup. Updated every 10min by Flink job for trending terms. On miss: query Trie Service, then populate Redis.

  

  

    
query_logs (Kafka → Flink → Redis)

    

      query      TEXT — the search string

      user_hash  TEXT — anonymized user (GDPR)

      timestamp  TIMESTAMP

      lang       TEXT
    

    
Raw events in Kafka, consumed by Flink Streaming. 24h sliding window aggregation produces query → count map. Updates Redis top-K every 10min. Count-Min Sketch for approximate counting (sub-linear memory).

  







  
Section 07

  
Key Design Decisions

  

    
Trie with Cached Top-K per Node — The Core Optimization

    

      Naive trie: traverse to prefix node, then DFS to collect all words → O(p + n) where n = all words with that prefix. Too slow at 58K QPS.


      Optimized: each node pre-caches its top-K completions (sorted by frequency). Lookup = traverse prefix → return node.topK → O(p) constant time.


      Trade-off: more memory (each node stores K query strings), harder to update (must propagate score changes up the tree). But the serving latency benefit is decisive.
    

  

  

    
Data Pipeline: Search Logs → Flink → Redis → Trie

    

      
    
        
  • Every search submit fires Kafka event with query + timestamp
    • 
        
  • Flink streaming job: 24h sliding window, Count-Min Sketch counting → top-K per prefix
    • 
        
  • Every 10 minutes: Flink pushes updated top-K to Redis, hot-patches Trie
    • 
        
  • Nightly: full Trie Builder rebuilds complete trie from Flink aggregates, uploads to S3
    • 
        
  • Servers hot-reload trie atomically — zero downtime
    • 
      

    

  

  

    
Personalization — Client-Side Merge

    

      Global trie gives the same results to all users for "app". To personalize: store user's recent 50 searches in client-side localStorage. On keyup, client merges global suggestions (from server) with local recent searches, boosts recency-matched results.


      Server-side personalization requires per-user index — expensive (1B user models). Client-side merge is the practical approach used by Google/YouTube.
    

  

  

    
CDN Caching — Huge Win

    

      The top-5 suggestions for "ap", "app", "appl", "apple" are the same for all users (before personalization). CDN caches GET /suggest?q=app with TTL=5min. Cache hit rate for common prefixes is >65%. This drops origin server load from 58K to ~20K QPS — the CDN absorbs the rest.

Section 09

  
Self-Check: Interview Questions

  

    

      Trie vs SQL LIKE query — why trie? Data Structure
      +
    

    
SQL LIKE 'app%' requires scanning all rows where the term starts with 'app' — even with a B-tree index, this is O(matching_rows). With 1 billion unique search terms, this is too slow. A trie traverses exactly prefix_length nodes to reach the prefix node, then returns top-K children by score — O(p+K) where p=prefix length (3) and K=10. The speed difference is dramatic: SQL might scan 50 million rows for a 3-letter prefix; the trie visits 3 nodes then pops 10 results. Memory tradeoff: a trie for 1B terms can be 50-100GB — that's why it lives in service memory on dedicated machines and is sharded by first letter.

  

  

    

      How do you keep suggestions fresh when a term trends? Freshness
      +
    

    
The Flink streaming job consumes all search events from Kafka in real-time. It uses a 24-hour sliding window with Count-Min Sketch to count phrase frequencies per prefix. Every 10 minutes, it pushes updated top-K to Redis, invalidating stale suggestions. The trie is rebuilt nightly from the complete dataset and hot-patched every 10 minutes with the top trending terms from Flink. TTL-based expiry (CDN: 5min, Redis: 10min) ensures stale data eventually evicts. A term going viral (e.g. "apple vision pro" on announcement day) appears in suggestions within ~10 minutes.

  

  

    

      What is Count-Min Sketch and why use it? Algorithms
      +
    

    
Exact frequency counting for 1 billion unique terms would require a hash map with 1 billion entries — ~32GB just for counters. Count-Min Sketch uses a fixed 2D array (e.g., 8 hash functions × 1M buckets = 8MB). Each term is hashed 8 times; each bucket is incremented. To query: hash 8 times, return the minimum count seen. The minimum eliminates over-counting from collisions. The result is approximate (always over-counts, never under-counts) within a provable error bound. For ranking autocomplete suggestions, approximate top-K is perfectly fine — whether 'apple' was searched 1,000,000 or 1,001,234 times doesn't change its rank.

  

  

    

      How does personalization work without adding latency? Personalization
      +
    

    
Personalized results can't be CDN-cached (they're per-user). Strategy: two-phase approach. (1) Return CDN/Redis result immediately (<10ms). (2) Simultaneously fetch personalized top-5 from the API (with auth token, ~50ms). (3) If personalization arrives before user selects, update the dropdown. This is 'stale-while-revalidate' — the user sees generic results instantly, personalization augments them within 50ms without blocking the initial render. Personalization data (user's recent searches) stored in Redis per-user with 7-day TTL: key 'user_history:{userId}', sorted set of (term, last_searched_timestamp).

  

  

    

      A user types 'autocomplete system design' — how many API calls? Optimization
      +
    

    
Without debouncing: 26 characters = 26 API calls in ~3 seconds. With 150ms debounce: only fires when user pauses typing for 150ms. For a 3-word phrase, typical pauses are after each word: after 'autocomplete', after 'system', after 'design'. Result: ~3-4 API calls instead of 26. Browser sessionStorage cache: if user types 'app', sees results, types more, then backspaces to 'app' — second time is instant from session cache, zero API call. This combination (debounce + session cache) reduces API calls by ~90% compared to naive per-keystroke requests.

  







  
Section 10

  
Scaling & Bottlenecks

  

    

      
CDN Offload

      
Common prefixes (2–4 chars) are served from CDN edge. 65%+ cache hit rate means origin servers only handle rare/long prefixes. Most critical scaling lever.

    

    

      
Stateless Servers

      
All autocomplete servers load the same trie from S3 — fully stateless. Add servers behind the load balancer with no coordination needed. Scale to 100s of servers.

    

    

      
Trie Sharding

      
If trie exceeds available RAM: shard by first letter (a-f → server group 1, g-m → group 2, etc.). 26 shards × 500MB = fits in small fleet. Route by first char of query.

    

    

      
Hot-Reload Zero Downtime

      
Hourly trie rebuild uses atomic reference swap (CAS). Rolling reload across servers means some serve old trie briefly — acceptable since updates are every 10min anyway.

    

    

      
Kafka Partitioning

      
Search log topic partitioned by language. Flink consumes per-language partition → builds per-language trie. Multilingual support with clean separation.

    

    

      
Debouncing on Client

      
Client fires suggestion request only after 150ms of no keystrokes (debounce). Reduces actual QPS by ~5x vs sending on every keyup. Critical for mobile keyboards.

    

  







  
Section 11

  
Interview Tips

  
🌳
Top-K per node is the key insight: Without it, suffix traversal makes every suggestion O(p+n). With it, every lookup is O(p) — prefix length only. This is the main algo question in this problem.

  
⚡
CDN before talking about servers: Mention CDN caching of prefix responses early. "app" has the same 5 suggestions for everyone — CDN can handle 65% of traffic. Interviewers love this insight.

  
🔄
Separate serving from data pipeline: The trie serving layer is read-only and ultra-fast. The data pipeline (Kafka → Flink → Redis) is async and streaming. Never let pipeline latency affect suggestion latency.

  
📉
Eventual consistency is fine here: Trending queries appearing in suggestions within 10 minutes is acceptable. Real-time exact counting is overkill — Count-Min Sketch + 10min batch is the right call.

  
🔍
Elasticsearch trade-off: If asked about alternatives, Elasticsearch with edge n-gram gives you fuzzy matching and easier ops at the cost of higher latency (~50-200ms vs <5ms for trie). Know when to recommend it over a custom trie.
