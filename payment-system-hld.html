8. Key Design Decisions

Idempotency — Preventing Double Charges
The problem: Network fails after charge hits Visa but before response returns. Client retries. Naive system charges twice.

Solution: Client sends unique Idempotency-Key header per logical operation. Server algorithm:
1. Compute Redis key = hash(merchant_id + idempotency_key)
2. Run Lua script: if not EXISTS key then SET key "PROCESSING" EX 86400; return nil else return GET key end
3. If nil → process payment → update Redis key with full response
4. If "PROCESSING" → another thread is handling it → wait + retry
5. If response JSON → return cached response immediately

Key invariant: The Lua script is atomic in Redis — no two threads can both see "not exists" simultaneously.
Double-Entry Ledger — Immutable Financial Records
Rule: Every payment creates exactly two ledger entries — a debit and a credit. The ledger always sums to zero.

Example — $20 payment from customer to merchant:
• DEBIT   customer_receivable    $20
• CREDIT merchant_payable        $20

Why: Enables reconciliation (sum both sides = 0 always), audit trail (no UPDATE ever — only INSERT), and debugging (replay ledger to reconstruct any account balance at any point in time).

Implementation: ledger_entries table has no UPDATE permission granted to the payment service. Immutability enforced at DB level.
Distributed Transaction — Saga Pattern
Payment flow spans multiple services. Two-phase commit (2PC) is too slow and locks resources.

Saga approach:
1. PaymentService calls Visa API → success (charge authorized)
2. PaymentService writes to LedgerService → success
3. PaymentService emits payment.succeeded to Kafka

Failure at step 2: Compensating transaction — PaymentService calls Visa to void the authorization. If void fails → alerts for manual reconciliation.

Outbox pattern: LedgerService write + Kafka emit are in the same DB transaction. A relay process publishes to Kafka from the outbox table asynchronously, guaranteeing atomicity.
PCI-DSS Scope Reduction
Strategy: minimize PCI scope to a single isolated service.

• Card number tokenized at API Gateway — never reaches Payment Service
• Token (e.g., pm_xxx) is stored and passed everywhere
• Only Vault service has the token→PAN mapping, in isolated network zone
• All logs must scrub card numbers — enforced by log sanitization middleware
• Vault is the only system that communicates with card networks directly

9. Scaling & Bottlenecks

PostgreSQL (Ledger) ACID transactions are non-negotiable for financial records. Scale reads with read replicas for reporting. Partition ledger_entries by created_at month — old partitions become read-only and can be archived to cold storage.
Redis (Idempotency) 200 GB for 1B daily keys. Use Redis Cluster for horizontal scale. Keys expire in 24h — storage is self-maintaining. For 35K TPS peak, Redis handles ~1M ops/sec comfortably.
Fraud Detection Latency ML model must score in <50ms. Pre-compute user feature vectors and store in feature store (Redis). Scoring = low-latency vector lookup + model inference. Rule-based engine as fallback if ML unavailable.
Kafka (Async Fan-out) payment.succeeded → webhook delivery, payout scheduler, analytics, fraud model retraining. All async — keeps payment API fast. Webhook worker retries with exponential backoff + dead-letter queue.
Multi-Region Active-Active Use active-active with global database (CockroachDB or Aurora Global). Route user to nearest region. Idempotency keys must be globally unique — include region prefix. Quorum write for ledger entries.

10. Self-Check

+
What happens if a client retries with the same idempotency key after a network timeout?
Idempotency
The Lua script on the second request sees the cached response in Redis (or "PROCESSING" if still in-flight). If cached: return the same 200 response immediately — no charge happens. If "PROCESSING": the client should wait and retry. This guarantees exactly-once semantics even across network failures. The key insight: the idempotency key is per logical payment attempt, not per HTTP request — the same key must be reused on all retries of the same payment.
+
Why store monetary amounts as BIGINT in cents instead of FLOAT or DECIMAL?
Data Types
FLOAT uses binary floating-point — 0.1 + 0.2 = 0.30000000000000004 in IEEE 754. For money this is catastrophic: rounding errors compound over millions of transactions. DECIMAL avoids binary issues but is slower and variable-width. BIGINT in smallest unit (cents): $20.00 stored as 2000 — exact integer arithmetic, no rounding, fixed 8 bytes, fast. Always convert at display time: amount / 100. This is non-negotiable in financial systems; Stripe, PayPal, and every serious payment processor use integer cents.
+
What is a double-entry ledger and why is it required in payment systems?
Accounting
Double-entry bookkeeping records every financial transaction in at least two accounts: a debit to one and a credit to another. Every payment creates exactly two rows: DEBIT customer_account $20 and CREDIT merchant_account $20. The sum of all debits always equals the sum of all credits — this invariant detects data corruption. It also enables: 1) reconciliation at any time by summing all entries, 2) an immutable audit trail (never UPDATE, only INSERT), 3) replay ability (compute any balance at any historical point purely from the ledger).
+
If the Ledger write fails after Visa has already authorized the charge, what happens?
Saga Pattern
This is the distributed transaction problem. With the Saga pattern, the Payment Service runs a compensating transaction: it calls the Visa API to void the authorization. The customer is never charged. If the void itself fails, the system alerts on-call for manual intervention and logs the orphaned authorization. The Outbox pattern prevents this scenario at the source: the Ledger write and the Kafka emit are in the same DB transaction — if either fails, both roll back. The relay retries the Kafka publish from the outbox table until it succeeds.
+
Why tokenize card data at the API Gateway rather than in a backend service?
PCI-DSS
PCI-DSS compliance requires strict controls on any system that touches raw card data (PAN). By tokenizing at the API Gateway, the raw card number is immediately replaced with a token (pm_xxx) — the Payment Service, Fraud Service, and all downstream systems only ever see the token. This reduces PCI scope to a single, hardened gateway + Vault service. Without this, every backend service, log pipeline, and cache that handles payment data would need PCI audit — an enormous compliance burden. The SDK iframe approach goes further: the PAN never even hits the merchant's servers.

11. Interview Tips

🔑
Lead with idempotency. Interviewers at payment companies expect you to immediately identify the double-charge problem. Explain the Lua atomic check-and-set in Redis before they ask.
📚
Double-entry ledger is not optional. Any answer without immutable double-entry bookkeeping misses the fundamentals. Explain: DEBIT + CREDIT, always balanced, never UPDATE.
🔒
Mention PCI-DSS scope reduction proactively. "I'd tokenize card data at the API gateway so only Vault is in PCI scope." This shows production awareness beyond pure system design.
⚠️
Never use FLOAT for money. FLOAT has binary rounding errors. Always store amounts as BIGINT in smallest currency unit (cents). Classic gotcha question.
🔄
Webhooks need reliability. Deliver with retry + exponential backoff. Store delivery attempts. Let merchants replay. Idempotency on webhook delivery too — merchants may receive same event twice.
📊
Consistency model choice: Ledger needs strong consistency (ACID). Analytics and fraud dashboards can be eventually consistent. State this explicitly — shows understanding of trade-offs.
