SOLID Principles

5 principles for writing maintainable, extensible object-oriented code. The vocabulary every LLD interviewer expects you to use fluently.

What Is ItS — SRPO — OCPL — LSP I — ISPD — DIPReal WorldInterview Tips
01

What Is It

Coined by Robert C. Martin ("Uncle Bob"). Five principles that, when followed together, produce code that is easy to extend without breaking existing functionality.

S
Single Responsibility
A class should have only one reason to change.
O
Open/Closed
Open for extension, closed for modification.
L
Liskov Substitution
Subtypes must be substitutable for their base types.
I
Interface Segregation
Many specific interfaces over one general interface.
D
Dependency Inversion
Depend on abstractions, not concretions.
02

S — Single Responsibility Principle

"A class should have only one reason to change." If a class does multiple jobs, changing any one of them can break the others.

S
Single Responsibility
One job, one reason to change

If UserService creates users, sends emails, AND saves to DB — it has 3 reasons to change: business logic, email template, DB schema. Any change can break the others.

✗ Violation
class UserService {
  void createUser(User u) {
    db.save(u);           // DB job
    email.sendWelcome(u); // Email job
    log.info("created"); // Logging job
  }
}
✓ Fix
class UserService {    // only business logic
  void createUser(User u) { repo.save(u); }
}
class WelcomeEmailJob { // only email
  void send(User u) { email.sendWelcome(u); }
}
class UserRepository { // only persistence
  void save(User u) { db.insert(u); }
}
03

O — Open/Closed Principle

"Open for extension, closed for modification." Add new behavior by adding new code, not by changing existing tested code.

O
Open/Closed
Extend without modifying — the Strategy pattern in practice

Adding a new customer tier should not require modifying the discount calculator. Each new type = a new class, zero changes to existing code.

✗ Violation — add tier = edit this
double getDiscount(Customer c) {
  if (c.type == PREMIUM) return 0.10;
  if (c.type == VIP)     return 0.20;
  return 0; // add GOLD? modify here ← bad
}
✓ Fix — add tier = new class only
interface DiscountStrategy {
  double calculate(Customer c);
}
class PremiumDiscount implements DiscountStrategy {
  public double calculate(Customer c) { return 0.10; }
}
class GoldDiscount implements DiscountStrategy {
  public double calculate(Customer c) { return 0.30; }
}
04

L — Liskov Substitution Principle

"Subtypes must be substitutable for their base types without altering correctness." If code works with a Rectangle, it must work with any Rectangle subclass.

L
Liskov Substitution
The Square/Rectangle trap — the most misunderstood principle

Square "is-a" Rectangle geometrically. But Square violates Rectangle's behavioral contract: setting width must not change height. The fix: don't model this with inheritance.

✗ Violation
class Rectangle {
  int w, h;
  void setWidth(int w) { this.w=w; }
  void setHeight(int h){ this.h=h; }
  int area() { return w*h; }
}
class Square extends Rectangle {
  void setWidth(int w) { this.w=this.h=w; } // breaks contract!
}
// Test: r.setWidth(4); r.setHeight(5); r.area()==20
// For Square: area()==25  ← WRONG
✓ Fix
interface Shape { int area(); }

class Rectangle implements Shape {
  int w, h;
  public int area() { return w*h; }
}
class Square implements Shape {
  int side;
  public int area() { return side*side; }
}
// No inheritance — no contract violation
05

I — Interface Segregation Principle

"Clients should not be forced to depend on interfaces they don't use." A fat interface forces implementations to define methods they don't need.

I
Interface Segregation
Many small interfaces > one large interface
✗ Violation — Dog forced to fly
interface Animal {
  void eat();
  void fly();   // Dog can't fly!
  void swim();
}
class Dog implements Animal {
  public void eat()  { /* ok */ }
  public void fly()  { throw new UnsupportedOperationException(); }
  public void swim() { /* ok */ }
}
✓ Fix — compose only what's needed
interface Eatable  { void eat(); }
interface Flyable  { void fly(); }
interface Swimmable{ void swim(); }

class Dog implements Eatable, Swimmable {
  public void eat()  { /* ok */ }
  public void swim() { /* ok */ }
}
class Bird implements Eatable, Flyable {
  public void eat() { /* ok */ }
  public void fly() { /* ok */ }
}
06

D — Dependency Inversion Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions." The foundation of dependency injection.

D
Dependency Inversion
Inject abstractions — enables testing and swapping implementations
✗ Violation — hardcoded MySQL
class OrderService {
  // Hardcoded — can't test, can't swap
  private MySQLDatabase db = new MySQLDatabase();

  void placeOrder(Order o) {
    db.save(o);
  }
}
✓ Fix — injected abstraction
interface Database { void save(Order o); }

class OrderService {
  private final Database db; // abstraction
  OrderService(Database db) { this.db = db; }
  void placeOrder(Order o) { db.save(o); }
}
// Production: new OrderService(new MySQLDatabase())
// Tests:      new OrderService(new InMemoryDatabase())
07

Real-World Usage

Spring Framework
DIP via IoC
Spring's entire dependency injection container is DIP automated. @Autowired injects abstractions, not concretions.
JPA / Hibernate
OCP
You extend behavior by adding entity classes and repository interfaces — zero modification to JPA internals.
Java Collections
ISP
List, Set, Map are separate interfaces. Iterable, Collection, List are a hierarchy — each adds only what's needed.
Strategy Pattern
OCP in action
Every Strategy pattern implementation directly demonstrates OCP: new algorithm = new class, no changes to context.
08

Interview Tips

1
Don't just recite definitions. For each principle, have a violation and a fix ready. "SRP means one reason to change — here's a class with 3 reasons and here's how I'd split it."
2
OCP is the most visible in LLD designs. When an interviewer asks "how do you add a new vehicle type?", your answer should demonstrate OCP: new class, no modification to existing code.
3
LSP is the most subtle. The Square/Rectangle example is the canonical one — know it cold. The test: can you substitute a subtype everywhere the base type is used without changing behavior?
4
DIP is the foundation of testability. "I inject dependencies so I can swap in a mock in tests." This connects SOLID to testing — interviewers love seeing this connection.
5
Know when NOT to apply SOLID. Over-engineering is a trap. A 20-line utility class doesn't need 5 sub-interfaces. Mention: "I'd apply OCP here because I expect this code to be extended. For this simpler class, the overhead isn't justified."
6
SOLID in your LLD answers. When presenting your design, explicitly name the principles: "I used an interface here to follow DIP." "Adding a new split type follows OCP." This is the vocabulary interviewers listen for.
