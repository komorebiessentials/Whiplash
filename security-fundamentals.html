What Is It

Application security covers two core problems: Authentication (who are you?) and Authorization (what are you allowed to do?). At senior level, you're expected to design systems that handle these correctly across service boundaries, at scale, and under adversarial conditions.

OAuth2 / OIDC

Industry-standard protocols for delegated authorization and federated identity. The foundation of every modern auth system.

JWT

JSON Web Tokens — signed, self-contained tokens for stateless authentication. Widely used but with sharp edges.

mTLS

Mutual TLS — both client and server present X.509 certificates. The standard for service-to-service auth in zero-trust environments.

Zero Trust

"Never trust, always verify" — no implicit trust even inside the network perimeter. Every call must be authenticated and authorized.

The Problem It Solves

Session scalability: Server-side sessions require shared storage (sticky sessions or Redis). At millions of users, this becomes a bottleneck. JWTs are stateless — the server validates a signature without any lookup.

Service identity: In microservices, Service A calls Service B. How does B know the caller is actually A and not an attacker who compromised the network? mTLS provides cryptographic proof of identity.

Delegated authorization: A user wants to grant your app access to their Google Drive without giving you their password. OAuth2 solves this with access tokens scoped to specific permissions.

The confused deputy problem: A service has elevated privileges; an attacker tricks it into performing actions on their behalf. Zero-trust and per-request authorization prevent this.

OAuth2 Authorization Code + PKCE Flow

PKCE (Proof Key for Code Exchange): The browser generates a random code_verifier, sends its hash code_challenge in the redirect. When exchanging the code for a token, it sends the original code_verifier. The auth server verifies the hash — prevents an attacker who intercepts the code from using it without the verifier.

Mechanisms in Depth

JWT Deep Dive

A JWT has three Base64url-encoded parts: header (algorithm, token type), payload (claims: iss, sub, aud, exp, iat + custom), signature (HMAC-SHA256 or RSA/ECDSA of header.payload).

RS256 vs HS256: HS256 uses a shared secret — all services must know it (hard to rotate, any service can forge tokens). RS256 uses a private key (only auth server signs) + public key (anyone can verify). Always use RS256 for production.

JWT pitfalls: The alg:none attack — always validate the algorithm explicitly, never trust the header's alg. The algorithm confusion attack (RS256 → HS256) — attacker sends a token signed with the public key using HS256; if the server accepts the public key as an HS256 secret, it validates. Use libraries with explicit algorithm configuration.

Token revocation: JWTs can't be revoked before expiry (they're stateless). Use short TTL (15 min) + opaque refresh token (stored in HttpOnly cookie, validated against a DB/Redis store). On logout, delete the refresh token.

OAuth2 Grant Types

Authorization Code + PKCE

For browser/mobile apps (public clients). Most secure. PKCE prevents authorization code interception attacks.

Client Credentials

For service-to-service auth (no user involved). client_id + client_secret → access token. Use for backend-to-backend API calls.

Device Code Flow

For CLI tools and smart TVs. Device shows a code; user authenticates on phone. Polls for token.

Implicit Flow (DEPRECATED)

Tokens returned in URL fragment — visible in browser history and server logs. Never use in new systems.

Transport Security: TLS and mTLS

TLS 1.3: 1-RTT handshake (vs TLS 1.2's 2-RTT). ECDHE for forward secrecy — session keys are ephemeral, so past sessions can't be decrypted even if the server's private key is compromised.

mTLS: Both client and server present X.509 certificates. The server verifies the client's cert against a CA. Used in service mesh (Istio/Linkerd) where each service gets a SPIFFE/X.509 identity. Prevents impersonation even inside the VPC — an attacker who compromises the network can't make requests as Service A without its cert/key.

Zero Trust Architecture

"Never trust, always verify" — no implicit trust based on network location. Every service-to-service call must be authenticated (mTLS) and authorized (RBAC/ABAC). Micro-segmentation limits blast radius. Continuous verification re-checks identity on sensitive operations. BeyondCorp (Google) and SPIFFE/SPIRE are the reference implementations.

API Security — OWASP Top 10

BOLA / IDOR

Broken Object Level Authorization — user accesses another user's resource by changing an ID in the URL. Fix: always check resource.ownerId == authenticatedUserId.

SSRF

Server-Side Request Forgery — attacker causes server to make requests to internal services. Fix: validate/allowlist URLs, block metadata service IP (169.254.169.254).

Secrets Management

Never store secrets in code or env vars. Use HashiCorp Vault, AWS Secrets Manager, or Kubernetes Secrets with KMS encryption. Rotate regularly.

Input Validation

Validate all input at API boundary. Allowlist > denylist. Reject unexpected fields. Parameterised queries prevent SQL injection.

// JWT Validation — always explicit algorithm, validate all claims
class JwtValidator {
    private final JWTVerifier verifier;

    JwtValidator(RSAPublicKey publicKey) {
        verifier = JWT.require(Algorithm.RSA256(publicKey, null)) // explicit alg
            .withIssuer("https://auth.example.com")
            .withAudience("api.example.com")
            .acceptLeeway(30) // 30s clock skew
            .build();
    }

    DecodedJWT validate(String token) {
        try {
            return verifier.verify(token); // throws on invalid sig/exp
        } catch (JWTVerificationException e) {
            throw new UnauthorizedException("Invalid token: " + e.getMessage());
        }
    }

    String getUserId(DecodedJWT jwt) {
        return jwt.getSubject(); // "sub" claim
    }
}
// OAuth2 Client Credentials — service-to-service
@Component
class ServiceTokenProvider {
    private final WebClient webClient;
    private volatile CachedToken cachedToken; // cache to avoid per-request fetches

    String getToken() {
        if (cachedToken != null && !cachedToken.isExpiredSoon()) {
            return cachedToken.value();
        }
        return fetchNewToken();
    }

    private synchronized String fetchNewToken() {
        if (cachedToken != null && !cachedToken.isExpiredSoon()) {
            return cachedToken.value(); // double-check after lock
        }
        var response = webClient.post()
            .uri("/oauth2/token")
            .bodyValue(Map.of(
                "grant_type", "client_credentials",
                "client_id", clientId,
                "client_secret", clientSecret,
                "scope", "inventory:read"
            ))
            .retrieve().bodyToMono(TokenResponse.class).block();
        cachedToken = new CachedToken(response.accessToken(), response.expiresIn());
        return cachedToken.value();
    }
}
// IDOR Prevention — always verify ownership
@GetMapping("/orders/{orderId}")
OrderResponse getOrder(
    @PathVariable String orderId,
    @AuthenticationPrincipal JwtUser user
) {
    var order = orderRepository.findById(orderId)
        .orElseThrow(() -> new NotFoundException());

    // CRITICAL: check ownership — don't trust the client's orderId
    if (!order.getOwnerId().equals(user.getId())) {
        throw new ForbiddenException("Access denied"); // 403, not 404
        // Note: return 403, not 404, to avoid leaking resource existence
    }

    return OrderMapper.toResponse(order);
}

// Better: use a query that enforces ownership at DB level
Optional<Order> findByIdAndOwner(String id, String ownerId) {
    return orderRepository.findByIdAndOwnerId(id, ownerId); // WHERE id=? AND owner_id=?
}
// HashiCorp Vault — fetch secrets at runtime, not config
@Configuration
class VaultConfig {

    @Bean
    DataSource dataSource(VaultTemplate vault) {
        // Vault returns dynamic short-lived DB credentials
        var creds = vault.read("database/creds/my-role");
        var username = (String) creds.getData().get("username");
        var password = (String) creds.getData().get("password");
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://db:5432/app")
            .username(username).password(password).build();

Real-World Usage

Auth0 / Okta

Managed OIDC/OAuth2 providers. Handles authorization servers, user stores, MFA, and token issuance. Used by most SaaS companies.

Istio / Linkerd

Service meshes that implement automatic mTLS between all services using SPIFFE/X.509 certificates. Zero-trust networking at the platform level.

HashiCorp Vault

Secrets management — dynamic credentials, PKI certificates, encryption as a service. Standard in enterprise and cloud-native environments.

AWS IAM

Role-based access control for AWS services. Service accounts for pods (IRSA on EKS). Instance profiles eliminate static credentials.

Google BeyondCorp

Invented Zero Trust networking in 2011. Every resource requires authentication regardless of network location. Basis for modern ZTNA products.

SPIFFE / SPIRE

Standard for service identity in cloud-native environments. Issues X.509 SVIDs to workloads — used by Istio, Envoy, and HashiCorp Vault.

Interview Tips

"JWT vs session-based auth — when to use each?"

JWT for stateless, horizontally-scaled APIs and microservices. Sessions when you need instant revocation (banking logout) or can afford a shared session store.

"How do you revoke a JWT before expiry?"

Maintain a token revocation list in Redis (jti claim as key, TTL = token expiry). Or switch to short-lived JWTs (15 min) + opaque refresh tokens with a DB-backed revocation list.

"What is PKCE and why is it needed?"

Prevents authorization code interception in public clients (browser/mobile) that can't keep a client_secret. The code_verifier proves the original requestor is the one exchanging the code.

"What's the difference between authn and authz?"

Authentication = who are you? (JWT, cert, cookie). Authorization = what can you do? (RBAC, ABAC, OPA policies). They're separate concerns — always implement both.

"How does mTLS prevent the confused deputy problem?"

Service identity is based on the X.509 cert, not network position. Even if an attacker reaches Service B's port, they can't present Service A's cert without its private key.

"How do you handle secrets in a Kubernetes deployment?"

Kubernetes Secrets base64-encoded (not encrypted by default) → enable etcd encryption at rest + KMS key. Better: use Vault with the sidecar injector or AWS Secrets Manager CSI driver.
