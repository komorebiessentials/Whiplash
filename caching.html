Caching Strategies

Cache strategies define when data enters and leaves a cache. Eviction policies decide what to remove when it's full. Together they determine latency, consistency, and cost.

What Is ItThe ProblemStrategies EvictionCodeTrade-offs Real WorldInterview Tips
01

What Is It

A cache is a fast, in-memory store that sits in front of a slower data store. A cache hit (<1ms) vs a cache miss (DB query: 50-200ms) is a 100-1000× latency difference.

02

The Problem It Solves

Without caching: every request hits the database. A single server can handle ~1,000 DB queries/sec. With caching: the same server handles 100,000+ requests/sec for cached data.

The two hard problems in caching (Phil Karlton): "Cache invalidation and naming things." When the DB changes, how do you know which cache entries are now stale? This is the core challenge that each strategy addresses differently.

03

Caching Strategies

Four strategies — each answers: when does data enter the cache, and how does it stay consistent with the DB?

Most Common
1. Cache-Aside (Lazy Loading)
App checks cache → HIT: return. MISS: load from DB, store in cache, return. Cache populated on demand.
  • Only caches what's actually read
  • Cache failure doesn't break app
  • Simple to implement
  • First read always slow (cold cache)
  • Stale data until TTL expires
  • Cache stampede risk on miss
Read-Heavy
2. Write-Through
Write to cache AND DB simultaneously. Cache always mirrors DB. Reads always fast.
  • No stale reads
  • Cache always warm
  • Write latency = cache + DB combined
  • Caches data that may never be read
Write-Heavy
3. Write-Back (Write-Behind)
Write to cache only. Cache asynchronously flushes to DB. Fastest write path.
  • Fastest writes
  • Batches DB writes for efficiency
  • Data loss if cache crashes before flush
  • Complex implementation
Write-Heavy, Rarely Read Back
4. Write-Around
Writes go directly to DB (bypass cache). Cache populated only on reads. Good for data written once and rarely re-read.
  • Avoids caching cold write data
  • Good for log/event data
  • First read after write is always a miss

Code

class LRUCache<K, V> {
    private final int capacity;
    private final LinkedHashMap<K, V> cache;

    LRUCache(int capacity) {
        this.capacity = capacity;
        // accessOrder=true: get() moves entry to tail (most-recently-used)
        this.cache = new LinkedHashMap<>(capacity, 0.75f, true) {
            protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
                return size() > capacity; // auto-evict LRU when full
            }
        };
    }

    public synchronized V get(K key) {
        return cache.getOrDefault(key, null);
    }
    public synchronized void put(K key, V value) { cache.put(key, value); }
    public synchronized void invalidate(K key)    { cache.remove(key); }
}
public User getUser(String userId) {
    String key = "user:" + userId;

    // 1. Check cache
    User cached = cache.get(key);
    if (cached != null) return cached; // HIT

    // 2. Miss — load from DB
    User user = userRepo.findById(userId);

    // 3. Populate cache with TTL
    cache.put(key, user, Duration.ofMinutes(15));
    return user;
}

public void updateUser(User user) {
    userRepo.save(user);
    cache.invalidate("user:" + user.getId()); // invalidate stale entry
}
// Cache stampede: 10,000 threads miss simultaneously → 10,000 DB queries
// Fix: only one thread loads, others wait on the same lock
private final ConcurrentHashMap<String, Object> locks = new ConcurrentHashMap<>();

public User getUserSafe(String userId) {
    String key = "user:" + userId;
    User hit = cache.get(key);
    if (hit != null) return hit;

    Object lock = locks.computeIfAbsent(key, k -> new Object());
    synchronized (lock) {
        User recheck = cache.get(key); // double-check after lock
        if (recheck != null) return recheck;

        User user = userRepo.findById(userId);
        cache.put(key, user, Duration.ofMinutes(15));
        locks.remove(key);
        return user;
    }
}

Real-World Usage

Redis
Remote cache
Supports allkeys-lru, allkeys-lfu, volatile-ttl eviction. Used for cache-aside at scale.
CDN (CloudFront, Fastly)
Write-around
Origin writes to S3; CDN caches on first read per region. Cache-Control headers define TTL.
CPU L1/L2
Write-back
Hardware-managed write-back cache. CPU writes to cache; hardware flushes to RAM asynchronously.
Browser Cache
TTL-based
Cache-Control: max-age=3600. Stale-while-revalidate for fresh-enough responses.
08

Interview Tips

1
Ask before designing: read-heavy or write-heavy? What's acceptable staleness? These two questions determine the strategy. Don't pick one without asking.
2
Always mention TTL. Without TTL, stale data lives forever. "I'd set a 15-minute TTL here and invalidate on write." Both mechanisms together — proactive invalidation AND TTL as a fallback.
3
Cache stampede is the follow-up question. "What if 10,000 requests all miss simultaneously?" — use a mutex/lock per key so only one thread loads from DB. Double-check after acquiring the lock.
4
LRU is the default eviction policy. Know its O(1) implementation: LinkedHashMap with accessOrder=true, or doubly-linked list + HashMap. LFU is better when you have a small hot set accessed millions of times.
5
Cache invalidation strategies: (a) invalidate on write — immediate consistency; (b) TTL — eventual consistency; (c) event-driven — subscribe to DB changes. "I'd use (a) + (b) as fallback."
6
Hot key problem: one viral key overwhelms one cache shard. Solutions: read replicas for that key, local in-process cache, key splitting (user:123:shard-0..2). Mention this shows depth.
