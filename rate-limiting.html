Rate Limiting
Algorithms

Five algorithms for throttling request rates — from simple fixed windows to distributed Redis-based solutions.

1. What Is It

Rate limiting controls how many requests a client or service can make within a time window. It protects APIs from abuse, prevents DoS attacks, enforces fair usage quotas, and reduces downstream load spikes.

The core decision: how do you count requests and what do you do when the limit is exceeded? The answer determines burst behavior, memory usage, and accuracy.

2. The Problem It Solves

Without rate limiting, a single misbehaving client can overwhelm your API (resource exhaustion), trigger cascading failures downstream, or monopolize quota at the expense of other users.

Secondary problems: boundary bursts (fixed windows allow 2× the rate at window edges), distributed state (multiple server nodes must share counters), and burst tolerance (legitimate clients sometimes need short bursts).

3. How It Works — Token Bucket

The token bucket is the most intuitive algorithm. Tokens refill at a constant rate up to a max capacity. Each request consumes one token. If the bucket is empty, the request is rejected.

4. The Five Algorithms

01
Fixed Window Counter
Divide time into fixed windows (e.g., each minute). Count requests per window. Simple to implement with a single atomic counter + TTL.
Boundary burst risk
02
Sliding Window Log
Store a timestamp for every request in a sorted list. Count entries within the rolling window. Most accurate but O(N) memory per user.
High memory
03
Sliding Window Counter
Hybrid: prev × overlap% + curr. Approximates sliding window using two fixed-window counters. Low memory, no boundary bursts.
Best tradeoff
04
Token Bucket
Refill tokens at rate R, capacity B. Allows bursts up to B. Requests consume a token; empty bucket = reject. Used by most CDNs and APIs.
Burst-friendly
05
Leaky Bucket
Requests queue up; processed at a fixed rate. Excess beyond queue capacity is dropped. Guarantees smooth output but no burst tolerance.
Strict smoothing

The Boundary Burst Problem

Fixed Window allows 2× the rate limit in a 2-second window straddling two windows:

00:00–00:58
0 reqs
00:59
100 reqs ✓
WINDOW RESETS
01:01
100 reqs ✓
01:02–02:00
0 reqs

Result: 200 requests in ~2 seconds — double the allowed rate. Sliding Window Counter fixes this by weighting the previous window.

5. Trade-offs

Algorithm Burst Handling Memory Accuracy Distributed?
Fixed Window Boundary burst (2×) O(1) Low at edges Easy (Redis INCR)
Sliding Window Log None (exact) O(N) per user Exact Hard (sorted set)
Sliding Window Counter Minimal O(1) ~99% Easy (2 counters)
Token Bucket Yes (up to B) O(1) Good Medium (CAS)
Leaky Bucket None (queue) O(queue size) Exact rate Hard (queue sync)

6. Code

public class TokenBucket {
    private final long capacity;
    private final long refillRatePerSec;
    private AtomicLong tokens;
    private volatile long lastRefillTime;

    public TokenBucket(long capacity, long refillRatePerSec) {
        this.capacity = capacity;
        this.refillRatePerSec = refillRatePerSec;
        this.tokens = new AtomicLong(capacity);
        this.lastRefillTime = System.nanoTime();
    }

    public synchronized boolean tryConsume() {
        refill();
        if (tokens.get() > 0) {
            tokens.decrementAndGet();
            return true;
        }
        return false; // 429 Too Many Requests
    }

    private void refill() {
        long now = System.nanoTime();
        long elapsed = now - lastRefillTime;
        long tokensToAdd = elapsed * refillRatePerSec / 1_000_000_000L;
        if (tokensToAdd > 0) {
            tokens.set(Math.min(capacity, tokens.get() + tokensToAdd));
            lastRefillTime = now;
        }
    }
}
// Sliding Window Counter: uses two fixed windows + overlap ratio
// Redis implementation: INCR key:window + EXPIRE

public class SlidingWindowCounter {
    private final int limit;
    private final long windowMs;

    public boolean isAllowed(String userId) {
        long now = System.currentTimeMillis();
        long currWindow = now / windowMs;
        long prevWindow = currWindow - 1;
        double overlap = 1.0 - ((now % windowMs) / (double) windowMs);

        // From Redis (pseudocode shown as Java):
        long prevCount = getCount(userId, prevWindow);
        long currCount = getCount(userId, currWindow);

        double estimated = prevCount * overlap + currCount;
        if (estimated < limit) {
            increment(userId, currWindow);
            return true;
        }
        return false;
    }

    // Redis equivalent:
    // INCR user:{id}:window:{n}
    // EXPIRE user:{id}:window:{n} {windowSeconds * 2}
}
public class FixedWindowRateLimiter {
    private final int limit;
    private final long windowMs;
    // userId → [windowStart, count]
    private final ConcurrentHashMap<String, long[]> counters =
        new ConcurrentHashMap<>();

    public synchronized boolean isAllowed(String userId) {
        long now = System.currentTimeMillis();
        long windowStart = (now / windowMs) * windowMs;

        long[] data = counters.getOrDefault(userId, new long[]{windowStart, 0});

        if (data[0] != windowStart) {   // new window — reset
            data[0] = windowStart;
            data[1] = 0;
        }

        if (data[1] < limit) {
            data[1]++;
            counters.put(userId, data);
            return true;
        }
        return false;
    }
}
-- Redis Lua script: atomic check-and-increment
-- Called as: EVAL script 1 {key} {limit} {windowSeconds}
local key    = KEYS[1]
local limit  = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = tonumber(redis.call('GET', key) or 0)

if current < limit then
    redis.call('INCR', key)
    redis.call('EXPIRE', key, window)
    return 1   -- allowed
else
    return 0   -- rejected (429)
end

7. Real-World Usage

Nginx
limit_req uses leaky bucket. Requests queue up; excess returns 503. Used for API gateways and CDN edge nodes.
Stripe API
Token bucket per API key. Returns Retry-After header. Stripe also has request-level metering by endpoint type.
GitHub API
Fixed window (60 req/hr unauthenticated, 5000/hr authenticated). Headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset.
AWS API Gateway
Token bucket — steady-state rate + burst limit. Configurable per stage and per route.
Redis Rate Limiter
Redis Sorted Set (ZADD + ZREMRANGEBYSCORE) for sliding window log. Used by Kong, Envoy, custom API proxies.

8. Interview Tips

🎯
Default answer: Token Bucket — it's burst-friendly, O(1) memory, and used in production by most major APIs. Justify your choice with "allows short bursts while maintaining average rate."
⚠️
Mention the boundary burst problem — it immediately shows depth. Then pivot to sliding window counter as the fix that keeps O(1) memory.
🌐
Distributed rate limiting requires centralized state — Redis with a Lua script or INCR+EXPIRE. Otherwise each server node counts independently and the limit is multiplied by N servers.
📡
Return 429 Too Many Requests with a Retry-After header. Mention graceful degradation — queue the request vs. reject immediately vs. serve degraded response.
📐
Token Bucket vs. Leaky Bucket: Token Bucket allows bursts (variable output), Leaky Bucket smooths output (constant rate). If asked "which for an API?", say Token Bucket — users appreciate burst capacity.
