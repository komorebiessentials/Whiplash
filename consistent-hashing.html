Consistent Hashing

A technique that minimises key remapping when servers are added or removed — from remapping nearly everything to remapping only ~1/N of keys.

01

What Is It

Consistent hashing is a way to decide which server owns a key without making every key depend on the current number of servers.

The core idea

Consistent hashing decides which server owns a key without depending directly on the current number of servers.

It puts both keys and servers into one fixed hash space, usually drawn as a ring. A key belongs to the first server you find when moving clockwise from the key's position.

Because the ring stays fixed, adding or removing a server only changes ownership for nearby keys instead of reshuffling everything.

Why do we need it?

Imagine a cache cluster with millions of hot keys. If you add one more cache server and 70-80% of keys suddenly point to different servers, most cache reads become misses. Those misses fall through to the database at the same time. This is not just inefficient; it can overload the database and make the system slower exactly when you were trying to add capacity.

Consistent hashing makes cluster changes gradual. Adding one server only moves the keys that now fall into that new server's range. Removing one server only moves the keys that belonged to the removed server. Everything else keeps going to the same place.

Modulo hashing
server = hash(key) % N

N is part of the formula. If 3 servers become 4, every key is recalculated with a different divisor. Most keys move even though only one server was added.

Consistent hashing
owner = nextServerClockwise(hash(key))

The hash space is fixed. Adding a server inserts a new point on the ring. Existing server positions do not get renumbered, so most keys keep the same owner.

Important wording: In modulo hashing, N is the number of servers and changing it changes the result for most keys. In consistent hashing, the ring size is fixed; you add or remove server positions inside that fixed space. That is why the whole mapping does not reshuffle.
02

The Problem It Solves

Naive modulo hashing is catastrophically unstable when the number of servers changes.

Imagine 3 cache servers. You route requests with server = hash(key) % 3. Works perfectly until you add a 4th server:

user:2204Server 0Server 0no
user:3307Server 1Server 3YES
user:4412Server 1Server 0YES
user:5515Server 2Server 3YES
Result: ~75% of keys move on average when N changes by 1. Every miss hits the database. In a real system with millions of keys, this is a cache stampede that can take down your DB.
03

How It Works

A hash ring maps both servers and keys into the same space. Each key is owned by the first server clockwise from its hash position.

Hash onto a circle, not a line

Colored arcs show ownership. A key belongs to the first node clockwise from its position.

When a node is added or removed, only the keys in the affected arc move. Dots outside that arc keep their color.

4 nodes · 32 keys · — moved on last change

Adjust the pool and watch which keys actually move.

  1. Create the ring — hash space 0 → 2³²−1 is mapped into a circle. Think of it as a clock with 4 billion positions.
  2. Place servers — hash each server's identifier (IP:port or name) → place it at that position on the ring.
  3. Route a key — hash the key → find its ring position → walk clockwise until you hit a server → that server owns the key.
  4. Add a server — insert its hash position on the ring. Only keys between the new server and its counter-clockwise predecessor move to the new server. All others stay.
  5. Remove a server — delete it from the ring. Its keys walk clockwise to the next server. Only those keys are affected.

Virtual Nodes — solving uneven distribution

Problem: With only 3 physical servers on the ring, they land at 3 random positions. One server may end up owning 60% of the ring arc — hotspot.

Solution: Represent each server with K virtual nodes: server-A-0, server-A-1, ..., server-A-149, each hashed to a different ring position. With 150 virtual nodes per server, distribution is statistically uniform (law of large numbers).
Without virtual nodes
Uneven arcs
Server A: 60% of keys. Server B: 30%. Server C: 10%. One server is a hotspot.
With 150 virtual nodes
~33% each
Each physical server handles ~1/N of total keys. Adding a server of double capacity? Give it 2× virtual nodes.
04

Types & Variants

Different systems adapt the core ring idea for their needs.

Classic
Virtual-node ring
TreeMap of virtual positions → physical server. Used by Memcached clients, Cassandra. K = 100–200 virtual nodes.
Redis Cluster
Hash slots
16,384 fixed slots (CRC16 mod 16384). Each node owns a contiguous range. Inspired by consistent hashing but simpler to reason about slot ownership.
Rendezvous
Highest-random-weight
For each key, score every server as hash(key, server). Pick the highest score. Simpler than a ring, same O(1/N) remapping property. Used in some CDNs.
Jump Hash
Google's variant
O(log N) lookup, zero memory overhead (no ring stored). Bucket count must be known. Useful when N changes are rare and cluster size is fixed.

Code

Java implementation using TreeMap as the ring. O(log N) lookup via ceilingEntry().

class ConsistentHashRing {
    private final TreeMap<Long, String> ring = new TreeMap<>();
    private final int virtualNodes;

    public ConsistentHashRing(int virtualNodes) {
        this.virtualNodes = virtualNodes;
    }

    public void addServer(String server) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(server + "-vn-" + i);
            ring.put(hash, server);
        }
    }

    public void removeServer(String server) {
        for (int i = 0; i < virtualNodes; i++) {
            ring.remove(hash(server + "-vn-" + i));
        }
    }

    public String getServer(String key) {
        if (ring.isEmpty()) throw new IllegalStateException("No servers");
        long hash = hash(key);
        // First entry >= hash. If none (key is past last server), wrap to first.
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        return (entry != null ? entry : ring.firstEntry()).getValue();
    }

    private long hash(String key) {
        // In production: MurmurHash3 or MD5 truncated to 64 bits
        return Math.abs(key.hashCode()) & 0xFFFFFFFFL;
    }
}
ConsistentHashRing ring = new ConsistentHashRing(150);
ring.addServer("cache-A");
ring.addServer("cache-B");
ring.addServer("cache-C");

// Route a key — always stable as long as topology doesn't change
String server = ring.getServer("user:12345"); // → e.g., "cache-B"

// Add a server — only ~1/3 of keys remap
ring.addServer("cache-D");
// "user:12345" might now route to "cache-D" if it's between B's predecessor and B

// Remove a server — its keys move to the next clockwise server
ring.removeServer("cache-B");
// Those keys now route to "cache-C" or "cache-D"
// MODULO HASHING — breaks on topology change
int N = 3;
String server = servers[Math.abs(key.hashCode()) % N]; // simple but fragile
// Add 1 server: N=4. Now hash % 4 ≠ hash % 3 for ~75% of keys → mass invalidation.

// CONSISTENT HASHING — stable on topology change
String server = ring.getServer(key); // O(log N*K) lookup
// Add 1 server: only ~1/N keys remap. Everything else stays on same server.

// Comparison:
// Keys remapped on add: modulo=~75%,  consistent=~1/N (e.g., ~33% with 3 servers)
// Keys remapped on remove: modulo=~67%, consistent=~1/N
// Memory: modulo=O(N), consistent=O(N*K)
// Lookup: modulo=O(1), consistent=O(log N*K)

Real-World Usage

Apache Cassandra
Partitions rows across nodes using a token ring. Each node owns a token range. vnodes (virtual nodes) enabled by default since v1.2.
Amazon DynamoDB
Inspired by Amazon's Dynamo paper (2007) — the paper that popularised consistent hashing for databases.
Redis Cluster
16,384 hash slots assigned to nodes. A variant of consistent hashing with fixed discrete slots instead of a continuous ring.
Memcached clients
Client-side consistent hashing (ketama algorithm). Client holds the ring locally; no server-side coordination needed.
CDN / Load balancers
Sticky session routing — same client IP consistently routes to same edge server without session state replication.
Peer-to-peer (DHT)
Chord, Kademlia, BitTorrent DHT all use consistent hashing to distribute file lookups across peers.
08

Interview Tips

1
"What happens when a server is added or removed?" This is asked every time. Answer: only keys between the new/removed node and its counter-clockwise predecessor are affected — approximately 1/N of total keys.
2
Always mention virtual nodes unprompted. Without them, servers land unevenly on the ring and one may handle 60–80% of traffic. Virtual nodes (K=150) give statistically uniform distribution.
3
"How do clients know which server owns a key?" Two common answers: (a) client-side ring — client maintains a copy of the ring and computes routing locally (Memcached/ketama); (b) coordinator node — one node knows the ring and forwards requests (Cassandra coordinator).
4
Distinguish Redis Cluster from the classic ring. Redis uses 16,384 discrete hash slots, not a continuous ring. Each node owns a slot range. It's consistent-hashing-inspired but simpler: slot = CRC16(key) % 16384.
5
Hot key problem is separate. Consistent hashing solves data distribution across servers but not hot keys — one viral key still hits one server. Solutions: read replicas for that key, or split the key into shards (user:12345:0..9).
6
TreeMap is your implementation hint. The ring is a sorted map of hash → server. ceilingEntry(hash) finds the next server clockwise in O(log N). Wrap around using firstEntry() if result is null.
