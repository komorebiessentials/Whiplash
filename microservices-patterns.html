01 What Is It

Microservices patterns are proven architectural solutions to the failure modes and operational challenges that emerge when you decompose a system into many small, independently deployable services. Unlike monoliths where a function call is a stack frame, microservices communicate over networks — introducing latency, partial failure, and cascading failures as first-class concerns.

These patterns fall into four categories: resilience (circuit breakers, retries, timeouts), service discovery (how services find each other), traffic management (API gateways, service meshes), and decomposition (how to split the monolith).

02 The Problem It Solves

Cascade Failure

One slow service kills everything

Service A calls B which calls C. C hangs. B's threads fill up waiting for C. A's threads fill up waiting for B. Your entire system times out.

Service Location

How do I find you?

In cloud environments, IPs are ephemeral. Pods restart, scale up/down, shift IPs constantly. You need dynamic service discovery, not hardcoded addresses.

Cross-Cutting Concerns

Auth, tracing, retries — everywhere

Every service needs auth, rate limiting, logging, tracing. Without a mesh or gateway, you duplicate this logic across 50 services.

Distributed Monolith

Microservices without the benefits

Services tightly coupled via synchronous chains or shared DBs — you get all the complexity of distribution with none of the independence benefits.

03 How It Works

Each service has an Envoy sidecar proxy injected by the service mesh control plane (Istiod). All inter-service traffic flows through these sidecars — enabling mTLS encryption, circuit breaking, retries, and distributed tracing without touching application code.

04 Patterns

Resilience Patterns

Circuit Breaker

● Closed — requests pass ● Open — fail fast ● Half-Open — probe

Opens after N% failures in a sliding window (e.g., 50% failure rate in last 10 calls). Waits waitDurationInOpenState (e.g., 30s), then sends one probe request. If it succeeds → Closed. If it fails → Open again. Prevents a slow downstream from exhausting your thread pool while it recovers.

Bulkhead

Isolate failures by allocating a dedicated thread pool or semaphore per downstream service. If calls to the Payment service fill its bulkhead (e.g., 10 concurrent threads), the Order service still serves requests using other thread pools. Prevents one slow dependency from starving all others.

Retry with Exponential Backoff + Jitter

Don't retry immediately — that creates a thundering herd. Formula: wait = min(cap, base × 2^attempt) + random(0, jitter). E.g., base=100ms, cap=30s, jitter=100ms. Jitter spreads retries across time so multiple clients don't all retry in sync.

Timeout

Always set timeouts on every network call. Without a timeout, a thread blocks indefinitely waiting for a dead service, exhausting the thread pool. Set aggressive timeouts (100-500ms for internal calls). Pair with circuit breaker — repeated timeouts open the circuit.

Service Discovery

Client-Side

Eureka / Consul

Client queries registry for available instances, load-balances itself (Ribbon). More control but requires SDK in each client.

Server-Side

AWS ALB + ECS

LB queries registry internally. Client just hits the LB. Zero client-side logic — simpler but less flexible.

DNS-Based

Kubernetes

kube-dns resolves svc.namespace.svc.cluster.local to ClusterIP. kube-proxy routes to pods. Simplest for K8s — no SDK needed.

API Gateway & Service Mesh

API Gateway: Single entry point handling auth, SSL termination, rate limiting, request routing. The BFF (Backend for Frontend) pattern creates separate gateways per client type — mobile app gets a mobile-optimized API, web app gets its own. Downside: becomes a bottleneck and single point of failure if too much logic lives there.

Service Mesh (Istio/Linkerd): Envoy sidecar proxies injected next to every service intercept 100% of traffic. Features handled at infrastructure level: mTLS, circuit breaking, retries, canary traffic splits, distributed tracing. Adds ~10ms latency per hop and significant memory overhead (~50MB per sidecar). Right for 50+ services; overkill for 5.

Strangler Fig Pattern

Incrementally migrate from monolith: build new functionality as microservices, route traffic for that feature via API Gateway to the new service, leaving the monolith alone. Over time, strangle it. Avoids a risky big-bang rewrite.

06 Code Examples

// Resilience4j Circuit Breaker — Spring Boot
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)              // open if 50% fail
    .slidingWindowSize(10)                  // over last 10 calls
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .permittedNumberOfCallsInHalfOpenState(3)
    .build();

// Annotate service method
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
public PaymentResult chargeCard(String userId, long amount) {
    return paymentClient.charge(userId, amount);
}

public PaymentResult paymentFallback(String userId, long amount, Throwable ex) {
    log.warn("Circuit open for user {}, queueing payment", userId);
    pendingQueue.enqueue(new PendingPayment(userId, amount));
    return PaymentResult.queued();
}
// Exponential backoff with jitter
RetryConfig config = RetryConfig.custom()
    .maxAttempts(4)
    .intervalFunction(IntervalFunction
        .ofExponentialRandomBackoff(
            200,          // base ms
            2.0,          // multiplier
            30_000        // cap ms
        ))
    .retryExceptions(ConnectTimeoutException.class,
                     ServiceUnavailableException.class)
    .ignoreExceptions(BadRequestException.class)  // 4xx = don't retry
    .build();

// Wait schedule: 200ms → 400ms → 800ms (+ random jitter each)
// ONLY retry idempotent ops — never retry payment charges without dedup key
// Bulkhead — limit concurrent calls per downstream
BulkheadConfig config = BulkheadConfig.custom()
    .maxConcurrentCalls(10)       // max 10 threads for this service
    .maxWaitDuration(Duration.ofMillis(100))  // wait up to 100ms for a slot
    .build();

@Bulkhead(name = "inventoryService", type = Bulkhead.Type.SEMAPHORE)
public Inventory checkStock(String productId) {
    return inventoryClient.getStock(productId);
}

// If inventory is slow and 10 threads are waiting,
// new calls get BulkheadFullException immediately.
// Order service can still process using other thread pools.
// Combine: Timeout + Circuit Breaker + Fallback
TimeLimiterConfig tlConfig = TimeLimiterConfig.custom()
    .timeoutDuration(Duration.ofMillis(300))
    .build();

@TimeLimiter(name = "recommendationService")
@CircuitBreaker(name = "recommendationService",
                fallbackMethod = "defaultRecommendations")
public CompletableFuture<List<Product>> getRecommendations(String userId) {
    return CompletableFuture.supplyAsync(() ->
        recommendationClient.forUser(userId));
}

public CompletableFuture<List<Product>> defaultRecommendations(String userId, Exception ex) {
    return CompletableFuture.completedFuture(cache.getTopSellers());
}

07 Real-World Usage

Netflix

Hystrix → Resilience4j

Netflix open-sourced Hystrix and pioneered the circuit breaker pattern. Now in maintenance mode; ecosystem moved to Resilience4j.

Kubernetes

DNS + Istio

K8s service DNS for discovery; Istio for mTLS, traffic management, and observability across the mesh. Standard in large cloud-native deployments.

AWS

API Gateway + ALB

AWS API Gateway handles auth, rate limiting, SSL. ALB for server-side service discovery with ECS/EKS target groups. No SDK required in services.

HashiCorp

Consul Service Mesh

Consul provides both service registry and service mesh capabilities. Popular in non-K8s environments (VMs, hybrid cloud).

08 Interview Tips

⚡
"Why not microservices from day one?"
Distributed systems complexity (network failures, distributed tracing, eventual consistency) is substantial. Monoliths are easier to develop, test, and deploy. Extract services when a specific bounded context has clearly different scaling or team ownership needs.
🔗
"What is a distributed monolith?"
Services that can't be deployed independently — they share a database (schema coupling), or call each other synchronously in chains (A→B→C→D). You pay all the costs of distribution (network, observability) with none of the benefits (independent deployability, scaling).
🗄
"How do you handle shared database anti-pattern?"
Each service owns its data (database-per-service). For cross-service reads, use CQRS read models or API calls. For writes that span services, use Saga + outbox pattern. Accept eventual consistency — it's the right tradeoff.
🌊
"Walk me through a cascade failure and how you'd prevent it."
DB gets slow → query timeouts → threads pile up waiting → thread pool exhausted → service unresponsive → upstream callers also time out → cascade. Defense: timeout (bound wait) → bulkhead (isolate failure) → circuit breaker (stop calling the dead service) → fallback (serve degraded response).
🔐
"How does mTLS work in a service mesh?"
Control plane (Istiod) acts as a CA, issuing SPIFFE X.509 certs to each sidecar. On every connection, sidecars exchange certs and verify the peer's identity. Even if you're inside the VPC, a compromised service can't impersonate another service.
