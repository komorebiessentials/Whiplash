Concurrency &
Thread Safety

Primitives for safe concurrent access, deadlock conditions, and the patterns that appear in every Java interview.

1. What Is It

Concurrency is the ability for multiple threads to make progress simultaneously, sharing CPU time and memory. Thread safety means a piece of code behaves correctly when multiple threads execute it at the same time — without data corruption, visibility bugs, or race conditions.

Java provides a layered toolkit: language-level synchronized, explicit Lock objects, non-blocking atomics, and high-level concurrent collections in java.util.concurrent.

2. The Problem It Solves

Without synchronization, multiple threads reading and writing shared state produce non-deterministic results. Two classic bugs:

Race Condition
Two threads both read counter=5, both increment to 6, both write 6. Net result: one increment lost. Expected: 7.
Visibility Bug
Thread A writes a flag to CPU cache. Thread B never sees it because it reads a stale value from its own cache line.

3. How It Works

Threads compete to acquire a lock (monitor). Only the holder can enter the critical section. When it exits, one waiting thread is unblocked.

The Java Memory Model (JMM) guarantees that actions inside a synchronized block happen-before actions after the lock is released — ensuring both atomicity and visibility.

4. Primitives & Variants

Intrinsic Lock
synchronized
Every Java object has a built-in monitor. synchronized on a method or block acquires it automatically and releases on exit (even on exception).
Coarse-grained
Explicit Lock
ReentrantLock
Same semantics as synchronized but adds tryLock(), timed acquisition, and a fairness flag (FIFO ordering).
Fine-grained
Read/Write Split
ReadWriteLock
Allows unlimited concurrent readers OR one exclusive writer. Great when reads vastly outnumber writes (caches, config).
High Read Throughput
Visibility Only
volatile
Guarantees that writes are visible to all threads immediately. Does not provide atomicity — volatile int x; x++ is still a race.
No Atomicity
Lock-Free
AtomicInteger / CAS
Uses CPU-level Compare-And-Swap. No lock acquisition — extremely fast under low contention. AtomicInteger, AtomicLong, AtomicReference.
Lock-Free
Counting
Semaphore
Allows N concurrent threads to access a resource simultaneously. Useful for connection pools, rate limiting concurrent threads.
N-Permit

5. Trade-offs

Primitive Granularity Deadlock Risk Performance Use When
synchronized Method / block Yes Good (JVM-optimized) Simple critical sections, low contention
ReentrantLock Block / manual Yes (must unlock) Similar to synchronized Need tryLock, timed lock, or fairness
ReadWriteLock Read / write split Yes (write starvation) High (concurrent reads) Read-heavy workloads (caches)
volatile Field No Very fast Single-writer, many-reader flags
AtomicInteger Field (int/long/ref) No Excellent (low contention) Counters, sequence numbers, CAS loops

6. Code

// Approach 1 — synchronized method
public class SyncCounter {
    private int count = 0;
    public synchronized void increment() { count++; }
    public synchronized int get() { return count; }
}

// Approach 2 — AtomicInteger (lock-free)
public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);
    public void increment() { count.incrementAndGet(); }
    public int get()       { return count.get(); }
}

// Approach 3 — LongAdder (best throughput under high contention)
public class AdderCounter {
    private final LongAdder count = new LongAdder();
    public void increment() { count.increment(); }
    public long get()       { return count.sum(); }
    // LongAdder splits across CPU cells — reduces CAS contention
}
import java.util.concurrent.*;

public class ProducerConsumerDemo {
    private static final BlockingQueue<String> queue =
        new ArrayBlockingQueue<>(10); // bounded

    static class Producer implements Runnable {
        public void run() {
            try {
                queue.put("item"); // blocks if queue full
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    static class Consumer implements Runnable {
        public void run() {
            try {
                String item = queue.take(); // blocks if queue empty
                System.out.println("Consumed: " + item);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
import java.util.concurrent.locks.*;

public class CachedData {
    private String data;
    private final ReadWriteLock rwLock =
        new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    public String read() {
        readLock.lock();           // many readers OK
        try { return data; }
        finally { readLock.unlock(); }
    }

    public void write(String newData) {
        writeLock.lock();          // exclusive — blocks all readers
        try { data = newData; }
        finally { writeLock.unlock(); }
    }
}
// DEADLOCK scenario: T1 holds A, waits B; T2 holds B, waits A
// FIX: always acquire locks in a consistent global order

public class DeadlockFree {
    private final Object lockA;
    private final Object lockB;

    public DeadlockFree(Object a, Object b) {
        // Enforce order by System.identityHashCode
        if (System.identityHashCode(a) < System.identityHashCode(b)) {
            lockA = a; lockB = b;
        } else {
            lockA = b; lockB = a;
        }
    }

    public void transfer() {
        synchronized (lockA) {         // always A first
            synchronized (lockB) {     // then B
                // safe to operate on both
            }
        }
    }
}

Deadlock: 4 Necessary Conditions

All four must hold simultaneously for a deadlock to occur. Break any one to prevent it.

01
Mutual Exclusion
A resource can only be held by one thread at a time. (Inherent to locks — can't eliminate.)
02
Hold and Wait
A thread holds at least one resource and waits for more. Fix: request all resources atomically upfront.
03
No Preemption
Locks can't be forcibly taken. Fix: use tryLock(timeout) to give up and retry.
04
Circular Wait
T1→A→B→T2→A forms a cycle. Fix: impose a total lock ordering (always acquire in same sequence).

7. Real-World Usage

ConcurrentHashMap
Thread-safe map using segment-level (Java 7) or CAS + bucket-level (Java 8+) locking. Never use HashMap across threads.
CopyOnWriteArrayList
Every write makes a copy. Reads are lock-free. Ideal for read-heavy lists (event listeners). Bad for frequent writes.
ExecutorService
Thread pool — reuse threads rather than creating new ones. Use Executors.newFixedThreadPool(n) or custom ThreadPoolExecutor.
Connection Pools
HikariCP uses a Semaphore to limit concurrent DB connections. Threads block until a slot is released.

8. Interview Tips

🎯
Default to AtomicInteger for counters — it's lock-free and the interviewer will notice you know the better tool. Mention LongAdder for even higher contention.
⚠️
volatile ≠ atomic — a classic trap. volatile int x; x++ is still a race because increment is read-modify-write (3 operations).
🔒
Always release locks in finally — lock.lock(); try { ... } finally { lock.unlock(); }. Forgetting this causes lock leaks on exceptions.
📐
Name the 4 deadlock conditions — then explain which one you break and how. Lock ordering (break circular wait) is the most common answer.
📖
ReadWriteLock is a signal — if you use it in parking lot or cache problems, interviewers know you understand read-heavy vs write-heavy access patterns.
