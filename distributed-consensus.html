1. What Is It

Distributed consensus is the problem of getting a cluster of nodes to agree on a single value (or sequence of values) — even when some nodes crash, messages are delayed, or the network partitions. It's the foundation of any strongly-consistent distributed system: replicated state machines, distributed databases, and configuration stores.

The fundamental challenge: In a distributed system, there is no global clock and messages can arrive out of order or not at all. Without a consensus protocol, two nodes could simultaneously believe they are the leader and accept conflicting writes — split-brain.
Raft
Understandable consensus
Designed explicitly for understandability. Separate concerns: leader election, log replication, safety. The protocol used in etcd, Consul, CockroachDB, Kafka KRaft.
Paxos
The original protocol
Lamport's original consensus algorithm. Notoriously hard to implement correctly. Multi-Paxos (optimized form) is the basis for Google Chubby, Spanner.
ZAB
ZooKeeper Atomic Broadcast
ZooKeeper's variant. Epoch-based, similar to Raft. Used by Kafka (ZooKeeper mode) for metadata management before KRaft.

2. The Problem It Solves

Consider a 3-node replicated database. A client writes a value. Node 1 and Node 2 receive it; Node 3 is unreachable. Is this write committed? Without consensus, you can't know — if Node 1 crashes now, did Node 2 commit? Did they agree?

Consensus solves this by requiring a majority quorum (⌊N/2⌋ + 1 nodes) to agree on each operation before it's considered committed. With 3 nodes, any 2 must agree. This tolerates 1 failure. With 5 nodes, any 3 must agree — tolerates 2 failures.

FLP Impossibility: In an asynchronous system with even one faulty process, it's impossible to guarantee consensus. Raft and Paxos work around this by using randomized timeouts and making liveness probabilistic — they guarantee safety always, but liveness only under "sufficiently synchronous" conditions.

The distributed consensus problem also prevents split-brain: a scenario where a network partition creates two groups of nodes that each believe they are the authoritative cluster and accept conflicting writes. Requiring majority quorum ensures only one partition (the one with more than half the nodes) can make progress.

3. How Raft Works

Log Replication (the happy path)

1
Client writes to Leader: Client sends write request to any node. If not the leader, the node redirects. Leader receives the request.
2
Leader appends to its log: Entry is added with current term number. Not yet committed.
3
AppendEntries RPC to all followers: Leader sends the new log entry in parallel. Followers append to their local logs and acknowledge.
4
Commit once majority ACK: Once ⌊N/2⌋+1 nodes acknowledge, the Leader commits the entry and applies it to its state machine. Updates commitIndex.
5
Notify followers to commit: Next AppendEntries (or heartbeat) includes updated commitIndex. Followers apply committed entries to their state machines.

4. Raft, Paxos, and Quorums

Raft — Key Safety Properties

Election Safety
At most one leader per term
A candidate must get votes from a majority. Since each node votes for at most one candidate per term, only one candidate can achieve majority — impossible to have two leaders in the same term.
Log Completeness
Winners have all committed entries
A candidate can only win if its log is at least as up-to-date as any node in the majority that votes for it. Since committed entries exist on a majority, any winner must have all committed entries.
State Machine Safety
All nodes apply same entries in order
If any node applies log entry at index i, no other node can apply a different entry at index i. Combined with log completeness → all state machines are identical.

Paxos — Two Phases

Phase 1 — Prepare: Proposer sends Prepare(n) to quorum. Each Acceptor promises not to accept any proposal numbered < n, and returns the highest-numbered proposal it has already accepted (if any). Proposer collects these responses.

Phase 2 — Accept: If the Proposer got promises from a majority, it sends Accept(n, v) where v is the value from the highest-numbered accepted proposal returned (or its own value if none). Acceptors accept if they haven't promised a higher number. When a majority accepts, the value is chosen.

Multi-Paxos optimization: Elect a stable Proposer (leader) and skip Phase 1 for subsequent proposals within a term. Now it looks very similar to Raft — this is why real implementations converge on the same shape.

Quorum Read/Write Tuning

Rule: W + R > N guarantees at least one overlap node between write set and read set → reads always see latest write.

N=3 examples:
• W=1, R=1 → no guarantee (fast, eventual consistency — Cassandra ONE)
• W=2, R=2 → W+R=4 > 3 ✓ (Cassandra QUORUM)
• W=3, R=1 → W+R=4 > 3 ✓ (max write durability, fast reads)
• W=1, R=3 → W+R=4 > 3 ✓ (fast writes, expensive reads)

5. Trade-offs

AlgorithmUnderstandabilityPerformanceMembership ChangesUsed In
Raft High — designed for it Good Joint consensus or single-server changes etcd, Consul, CockroachDB, Kafka KRaft
Multi-Paxos Very low Excellent (skip Phase 1) Complex Google Chubby, Spanner, Zab (ZAB variant)
ZAB Moderate Good Epoch-based ZooKeeper (Kafka legacy metadata)
Viewstamped Replication Moderate Good View changes Academic, influenced Raft
CAP impact: All consensus protocols (Raft, Paxos, ZAB) are CP — they prioritize Consistency over Availability. During a leader election (after crash), the cluster is unavailable for writes for the duration of the election timeout (typically 150–300ms for Raft). This is an acceptable tradeoff for configuration stores, but not for user-facing APIs.

6. Code Examples

class RaftLeader {
  private List<LogEntry> log = new ArrayList<>();
  private int commitIndex = 0;
  private List<RaftPeer> peers;
  private int currentTerm;

  public CompletableFuture<Boolean> appendEntry(String command) {
    // 1. Append to own log
    LogEntry entry = new LogEntry(currentTerm, log.size(), command);
    log.add(entry);
    int entryIndex = log.size() - 1;

    // 2. Send AppendEntries RPC to all followers in parallel
    List<CompletableFuture<Boolean>> futures = peers.stream()
      .map(peer -> peer.appendEntries(currentTerm, entry))
      .collect(Collectors.toList());

    // 3. Wait for majority ack (quorum = floor(N/2) + 1)
    int quorum = peers.size() / 2 + 1; // includes self
    return waitForMajority(futures, quorum).thenApply(success -> {
      if (success) {
        // 4. Commit — apply to state machine, advance commitIndex
        commitIndex = entryIndex;
        applyToStateMachine(entry);
        // 5. Next heartbeat will carry updated commitIndex to followers
      }
      return success;
    });
  }

  private void applyToStateMachine(LogEntry entry) {
    // Execute the command (e.g., set key=value in distributed KV store)
    stateMachine.apply(entry.command);
  }
}

class RaftNode {
  enum Role { FOLLOWER, CANDIDATE, LEADER }
  private Role role = Role.FOLLOWER;
  private int currentTerm = 0;
  private String votedFor = null; // persisted to disk!
  private int votesReceived = 0;

  void onElectionTimeout() {
    // Election timeout fires if no heartbeat received (150-300ms random)
    role = Role.CANDIDATE;
    currentTerm++;           // increment term
    votedFor = this.nodeId; // vote for self
    votesReceived = 1;      // count own vote
    persistState();          // MUST persist term+votedFor before sending RPCs

    // Send RequestVote to all other nodes in parallel
    peers.forEach(peer -> peer.requestVote(
      currentTerm, nodeId, log.lastIndex(), log.lastTerm()
    ).thenAccept(this::handleVoteResponse));
  }

  synchronized void handleVoteResponse(VoteResponse resp) {
    if (role != Role.CANDIDATE) return; // already elected or stepped down
    if (resp.term > currentTerm) {
      // Stale — higher term node exists, step down
      currentTerm = resp.term;
      role = Role.FOLLOWER;
      votedFor = null;
      persistState();
      return;
    }
    if (resp.voteGranted) {
      votesReceived++;
      if (votesReceived >= peers.size() / 2 + 1) {
        becomeLeader(); // won the election!
      }
    }
  }

  void handleVoteRequest(VoteRequest req) {
    // Only grant vote if: candidate term >= ours AND
    // we haven't voted this term AND candidate's log is up-to-date
    boolean logOk = req.lastLogTerm > log.lastTerm() ||
      (req.lastLogTerm == log.lastTerm() && req.lastLogIndex >= log.lastIndex());
    boolean grant = req.term >= currentTerm
      && (votedFor == null || votedFor.equals(req.candidateId))
      && logOk;
    if (grant) { votedFor = req.candidateId; persistState(); }
    peer.sendVoteResponse(new VoteResponse(currentTerm, grant));
  }
}

class PaxosProposer {
  private int proposalNumber = 0;
  private List<PaxosAcceptor> acceptors;
  private int quorum;

  // Phase 1: Prepare — "promise not to accept lower-numbered proposals"
  Object propose(Object myValue) {
    proposalNumber++;
    int n = proposalNumber;
    // uniqueness: proposalNumber = (round * numProposers) + proposerId

    // Send Prepare(n) to quorum
    List<Promise> promises = acceptors.stream()
      .map(a -> a.prepare(n))
      .filter(Objects::nonNull)  // null = rejected (promised higher n)
      .collect(Collectors.toList());

    if (promises.size() < quorum) return null; // failed, retry with higher n

    // Phase 2: Accept — use highest-accepted value from promises, or our own
    Object value = promises.stream()
      .filter(p -> p.acceptedValue != null)
      .max(Comparator.comparingInt(p -> p.acceptedProposal))
      .map(p -> p.acceptedValue)
      .orElse(myValue); // no prior accepted value → use our own

    long acceptCount = acceptors.stream()
      .filter(a -> a.accept(n, value)) // returns true if accepted
      .count();

    return acceptCount >= quorum ? value : null; // consensus reached!
  }
}

class PaxosAcceptor {
  private int promisedN = -1;
  private int acceptedN = -1;
  private Object acceptedValue = null;

  Promise prepare(int n) {
    if (n <= promisedN) return null; // reject — already promised higher
    promisedN = n; persist(); // MUST persist before responding
    return new Promise(acceptedN, acceptedValue); // tell proposer what we've seen
  }

  boolean accept(int n, Object value) {
    if (n < promisedN) return false; // reject — promised not to accept < promisedN
    acceptedN = n; acceptedValue = value; persist();
    return true;
  }
}

class QuorumConfig {
  private final int n; // total nodes

  public QuorumConfig(int n) { this.n = n; }

  public int quorum() { return n / 2 + 1; } // floor(N/2) + 1
  public int maxFailures() { return n / 2; }   // N - quorum = floor(N/2)

  // W + R > N for overlap guarantee
  public boolean isConsistentReadWrite(int w, int r) {
    return w + r > n;
  }

  public void printConfigs() {
    System.out.printf("N=%d: quorum=%d, max failures=%d%n", n, quorum(), maxFailures());
    // N=3: quorum=2, max failures=1  → recommend for most use cases
    // N=5: quorum=3, max failures=2  → for higher availability
    // N=7: quorum=4, max failures=3  → geo-distributed across 3 regions
  }
}

7. Real-World Usage

etcd
Raft — Kubernetes backbone
Stores all Kubernetes cluster state (pods, services, config). 3 or 5 node Raft cluster. Typical election timeout: 1s. Critical — etcd downtime = k8s control plane downtime. Recommends NVMe SSDs for WAL latency.
CockroachDB
Raft per range
Data split into 64MB ranges, each with its own Raft group. 3 replicas per range by default. Allows fine-grained replication and failover at range level, not database level.
Kafka KRaft
Raft replacing ZooKeeper
Kafka 3.x+ uses KRaft (Kafka Raft) for controller metadata. Eliminates ZooKeeper dependency. Supports 3-node quorum controllers. Enables clusters with millions of partitions.
Google Spanner
Paxos + TrueTime
Uses Paxos across datacenters for globally-consistent transactions. TrueTime API gives bounded clock uncertainty — Spanner waits out the uncertainty before committing to guarantee external consistency.

8. Interview Tips

💡
"How does Raft prevent split-brain?" A leader can only be elected with a majority (quorum) of votes. Since a majority requires more than half the nodes, only one group can form a quorum in any partition. Similarly, commits require a majority ACK — a partitioned leader can't commit without the majority.
💡
"What happens if the leader crashes after appending to its log but before a majority ACK?" The entry is uncommitted. The new leader (elected by majority) will have an up-to-date log. If the new leader has the entry, it will commit it in the next write (leaders commit previous-term entries indirectly). If not, the entry is rolled back on the crashed node when it rejoins and syncs from the new leader.
💡
"Why do most production systems use Raft over Paxos?" The Raft paper explicitly optimized for understandability, which translates to fewer implementation bugs. Multi-Paxos requires many engineering decisions not specified in the original paper (leader election mechanism, membership changes, log compaction) — each implementation differs. Raft specifies all of these.
💡
"What's the CAP tradeoff for Raft?" CP — Raft prioritizes Consistency. During leader election, the cluster is unavailable for writes (typically 150–600ms). This is acceptable for config stores and coordination services, but not for user-facing APIs. That's why you'd use Cassandra (AP, tunable) for a user-facing service, not etcd.
💡
"Why must votedFor be persisted to disk before sending a vote response?" If a node votes, crashes, and restarts, it must remember it already voted in this term. Otherwise it might vote twice — once before the crash and once after restart — potentially allowing two leaders in the same term. Persistent state = currentTerm + votedFor + log entries.
