Requirements

Functional
  • Upload and download files (up to 5 GB each)
  • Organize files in folders
  • Share files/folders with view or edit permissions
  • Sync changes across multiple devices
  • Version history — restore previous versions
  • Collaborate on documents in real-time
  • Search files by name and content
Non-Functional
  • 1 billion users worldwide
  • 99.999% durability (11 nines for stored data)
  • Sync latency < 1s for small files
  • Resumable uploads for large files
  • Block-level deduplication to save storage
  • Strong consistency for permission changes
  • Data encrypted at rest and in transit

Key Design Decisions

Chunked Upload & Dedup

  • Split file into 4MB blocks; compute SHA-256 per block
  • Before uploading, check: does block_hash exist in Cassandra?
  • If yes → skip upload; just record block reference (massive savings)
  • If no → presigned S3 URL → client uploads directly to S3
  • Last block can be < 4MB (variable); stored with actual size
  • Result: a 1GB OS update shared by millions users → stored once

Sync Algorithm

  • Client maintains local manifest: {path → (size, modified_at, checksum)}
  • On file change (inotify/FSEvents): compute block hashes of changed regions
  • Query server for which blocks already exist → upload only new ones
  • Server sends sync events via Kafka → other devices pull delta
  • Polling fallback: GET /sync/delta?since=timestamp returns changed files
  • Conflict: both client and server changed same file → create conflict copy

Version History

  • Each upload creates new version record (not copy of blocks)
  • Version stores ordered list of block_hashes (the file's block manifest)
  • Restore: update current_version pointer → instant, no data copy
  • Storage cost of versions: only new/changed blocks stored
  • Retention policy: keep last 100 versions or 30 days, whichever is more

Permission Inheritance

  • Files inherit permissions from parent folder recursively
  • Explicit file permission overrides inherited folder permission
  • Permission check: traverse up folder tree until permission found
  • Optimization: cache resolved permissions in Redis with short TTL
  • Share link: generates token → stored with permission level + expiry
  • Strong consistency: permission grant must be visible immediately (Spanner)
// Client-side sync dedup logic (pseudocode)
public List<BlockHash> computeFileBlocks(File file) {
    List<BlockHash> blocks = new ArrayList<>();
    int CHUNK_SIZE = 4 * 1024 * 1024; // 4MB
    try (InputStream is = new FileInputStream(file)) {
        byte[] buf = new byte[CHUNK_SIZE];
        int read;
        while ((read = is.read(buf)) != -1) {
            blocks.add(SHA256.hash(buf, 0, read));
        }
    }
    return blocks;
}

public void uploadFile(File file) {
    List<BlockHash> blocks = computeFileBlocks(file);
    // Ask server which blocks are missing
    Set<BlockHash> missing = api.checkBlocks(blocks);
    // Upload only missing blocks via presigned S3 URLs
    missing.forEach(hash -> s3.upload(hash, api.getPresignedUrl(hash)));
    api.completeUpload(file.name, blocks); // register file manifest
}

Scaling & Bottlenecks

🌐

CDN for Downloads

File blocks cached at CDN edge. Download redirects to CDN URL. Popular shared files served at edge — zero origin load.

📤

Presigned S3 Upload

Client uploads chunks directly to S3 using presigned URLs — bypasses app servers entirely. Scales to 11,500 uploads/sec without bottleneck.

🗜️

Block Deduplication

Same 4MB block stored once globally. Popular files (OS updates, office templates) shared by millions — stored as one block. Estimated 40% storage savings.

🌍

Spanner for Global ACL

Permission changes must be globally consistent. Google Spanner provides distributed ACID transactions with external consistency — permission visible in <10ms globally.

📊

Kafka for Sync Events

File change → Kafka → sync events to all user's devices. Each device maintains a consumer offset → picks up changes from last sync point.

🔍

Elasticsearch for Search

File name and content indexed in Elasticsearch. On upload completion, Kafka event → indexer. Search returns within 100ms across 1 trillion file records.

Self-Check

Test your understanding. Click to reveal answers.

Why chunk files instead of uploading them whole? Core Design +

Three reasons: (1) Resumability — if a 1GB upload fails at 95%, resume from the last successful chunk rather than restarting from zero. (2) Delta sync — only changed chunks need uploading on subsequent saves. A small edit to a 200MB file uploads 4-8MB, not 200MB. (3) Deduplication — chunks are content-addressed by SHA-256. If 1000 users share the same PDF, only one copy of each chunk is stored in S3. This reduces storage by 30-50% across the fleet. Chunk size tradeoff: too small (1KB) = too many metadata records and HTTP round-trips; too large (100MB) = coarse-grained delta sync, long resume windows. 4MB is the standard sweet spot.

How does conflict resolution work when two users edit offline? Conflict Resolution +

Google Drive uses last-write-wins for file metadata but creates a 'conflict copy' when it detects a fork. Detection: each file version has a monotonic version number. If two clients independently edit version 5 and both try to push version 6, the second push sees that version 6 already exists. Drive creates 'filename (Joe's conflicting copy 2024-01-15).' Both versions exist; user must manually merge. Google Docs avoids this entirely with Operational Transformation (OT) — every keystroke is a delta operation that can be merged algorithmically. CRDTs (Conflict-free Replicated Data Types) are a modern alternative that enables automatic merge without coordination.

How does sharing/permissions work at scale? Permissions +

Files have an ACL (Access Control List) in MySQL: (file_id, principal_id, permission_level). Folder permissions inherit to children. Permission check on file access: (1) Is requester the owner? (2) Direct ACL entry for this file? (3) Walk up folder tree for inherited permission. Tree walking is expensive for deep nesting. Optimization: cache effective permissions in Redis with 5-min TTL. 'Share with link': generates a share token stored in DB; anyone with the token gets access. Permission checks are on the critical read path — keep them sub-10ms.

What happens when a popular shared file is downloaded by 10,000 people simultaneously? Scale +

Without CDN: 10,000 S3 GETs × 4MB chunks = 40GB egress from one region — expensive and slow. With CDN: first request fetches from S3 and caches at edge. Remaining 9,999 requests served from CDN edge at ~10ms. For truly viral files (company-wide announcements), Drive can proactively push to CDN before the flood. CDN cache TTL: short (5 min) for actively edited files; longer for stable files. Cache invalidation: on file update, purge CDN cache for that file's chunk URLs (or use content-addressed URLs that auto-invalidate because the chunk hash changes).

How do you estimate Google Drive's storage at scale? Scale Estimation +

~1 billion users × 15GB free tier = 15 exabytes raw. After ~30% deduplication: ~10.5EB. After 3× replication for durability: ~31EB. Storage cost at $0.023/GB/year: 31 × 10⁹ GB × $0.023 = ~$713M/year just for storage. This is cross-subsidized by paid Workspace plans. Per-user breakdown: 1B users, ~20% are monthly active uploaders, average 2GB uploaded/month = 2 exabytes new data per month. At 4MB chunks: 500 billion chunk writes per month. S3 can handle this with sharding across regions.

Interview Tips

🧱
Chunking + dedup is the hero concept. A 100MB file split into 4MB blocks where most blocks already exist = only 20MB uploaded. This drives both upload speed and storage savings. Walk through the algorithm clearly: hash → check → upload missing → register manifest.
🔒
Permission consistency is non-negotiable. If you share a file with someone, they must see it immediately. Use Spanner (or a strongly consistent DB) for the ACL table. Caching permissions in Redis is fine with a short TTL, but the source of truth must be strongly consistent.
🔄
Sync vs backup are different problems. Sync = bidirectional, real-time, conflict resolution needed. Backup = one-way, periodic, no conflicts. Google Drive is sync + backup. Draw the sync algorithm clearly: local change detection (inotify) → block diff → upload deltas → propagate to other devices.
📂
Folder hierarchy needs a design decision. Option 1: adjacency list (parent_folder_id FK) — simple but recursive queries are slow. Option 2: materialized path (store /a/b/c as string) — fast prefix queries. Option 3: closure table. For Google Drive scale, adjacency list + caching works fine.
📐
Scale anchor. 1B users × 15 GB = 15 EB. At $0.023/GB/month (S3), that's $345B/year — why Google charges for storage above 15 GB. Dedup saves ~40%, so effective storage is ~9 EB. CDN shifts 80% of download traffic away from origin. These numbers justify the architecture choices.
