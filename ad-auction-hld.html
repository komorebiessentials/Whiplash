6. Database Design

-- PostgreSQL: campaigns + creatives
CREATE TABLE campaigns (
  campaign_id   UUID PRIMARY KEY,
  daily_budget  NUMERIC(12,4),
  targeting     JSONB,              -- {geo:[], age:[], interests:[]}
  status        VARCHAR(20)         -- active/paused/budget_exhausted
);

-- Redis: hot-path data structures
-- Targeting inverted index (sorted set, score = max bid)
ZADD targeting:geo:US 3.50 campaign-uuid-1
ZADD targeting:interest:sports 4.10 campaign-uuid-2

-- User profile (pre-computed offline by Spark, cached in Redis)
HSET user:hashed-uid interests "[sports,fitness]" age_bucket "25-34" geo "US"

-- Frequency cap (atomic counter with 24h TTL)
INCR  user:hashed-uid:ad:ad-uuid-1:count
EXPIRE user:hashed-uid:ad:ad-uuid-1:count 86400

-- Budget token bucket (refilled every minute by background job)
DECRBY campaign:uuid-1:budget_tokens 1   -- consume on each impression win

7. Key Design Decisions

Auction Time Budget (<100ms total)

Every millisecond counts. Here's how the budget is allocated:

Parse + validate request      :  2ms
User profile lookup (Redis)   :  3ms
Fraud / IVT scoring           :  8ms
Targeting candidate retrieval : 12ms  (inverted index intersection)
Frequency cap check (Redis)   :  3ms
Budget check (Redis DECR)     :  2ms
Auction ranking (in memory)   :  3ms
Ad creative lookup + response :  7ms
Kafka emit (async, fire+fgt)  :  0ms  (non-blocking)
                              ------
Total                         : ~40ms  (60ms headroom)

All heavy I/O happens before the auction runs (targeting, user lookup). The auction itself is pure CPU — sorting 100 bids takes microseconds.

Targeting with Inverted Index

Evaluating all 10M campaigns per request is impossible at 1M req/sec. Instead: Redis sorted sets keyed by targeting dimension, scored by max bid. For a request with {geo:US, device:mobile, interest:sports}: ZREVRANGEBYSCORE on each dimension → intersect → top-100 candidates. This reduces 10M → 100 candidates in ~12ms. The union/intersection is done in Lua on Redis to avoid multiple round-trips.

2nd-Price Auction

Winner pays 2nd-highest bid + $0.01. This is incentive-compatible: bidding your true value is the dominant strategy. Even if you overbid, you won't overpay (you pay what the next person was willing to pay). Encourages honest participation in the market. Google moved to 1st-price in 2019 — simpler for advertisers to understand, but requires bid-shading algorithms on the buy side to avoid overpaying.

Quality Score: effective_CPM = bid × predicted_CTR × predicted_CVR. A $5 bid with 2% CTR beats a $8 bid with 0.5% CTR. The exchange shows more relevant ads and earns more revenue long-term.

Budget Pacing with Token Bucket

Naive approach: check spent_today < daily_budget before each auction. Problem: 1M auctions/sec → budget exhausted in seconds at peak morning traffic. Token bucket solution: each campaign gets daily_budget/1440 tokens per minute (refilled by background job). Redis DECRBY is atomic — thread-safe across 100 auction servers. When tokens hit 0, campaign excluded until next refill. Smooth pacing: Grafana shows steady spend curve, not front-loaded spike.

8. Scaling & Bottlenecks

Auction Servers

Stateless — horizontally scalable. All per-request state in Redis. LB distributes evenly. Auto-scale on QPS metric.

Redis Cluster

User profiles + targeting index sharded by key hash. 8-16 shards × 32GB each. Read replicas for user lookups (read-heavy, write-rare).

Kafka Async Logging

Bid logs async (fire-and-forget from auction perspective). Kafka consumers write to S3 + ClickHouse without blocking auctions.

Hot Campaign Cache

Top 1000 campaigns appear in 80% of auctions. Pre-fetch to local in-memory cache on auction servers (5-min TTL). Avoids Redis on critical path for these.

Global Deployment

Deploy per-region: US-East, EU-West, Asia-Pacific. User profiles geo-replicated. Ad requests served from nearest region (<10ms network). Cross-region only for reporting.

ClickHouse Reporting

Pre-aggregate hourly rollups via Flink. Campaign dashboards query rollups (<1s), not raw bid logs. Raw logs on S3 for audit.

9. Self-Check

Test your understanding. Click to reveal answers.

Why 2nd-price (Vickrey) instead of 1st-price auction? Auction Theory +

In a 1st-price auction, the winner pays their own bid — this incentivizes advertisers to shade their bids downward (bid less than their true value), creating a complex game theory problem. You never know if bidding $4 when your true value is $5 is the right strategy, because it depends on what others are bidding. In a 2nd-price auction, the dominant strategy (optimal regardless of what others do) is to bid your true value: if you win, you pay the 2nd-highest bid + $0.01, not your own bid. This simplifies the market and leads to better price discovery. Note: Google moved to 1st-price in 2019 partly to unify their different auction types and increase transparency for buyers.

How do you prevent an advertiser from spending their entire budget at 9am? Budget Pacing +

Token bucket pacing: divide daily budget into 1440 × 1-minute intervals. Each interval budget = daily_budget / 1440, adjusted by expected traffic patterns (more traffic at noon than 2am). Redis stores token count per campaign. Each impression win decrements the token count atomically (DECRBY). Background job refills tokens every minute. If tokens hit 0, campaign excluded from auctions that minute. Advanced: historical traffic curves used to distribute budget proportionally — 80% of traffic between 8am-8pm, so allocate 80% of budget there. Grafana dashboards show real-time spend rate vs budget curve.

Targeting engine takes 200ms — we need to fit in 100ms total. What do you do? Latency +

Pre-compute user profiles offline (Spark job every 30 minutes) and cache in Redis. On ad request, it's a single Redis GET (<2ms) not real-time ML inference. The inverted index (targeting:geo:US sorted set) is also in Redis — intersecting 3 dimensions takes <10ms. Fallback: if Redis miss (new user), serve contextual ads based on page URL/category only (no user targeting). Cascade timeout: any Redis call taking >20ms → use cached 'default' candidate list. Audit later. The auction must never fail to respond — showing a house ad is better than showing nothing.

How does click fraud detection work? Fraud +

Multi-layer approach: (1) Real-time: Redis counter per (userId, adId, hour) — if clicks > 3 in 1 hour, flag as suspicious; IP rate limits. (2) ML scoring: every click → Kafka → fraud model (features: click velocity, device fingerprint, time since impression, IP reputation, user agent). Score <0.5 = fraudulent. (3) Retroactive: daily Spark job runs anomaly detection on click patterns (click farms show identical time intervals, coordinated IPs). Fraudulent clicks: not billed to advertiser, not counted in conversion reports. Publishers with high fraud rates: suspended from exchange. Industry: Google estimates 10-15% of ad clicks are fraudulent before filtering.

How do you attribute a purchase to an ad impression? Attribution +

Attribution chain: impression logged (impressionId, userId, campaignId, timestamp) → if user purchases within conversion window (7 days by default) → look up their last N ad impressions in ClickHouse → apply attribution model. Models: (1) Last-click: attribute to the last ad the user clicked. Simple, favors bottom-of-funnel. (2) First-click: attribute to the first ad. Favors awareness campaigns. (3) Linear: split credit equally. (4) Data-driven: ML model weights impressions by their actual influence (trained on conversion data). Identity resolution: user may see ad on mobile, buy on desktop → probabilistic matching by email, device graph (hashed). Cross-device attribution reduces under-counting.
