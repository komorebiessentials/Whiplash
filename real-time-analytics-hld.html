Storage Architecture

Kafka (Ingest Buffer)

Topics per event type, 7-day retention for Kappa replay. Partitioned by userId for ordering. 20+ partitions for 1M/sec throughput.

ClickHouse (Hot OLAP)

Columnar store, pre-aggregated rollups, bitmap indexes on dimensions. Sub-second aggregation queries over billions of rows. Last 7–30 days hot tier.

S3 + Parquet (Cold Lake)

All raw events as Parquet files partitioned by date/hour. Cheap at scale. Queried by Spark/Trino for historical analysis. Indefinite retention.

Redis (Real-time Counters)

INCR per (event_type, minute) bucket. Pre-computed dashboard metrics cached with 10s TTL. Reduces ClickHouse load for hot queries.

PostgreSQL (Metadata)

Metric definitions, alert rules, experiment configs, user/team management. Transactional, low volume.

Key Design Decisions

Exactly-Once with Flink: Flink checkpoints its processing state to S3 every 30 seconds. On failure, it restarts from the last checkpoint and re-reads from the saved Kafka offset. For exactly-once: Kafka consumer offsets are committed only after the checkpoint succeeds. The sink (ClickHouse) must support idempotent writes (overwrite on duplicate key).

Tumbling Window

Fixed non-overlapping 1-minute buckets. Simple, no overlap. Use for time-series metrics like "events per minute."

Sliding Window

Overlapping windows (5-min window, every 1 min). Smoother trends. 5× more computation. Use for moving averages.

Session Window

Gap-based: window ends after N minutes of inactivity. Perfect for user sessions. Variable-length windows.

Scaling & Bottlenecks

Kafka Partitioning

Each partition processed by one Flink task. More partitions = more parallelism. Add partitions to scale at 1:1 with Flink task slots.

Flink Scaling

Add TaskManager instances. RocksDB state backend for large keyed state. Incremental checkpointing reduces S3 write size.

Pre-aggregation

Flink writes minute-level rollups to ClickHouse. Dashboards query rollups — 1000x smaller dataset vs raw event scans for common time-range queries.

Backpressure

Kafka buffers ingest spikes. Flink applies backpressure by slowing Kafka consumption. Prevents cascade failures during traffic bursts.

Tiered Storage

Hot (0–7 days): ClickHouse in-memory. Warm (7–90 days): ClickHouse historical. Cold (90d+): S3 + Trino query on demand. 10× cost reduction.

Late Events

Watermarks define expected lateness. Events within 30s of window close still processed. Beyond → side output (audit log). <0.1% of events arrive >5min late.

Self-Check

Test your understanding. Click to reveal answers.

What happens to events that arrive 2 minutes late to Flink? Late Data +

Flink uses watermarks to track event time progress. Window [10:00–10:01] closes when watermark passes 10:01:30 (30s grace period). Events arriving within this window are still processed. "Allowed lateness" (up to 5 min): Flink reopens the window, updates aggregates, emits a correction event downstream. Beyond 5 min: events go to a side output topic (logged for audit, not counted in metrics). In practice, <0.1% of events arrive >5 min late. Nightly Spark batch reprocesses raw Kafka data (7-day retention) to correct historical figures with late-arriving data.

Why ClickHouse instead of MySQL for analytics? Storage +

MySQL stores rows: all columns written contiguously on disk. For SELECT count(*) WHERE date=today, MySQL reads every column of every row. ClickHouse stores columns separately: reading 'count' and 'date' doesn't touch other columns. On 1B rows: MySQL might read 100GB+; ClickHouse reads <1GB. Additionally, ClickHouse uses vectorized execution (SIMD operations on column batches), MergeTree engine optimized for time-series inserts, and supports pre-aggregated materialized views. Result: analytical queries 10–1000× faster than MySQL at >10x lower cost (less data read = less I/O).

How do you handle a Flink job failure without losing data? Fault Tolerance +

Flink uses checkpointing every 30 seconds. A checkpoint is a consistent snapshot of all operator state (window buffers, aggregators) written to S3. The snapshot includes Kafka consumer offsets per partition. On failure: Flink restarts from last checkpoint, resets Kafka consumer offsets to match checkpoint, reprocesses events since then. Exactly-once semantics end-to-end. Recovery time: typically <1 minute for checkpoint load + replay. Kafka 7-day retention ensures replay window is large. The downstream sink (ClickHouse) must handle idempotent writes — Flink uses transaction markers to ensure exactly-once on the write side.

How do you scale to 10M events/second? Scale +

Kafka: add partitions (linear scale — 200 partitions = 200 parallel consumers at 50K events/sec each = 10M/sec). Flink: add TaskManagers; each handles a partition subset. Flink's state is sharded by key — adding parallelism redistributes key ranges. ClickHouse: shard across servers with ReplicatedMergeTree; Flink routes by hash(userId). Redis: Redis Cluster shards by key slot. The collection tier (Event Collector) is stateless — autoscales behind load balancer. The whole pipeline scales horizontally at every stage with no coordination needed.

Stream vs batch — when would you use each? Concepts +

Batch (Spark): processes bounded datasets — all data available at once. Latency: minutes to hours. Throughput: very high (can optimize globally). Use for: end-of-day reports, ML training datasets, ETL jobs that correct historical data. Stream (Flink): processes unbounded data as it arrives. Latency: milliseconds to seconds. State maintained across events. Use for: real-time dashboards, fraud detection, alerting, personalization. Hybrid (Lambda): run both — stream for low-latency approximate results, batch for accurate corrections. Modern alternative: Kappa — stream-only, replay Kafka for batch-style corrections. Netflix uses Kappa; Facebook historically used Lambda.
