Section 06

  
Internal Data Structures

  

    
String (most common)

    

      key       → SDS (Simple Dynamic String)

      value     → bytes (raw, int, embstr)

      ttl       → UNIX timestamp (ms) or -1

      lru_clock → uint24 — for LRU eviction sampling
    

    
Integers stored as actual int (not string) for INCR/DECR atomicity. Strings <44 bytes use embstr (single allocation).

  

  

    
Sorted Set (ZADD/ZRANGE)

    

      Small (<128 members): ziplist — compact, sequential scan

      Large: skiplist + hashtable

      — Skip list: O(log N) ZADD/ZRANGE

      — Hashtable: O(1) score lookup by member

      Both structures maintained in sync
    

    
Used for leaderboards, rate limiting (sliding window), timeline ordered by timestamp, geo indexes.

  

  

    
Hash Slot Distribution (Cluster)

    

      slot = CRC16(key) % 16384

      Each node owns a slot range (e.g. 0–5460)

      Node map stored in cluster state (gossip)

      MOVED redirect: if slot not on this node, return MOVED slot host:port
    

    
16384 slots chosen for balance: enough granularity to distribute evenly across 1000 nodes, but small enough to store slot-to-node map in 2KB (16384 bits).

Section 07

  
Key Design Decisions

  

    
Why Single-Threaded Command Processing

    

      Redis processes commands on a single thread (I/O multiplexed via epoll/kqueue). This eliminates mutex overhead and makes all operations atomically ordered — no race conditions on data structures.


      How is 1M ops/sec possible single-threaded? Most commands are sub-microsecond memory operations. The bottleneck is network I/O, not CPU. Redis 6.0+ uses multi-threaded I/O for reading/writing network buffers while keeping command execution single-threaded.
    

  

  

    
Consistent Hashing with 16384 Hash Slots

    

      Redis Cluster maps keyspace to 16384 slots. slot = CRC16(key) % 16384. Each primary node owns a contiguous range.


      Adding a node: Cluster migrates a subset of slots from existing nodes to new node — zero downtime. Keys move one by one, atomically. Requests for migrating keys get ASK redirect.


      Hash tags: {user:123}:session and {user:123}:cart share the same slot (CRC16 of the {} content). Ensures related keys co-locate on one shard for multi-key atomicity.
    

  

  

    
Replication & Automatic Failover

    

      
    
        
  • Writes go to primary → async replicated to replicas (configurable semi-sync with WAIT command)
    • 
        
  • Replicas serve read requests (scale read throughput) — may serve slightly stale data
    • 
        
  • If primary unreachable for cluster-node-timeout (default 15s): replicas hold an election via RAFT-like vote
    • 
        
  • Replica with most up-to-date replication offset wins → promoted to primary
    • 
        
  • Cluster reconfigures and clients reconnect — typically <30s downtime
    • 
      

    

  

  

    
Persistence: RDB vs AOF

    

      RDB (Redis Database Snapshot): fork() the process, child writes binary snapshot to disk. Compact, fast restart, but may lose up to minutes of writes.


      AOF (Append-Only File): log every write command. fsync=always for durability (slow), fsync=everysec for balance (lose max 1 second). AOF rewrite compacts the log periodically.


      Best practice: enable both. Use RDB for backups (compact), AOF for durability. On restart, load AOF (more complete) over RDB.

Section 09

  
Self-Check: Interview Questions

  

    

      What are the 3 cache failure modes and how do you handle each? Reliability
      +
    

    
Cache penetration: requests for keys that never exist (e.g. user:99999999). DB is hit every time since cache always misses. Fix: (1) cache null values with short TTL, (2) Bloom filter at cache layer to reject clearly-absent keys.

Cache stampede: popular key expires, thousands of requests hit DB simultaneously. Fix: (1) probabilistic early expiry — recompute cache slightly before TTL with some probability, (2) mutex-based refresh — one request acquires a lock to rebuild, others wait on a short sleep + retry.

Cache avalanche: many keys expire at the same time (e.g. all keys set with same TTL during load). DB overwhelmed. Fix: (1) add random jitter to TTLs (±10-20%), (2) use rolling expiry instead of absolute TTL, (3) circuit breaker to shed load from DB.

  

  

    

      Why does Redis use approx LRU instead of exact LRU? Eviction
      +
    

    
Exact LRU requires a doubly-linked list where every GET moves the accessed node to the head — this means every read operation involves pointer manipulation, which is expensive and cache-unfriendly. For a cache handling 1M ops/sec, this overhead is unacceptable.

Redis's approximation: sample 5 (configurable) random keys, evict the one with the oldest LRU clock. This achieves 95%+ of the effectiveness of exact LRU at <1% of the cost. The LRU clock is just a 24-bit integer stored with each value — no linked list needed. Redis 4.0+ also has LFU (Least Frequently Used) which tracks access frequency in the same 24-bit field using a Morris counter.

  

  

    

      How do hash tags enable co-location of related keys? Sharding
      +
    

    
Normally, related keys like user:123:session and user:123:cart hash to different slots and live on different shards. Multi-key operations like MGET or transactions (MULTI/EXEC) require all keys to be on the same shard — otherwise Redis returns a CROSSSLOT error.

Hash tags solve this: if a key contains {...}, only the content inside braces is used for slot computation. So {user:123}:session and {user:123}:cart both hash based on user:123 → same slot → same shard. This enables atomic multi-key operations on logically related data. Trade-off: overuse of hash tags can create hot shards if all keys use the same tag.

  

  

    

      Cache-aside vs write-through vs write-back — when to use each? Patterns
      +
    

    
Cache-aside (lazy loading): App reads cache → miss → reads DB → populates cache → returns. Most common. Pros: only hot data enters cache. Cons: cold start, first-request latency, stale data until TTL expires.

Write-through: Every write goes to cache AND DB synchronously. Pros: cache always fresh, never stale. Cons: write latency = cache + DB latency (both synchronous), wasted cache space for data never read.

Write-back (write-behind): Write to cache only, flush to DB async. Pros: very fast writes. Cons: risk of data loss if cache crashes before flush, complex to implement correctly. Used for write-heavy workloads (counters, session state) where some loss is acceptable.

  

  

    

      How does Redis implement a distributed lock? Concurrency
      +
    

    
Basic distributed lock: SET lock:resource random_value NX EX 30. NX = only set if not exists (atomic acquire), EX 30 = auto-expire after 30 seconds (prevents deadlock if holder crashes).

To release safely: use a Lua script to check-and-delete atomically — prevents releasing someone else's lock if yours expired: if redis.call("GET",KEYS[1])==ARGV[1] then return redis.call("DEL",KEYS[1]) else return 0 end.

Redlock (Martin Kleppmann algorithm): acquire lock on N/2+1 of N independent Redis instances. Majority quorum tolerates node failures. Controversial — some argue it's unsafe under GC pauses or clock skew. For most use cases (session locks, rate limiting), single-node Redis lock is sufficient.

  







  
Section 10

  
Scaling & Bottlenecks

  

    

      
Pipelining

      
Batch multiple GET/SET commands in one TCP round-trip. 10 pipelined commands = 1 RTT instead of 10. 10x throughput improvement for batch workloads. Client libraries support this natively.

    

    

      
Connection Pooling

      
Each Redis connection is lightweight but not free. Application servers maintain a pool of 10–50 connections per Redis shard. Avoids TCP handshake on every request. Use Jedis pool / Lettuce.

    

    

      
Hot Key Problem

      
A viral key (e.g. a celebrity's profile) hammers one shard. Solution: append a random suffix key:{0-9}, write to all, read from random suffix. Distributes hot key across 10 slots.

    

    

      
Cache Stampede

      
On expiry, thousands of requests hit DB simultaneously. Solution: probabilistic early expiry (recompute before TTL), or mutex-based refresh (only one request rebuilds, rest wait in Redis).

    

    

      
Cache Penetration

      
Requests for non-existent keys bypass cache and hit DB every time. Solution: cache null values with short TTL (60s), or use a Bloom filter to reject clearly-missing keys before Redis lookup.

    

    

      
Memory Optimization

      
Use appropriate data structures: hash instead of many small strings for user objects. Enable hash-max-ziplist-entries for small hashes. Use OBJECT ENCODING to verify actual encoding used.

    

  







  
Section 11

  
Interview Tips

  
🎯
Cache-aside vs write-through vs write-back: Cache-aside = app reads cache, on miss reads DB and populates cache. Write-through = every write goes to cache AND DB. Write-back = write to cache only, async flush to DB (risk of loss). Know all three and their trade-offs.

  
⚡
Single-threaded is a feature, not a bug: Redis is single-threaded for command execution — this is why all operations are atomic and why you can use INCR safely without locks. The interviewer may challenge this — explain I/O multiplexing and why CPU isn't the bottleneck.

  
🔀
Consistent hashing with 16384 slots: Explain why 16384 (2^14) — deliberate choice for balance. Every key maps deterministically to a slot, slots map to nodes. Adding a node only migrates a fraction of slots.

  
💀
Three cache failure modes: Cache penetration (non-existent keys), cache stampede (mass expiry), cache avalanche (many keys expire at once, overwhelming DB). Have a solution ready for each: null caching/Bloom filter, mutex refresh, staggered TTLs.

  
💾
Persistence trade-offs matter: If the interviewer asks "what if a Redis node crashes?", discuss RDB vs AOF. If it's a pure cache, you can skip AOF (just rebuild from DB). If Redis is primary store (sessions, rate limit counters), AOF with fsync=everysec is essential.
