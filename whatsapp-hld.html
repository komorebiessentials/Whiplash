// Message Service: routes message to recipient's WS server
void routeMessage(Message msg) {
    // Look up which WS server has recipient's connection
    String wsServer = redis.get("user:" + msg.recipientId + ":server");

    if (wsServer != null) {
        // Recipient is online — publish to their inbox channel
        redis.publish("user:" + msg.recipientId + ":inbox", msg);
        // The WS server subscribed to this channel delivers it
    } else {
        // Recipient offline — queue for later delivery
        kafka.publish("offline-queue", msg);
    }
}
// On reconnect: drain offline queue and deliver
void onUserConnect(String userId, String wsServer) {
    // Register presence: TTL 35s, refreshed by 30s heartbeat
    redis.setex("user:" + userId + ":server", 35, wsServer);

    // Fetch and clear offline queue from Cassandra
    List<Message> pending = cassandra.getUndelivered(userId);
    for (Message m : pending) {
        wsServer.deliver(userId, m);
        cassandra.markDelivered(m.msgId);
    }
}
// Group message fan-out: 1 send → N deliveries
void fanOutGroupMessage(Message msg, String groupId) {
    // Get all member IDs (cached in Redis for active groups)
    List<String> members = redis.smembers("group:" + groupId + ":members");

    // Write single message to Cassandra (shared storage)
    cassandra.write(msg);

    // Publish routing event per member (Kafka for scale)
    for (String memberId : members) {
        if (!memberId.equals(msg.senderId)) {
            kafka.publish("group-fanout", new RoutingEvent(memberId, msg));
        }
    }
}
// Group of 1000 = 999 Kafka messages. Workers route each to WS or offline queue.

Self Check

Answer before revealing. These are real interview questions.

How does a message from Client A reach Client B when they're on different WebSocket servers?Cross-Server Routing+

The key: Redis pub/sub as a message bus. When Client A sends a message, it arrives at WS Server 1. WS Server 1 forwards to Message Service, which looks up where Client B is connected via Redis: GET user:{b}:server → "ws-server-42". Then publishes to user:{b}:inbox pub/sub channel. WS Server 42 subscribes to that channel and pushes the message to Client B's WebSocket.

If B is offline (Redis returns null), the message goes to Kafka's offline queue. When B reconnects, WS Server 42 registers B's presence in Redis and drains the offline queue.

How do you handle 250M concurrent WebSocket connections?Scale+

Each WebSocket server can maintain ~10K persistent connections (limited by file descriptors, memory, and event loop). 250M / 10K = 25,000 servers. Key design choices: (1) WS servers are stateful but connection-state only — they store no durable data. (2) Presence/routing state is externalized to Redis, not kept in process memory. (3) Each server subscribes to Redis pub/sub channels for each connected user. (4) When a server crashes, users reconnect and Redis state is rebuilt within 30s (heartbeat TTL).

What does "end-to-end encrypted" mean architecturally? How does it work?Security+

E2EE means the server stores and routes ciphertext that it cannot decrypt. Each user generates a key pair on-device at registration. The public key is uploaded to the server. When Alice messages Bob: (1) Alice's app fetches Bob's public key from server. (2) Alice's app encrypts the message locally with a symmetric key, then encrypts that key with Bob's public key (Diffie-Hellman key exchange). (3) Only Bob's device, with his private key, can decrypt.

The WhatsApp server stores encrypted BLOB in Cassandra. Even with a government subpoena, WhatsApp can only hand over encrypted bytes — they cannot produce plaintext.

How are the three check marks (Sent ✓ / Delivered ✓✓ / Read ✓✓ blue) implemented?Feature Design+

Sent ✓: Client A receives an ACK from WS Server 1 confirming the message reached the server and was written to Cassandra.

Delivered ✓✓: When WS Server 2 pushes the message to Client B, it sends a receipt event back through the system: {type:"receipt", msgId:X, status:"delivered"}. This routes back to Client A via the same pub/sub mechanism.

Read ✓✓ (blue): When Client B opens the conversation, the app sends a {type:"read", chatId:X, lastReadMsgId:Y} event. This routes back to Client A. All receipt events are written to Cassandra so they survive server restarts.

How do you partition Cassandra for 10 TB/day of messages?Data Model+

Partition key: chat_id. All messages in a conversation live on the same Cassandra node — enabling fast range scans for history: WHERE chat_id = X ORDER BY msg_id DESC LIMIT 50.

Hot partition problem: A group with 1M members and high message volume could overload one node. Solution: (chat_id, bucket) composite partition key where bucket = floor(msg_id / N) — distributes across nodes while keeping time ranges scannable.

Retention: Messages older than 5 years are archived to S3 via TTL or compaction. Recent messages stay in Cassandra for fast reads. Cassandra's SSTable compaction handles deletes efficiently.
