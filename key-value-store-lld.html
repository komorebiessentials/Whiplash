Clarifying Questions

These questions define the scope and reveal hidden complexity.

  • ?What are the key and value types?String/String for simplicity; mention generic K/V if you want to show extensibility.
  • ?What's the eviction policy when full?Start with LRU. Ask if LFU or TTL-based eviction is needed — affects the EvictionPolicy interface design.
  • ?What transaction isolation level is needed?Read-your-own-writes within a transaction is simplest. Mention Snapshot Isolation if asked for more rigor.
  • ?TTL cleanup: eager or lazy?Lazy (check on get) is simpler to implement. Background thread (eager) is more accurate but adds complexity.
  • ?Persistence required?In-memory only for interview scope. Note where WAL would plug in if asked.
  • ?Single node or distributed?Single-node in-memory for LLD. Distributed would be an HLD question (consistent hashing, replication, etc.).

05

Key Flows

Three flows to know cold: put, get (with TTL check), and transaction commit.

put(key, value, ttlSeconds)

Check active transaction
If txLocal.get() != null, buffer in Transaction.pendingWrites. Return early — don't write to store yet.
Acquire write lock
lock.writeLock().lock() — all readers must finish before write proceeds.
Evict if at capacity
If storage.size() >= capacity and key is new: eviction.evict() → remove that key from storage.
Store ValueEntry
Create ValueEntry(value, expiresAt = now+ttl, version++). Put in storage map.
Update eviction + notify
eviction.onPut(key). Release lock. pubSub.notifyPut(key, value).

get(key) / Transaction commit

get: check transaction first
If tx active: check deletedKeys → return null; check pendingWrites → return buffered value.
get: read lock + expiry check
lock.readLock().lock(). Fetch ValueEntry. If entry.isExpired() → lazy delete, return null. eviction.onAccess(key).
commit: acquire write lock
lock.writeLock().lock() — all in-flight reads must finish before commit applies changes.
commit: apply writes atomically
Delete keys in deletedKeys. Put all pendingWrites into storage. All or nothing.
commit: clear transaction
Release lock. txLocal.remove() — thread is now transaction-free.
06

Code

Focus on ValueEntry, LRU eviction, InMemoryStore's get/put, and transactions.

ValueEntry & EvictionPolicy

class ValueEntry {
    private final String value;
    private final Instant expiresAt; // null = no TTL
    private final long version;

    public boolean isExpired() {
        return expiresAt != null && Instant.now().isAfter(expiresAt);
    }
}
class LRUEvictionPolicy implements EvictionPolicy {
    private final LinkedHashMap<String, Boolean> accessOrder;

    public LRUEvictionPolicy(int capacity) {
        // accessOrder=true: get() moves key to end (most recent)
        accessOrder = new LinkedHashMap<>(capacity, 0.75f, true);
    }
    public void onAccess(String key) { accessOrder.get(key); }
    public void onPut(String key)    { accessOrder.put(key, true); }
    public void onDelete(String key) { accessOrder.remove(key); }

    public String evict() {
        // first entry = least recently used
        String lru = accessOrder.keySet().iterator().next();
        accessOrder.remove(lru);
        return lru;
    }
}

InMemoryStore — get & put

public String get(String key) {
    Transaction tx = txLocal.get();
    if (tx != null) {
        if (tx.isDeleted(key))  return null;
        if (tx.hasPending(key)) return tx.getPending(key);
    }
    lock.readLock().lock();
    try {
        ValueEntry entry = storage.get(key);
        if (entry == null || entry.isExpired()) {
            if (entry != null) storage.remove(key); // lazy delete
            return null;
        }
        eviction.onAccess(key);
        return entry.getValue();
    } finally { lock.readLock().unlock(); }
}
public void put(String key, String value, long ttlSeconds) {
    Transaction tx = txLocal.get();
    if (tx != null) { tx.put(key, value); return; }

    lock.writeLock().lock();
    try {
        if (!storage.containsKey(key) && storage.size() >= capacity) {
            String evicted = eviction.evict();
            storage.remove(evicted);
        }
        Instant expiry = ttlSeconds > 0
            ? Instant.now().plusSeconds(ttlSeconds) : null;
        storage.put(key, new ValueEntry(value, expiry, nextVersion.incrementAndGet()));
        eviction.onPut(key);
    } finally { lock.writeLock().unlock(); }
    pubSub.notifyPut(key, value); // outside lock — don't hold lock during IO
}

Transactions

class Transaction {
    private final Map<String, String> pendingWrites = new LinkedHashMap<>();
    private final Set<String> deletedKeys = new HashSet<>();

    public void put(String k, String v) {
        pendingWrites.put(k, v);
        deletedKeys.remove(k); // un-delete if previously deleted
    }
    public void delete(String k) {
        deletedKeys.add(k);
        pendingWrites.remove(k);
    }
    public boolean hasPending(String k) { return pendingWrites.containsKey(k); }
    public boolean isDeleted(String k)  { return deletedKeys.contains(k); }
}
public void commit() {
    Transaction tx = txLocal.get();
    if (tx == null) throw new NoActiveTransactionException();

    lock.writeLock().lock();
    try {
        tx.getDeletedKeys().forEach(k -> {
            storage.remove(k);
            eviction.onDelete(k);
        });
        tx.getPendingWrites().forEach((k, v) -> {
            storage.put(k, new ValueEntry(v, null, nextVersion.incrementAndGet()));
            eviction.onPut(k);
        });
    } finally {
        lock.writeLock().unlock();
        txLocal.remove(); // clear regardless of success/failure
    }
}

public void rollback() {
    txLocal.remove(); // discard pending writes, done
}

Design Patterns Used

This problem is a showcase for Strategy, Decorator, and Observer.

Behavioral
Strategy
EvictionPolicy interface — LRU, LFU, FIFO, TTL-based — inject at construction. Swap policy without touching InMemoryStore.
Structural
Decorator
TTLDecorator wraps any KVStore, PersistenceDecorator wraps it again. Stack capabilities without inheritance — each decorator adds one concern.
Behavioral
Observer
PubSubManager notifies Subscribers after each put/delete — without coupling the store to any specific notification logic.
Concurrency
Thread-Local
Transaction stored in ThreadLocal — each thread manages its own transaction state with zero coordination overhead.
Creational
Singleton
Store instance shared across the application. Multiple named stores possible via a StoreRegistry.

Concurrency & Thread Safety

The entire design is driven by the read/write concurrency requirements — mention ReadWriteLock early.

Key insight: A key-value store is typically read-heavy. Using synchronized blocks all readers while any write happens. ReadWriteLock allows unlimited concurrent reads — only writes are exclusive.
RiskLocationSeverityFix
Concurrent get + put corrupts HashMapstorage mapHIGHReadWriteLock — exclusive write lock prevents structural corruption
LRU accessOrder map modified during concurrent readsLRUEvictionPolicy.onAccessHIGHLinkedHashMap not thread-safe; protect with write lock or use ConcurrentLinkedHashMap
Transaction commit visibilitycommit()HIGHWrite lock during commit ensures all changes visible atomically after unlock
Lazy TTL delete races with concurrent getget() expiry checkMEDCheck expiry inside read lock; delete inside write lock (upgrade or separate call)
Background TTL thread deletes key mid-transactionTTLManagerLOWTTLManager acquires write lock before deletion — consistent with all other writers

→ For a production implementation, consider ConcurrentHashMap with per-bucket locking for even higher throughput. Caffeine library implements this + LRU efficiently out of the box.

09

Interview Tips

Senior-level signals that differentiate your answer.

1
Lead with ReadWriteLock, not synchronized. A cache is read-heavy. synchronized blocks every reader when any write happens. ReadWriteLock is the correct default — say it in your first paragraph on concurrency.
2
LRU in one line: LinkedHashMap(cap, 0.75f, true). The third parameter (accessOrder=true) makes get() reorder entries. iterator().next() always returns the LRU. O(1) evict — know this cold.
3
Start with lazy TTL, then propose eager. Lazy expiry (check on get) is simpler. Background thread (DelayQueue + Executors.newSingleThreadScheduledExecutor) is more accurate. Mention both, implement lazy first.
4
Transaction isolation: name the level. This design implements Read-Your-Own-Writes within a transaction. Mention Snapshot Isolation as the next level up (take a snapshot of storage at begin() time).
5
Decorator is the key extensibility insight. TTLDecorator wraps KVStore. PersistenceDecorator wraps TTLDecorator. Each adds one concern. This is far cleaner than subclassing InMemoryStore for every combination.
