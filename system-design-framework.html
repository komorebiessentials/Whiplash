How to Ace Any
System Design Interview

The 6-Step Framework

A 45-minute system design interview, structured
1
Minutes 0–5
Clarify Requirements

The most important step. Never jump into design. Ask questions, then repeat back to confirm. Distinguish functional requirements (what the system does) from non-functional (how well it does it).

Do not assume scope — "design Twitter" could mean the timeline, the DM system, notifications, or all of it. Agree on what you're building before drawing anything.

Functional requirements Non-functional requirements Scope Constraints
2
Minutes 5–10
Scale Estimation

Back-of-envelope: convert DAU and usage patterns into QPS, storage, and bandwidth. This tells you whether you're designing for a single PostgreSQL instance or a distributed system with hundreds of nodes.

Identify: Is this read-heavy or write-heavy? What's the hot path? What needs strong consistency vs can tolerate eventual? These answers drive every architecture decision.

QPS calculation Storage estimation Read/write ratio Hot path
3
Minutes 10–20
High-Level Design

Start with the simplest design that satisfies the requirements. Draw boxes: Client → Load Balancer → API Servers → Database. Walk through the primary use case end-to-end, narrating the data flow.

Don't over-engineer at this stage. Add Kafka, Redis, and microservices only if you can justify them. The interviewer needs to see you understand the basics before the optimisations.

API design Data model Core flow DB choice
4
Minutes 20–35
Deep Dive

This is where senior engineers are separated from mid-level. Proactively identify the 2–3 hardest problems specific to this system. Say: "The most interesting challenge here is X, because…" Don't wait for the interviewer to lead.

For a URL shortener: the hashing algorithm and collision handling. For Twitter: the fan-out problem with celebrities. For payments: idempotency and double-charge prevention. Know the hard problem for each system.

System-specific challenges Algorithms Edge cases Failure scenarios
5
Minutes 35–42
Scaling

Systematically ask: "what happens to each component at 10x current load?" Walk through: add caching (Redis) for read-heavy paths, CDN for static assets, read replicas for DB, sharding for write-heavy DBs, async queues (Kafka) to decouple services.

Identify bottlenecks proactively rather than waiting to be challenged. Show you think about production realities, not just happy-path design.

Caching Sharding Replication Async processing
6
Minutes 42–45
Wrap Up

Summarise the design in 2–3 sentences. Mention monitoring: "I'd alert on p99 latency and error rate." Discuss what you'd do differently with more time. Acknowledge trade-offs you made.

Senior signal: mention operational concerns — how do you deploy this, how do you roll back a bad release, what does an incident look like, how long does recovery take?

Trade-off Vocabulary

Phrases that signal senior-level thinking to interviewers
"I'll start with the simplest design that satisfies requirements, then evolve it to meet scale..."
Shows you don't over-engineer. You build what's needed, then add complexity with justification.
"The trade-off here is between consistency and availability. For this use case, I'd prioritise availability because..."
Shows CAP theorem awareness. Shows you make principled decisions, not arbitrary ones.
"This is the hot path — everything else can be eventually consistent, but this must be strongly consistent."
Shows you identify what actually matters in a system versus what can be relaxed.
"We could use a distributed lock here, but I'm concerned about the performance overhead at this scale. An alternative is..."
Shows you think about production consequences of design choices, not just correctness.
"Let me think through failure scenarios — what happens if the cache is cold after a restart?"
Shows reliability engineering thinking. Senior engineers always think about failure modes.
"I'd monitor this with [error rate / p99 latency] and alert if it exceeds [X]. Here's the runbook..."
Shows operational maturity. Juniors think about building features; seniors think about running them.
"At 10× scale, the bottleneck will move to [X]. We'd address that by [sharding / async / caching]."
Shows you think about how the system evolves, not just its current state.
"This is a read-heavy system (100:1 read/write ratio), so I'd optimise the read path first — cache, CDN, read replicas..."
Shows data-driven design. Scale estimation directly informs architecture choices.

Common Mistakes

What kills otherwise good candidates
✗ Jumping into design without clarifying requirements
✓ Spend the first 5 minutes asking questions. Write requirements on the board before drawing anything.
✗ Designing for infinite scale from day 1
✓ Design for the given scale. Mention how to evolve, but don't add Kafka if the problem is 1K QPS.
✗ Not knowing your numbers (QPS, storage)
✓ Always do back-of-envelope. Numbers anchor every architectural decision.
✗ Treating all data as equally consistent
✓ Identify what needs strong consistency (payments, inventory) vs what can be eventual (timelines, notifications).
✗ Silent thinking — drawing without narrating
✓ Always narrate your thought process. Interviewers evaluate your reasoning, not just your final answer.
✗ Forgetting failure cases
✓ For each component: "what happens when this fails?" Circuit breakers, retries, fallbacks, graceful degradation.
✗ One-size-fits-all database (always PostgreSQL, or always NoSQL)
✓ Justify your DB choice: access patterns, consistency requirements, scale, query complexity. Multiple DBs in one system is fine.
✗ Not pushing back on vague requirements
✓ "Design a messaging system" is ambiguous. Push: 1:1 or group? Real-time or async? End-to-end encrypted? These change the architecture completely.

Questions to Ask

For any system design problem, these unlock the requirements

Functional

Who are the users? (consumers, businesses, internal engineers)
What are the core features vs nice-to-haves?
Do we need real-time updates or is polling acceptable?
Do we need search? If so, full-text or structured?
Do we need offline support or mobile sync?
What are the primary read and write access patterns?
Do we need multi-tenancy or is this single-tenant?

Non-Functional

What is the expected scale? (DAU, concurrent users, QPS)
What is the acceptable latency? (p50, p99 targets)
What consistency guarantees do we need?
What is the data retention requirement?
What is the availability target? (99.9% vs 99.999%)
Any compliance or regulatory requirements? (GDPR, PCI-DSS, HIPAA)
What is the read-to-write ratio? (helps select DB and cache strategy)
