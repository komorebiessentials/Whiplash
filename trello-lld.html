02

Clarifying Questions

Ask these before drawing any classes.

  • ?Can cards exist without a list (backlog)?Simplest: no — a card always belongs to exactly one list. Archiving the card detaches it.
  • ?Can the same label be reused across cards?Yes — labels are Board-scoped, not Card-scoped. One label object is shared across all cards on a board.
  • ?Is undo/redo required?Not usually. If yes, every action becomes a Command object — big design change.
  • ?Can cards be moved across boards?Trello supports it but it's complex (labels don't transfer cleanly). Clarify scope.
  • ?How is "position" managed for drag-and-drop ordering?Integer index (requires reorder on move) vs floating-point spacing (insert 1.5 between 1.0 and 2.0 = no reorder). Mention both.
  • ?Notification delivery: real-time or async?In-process Observer for interview; in prod, use a message queue (Kafka) to decouple notification delivery.

05

Key Flows

Walk these step-by-step — it shows you understand execution, not just structure.

Create Card Flow

Validate membership
Check caller is a BoardMember on the list's board. Throw if not.
Build Card via Builder
new Card.Builder(title, list).description(desc).build() — position = list.getCards().size()
Add to list
list.addCard(card) appends to the end.
Emit Activity
board.emit(new Activity(CARD_CREATED, actor, card, "...")) — all observers notified.
Return card
Return created Card with UUID assigned.

Move Card Flow (drag & drop)

Validate & fetch
Validate membership. Get card, sourceList = card.getList(), targetList from targetListId.
Guard: same board
Throw InvalidOperationException if source and target lists are on different boards.
Remove from source, reorder
sourceList.removeCard(card) then reindex remaining cards: position = index.
Insert at target position
targetList.getCards().add(targetPosition, card), reindex target list.
Emit CARD_MOVED Activity
board.emit(Activity(CARD_MOVED, "from X to Y")). Activity log updated, notification sent.

Code

Focus on the design-heavy parts — Builder, Observer, and the moveCard logic.

Card — Builder Pattern

class Card {
    private String id, title, description;
    private BoardList list;
    private List<User> assignees = new ArrayList<>();
    private List<Label> labels = new ArrayList<>();
    private LocalDateTime dueDate;
    private List<Checklist> checklists = new ArrayList<>();
    private List<Comment> comments = new ArrayList<>();
    private int position;
    private long version = 0; // optimistic locking

    private Card() {}
}
public static class Builder {
    private final String title;
    private final BoardList list;
    private String description;
    private LocalDateTime dueDate;

    public Builder(String title, BoardList list) {
        this.title = title; this.list = list;
    }
    public Builder description(String d) { this.description = d; return this; }
    public Builder dueDate(LocalDateTime dt) { this.dueDate = dt; return this; }

    public Card build() {
        Card c = new Card();
        c.id = UUID.randomUUID().toString();
        c.title = title; c.list = list;
        c.description = description; c.dueDate = dueDate;
        c.position = list.getCards().size();
        return c;
    }
}

Board — Observer Pattern

interface ActivityObserver {
    void onActivity(Activity activity);
}
class Board {
    private final List<ActivityObserver> observers = new ArrayList<>();
    private final List<Activity> activityLog =
        new CopyOnWriteArrayList<>(); // thread-safe append

    public void emit(Activity activity) {
        activityLog.add(activity);
        observers.forEach(o -> o.onActivity(activity));
    }

    public void addObserver(ActivityObserver obs) {
        observers.add(obs);
    }
}
class NotificationObserver implements ActivityObserver {
    private final NotificationService svc;

    public void onActivity(Activity activity) {
        if (activity.getType() == ActivityType.CARD_ASSIGNED) {
            // notify each newly assigned user
            activity.getTargetCard().getAssignees().forEach(u ->
                svc.notify(u, "You were assigned to: " + activity.getDescription())
            );
        }
    }
}

TrelloService — moveCard & helpers

public void moveCard(String cardId, String targetListId, int targetPosition) {
    Card card = findCard(cardId);
    BoardList src = card.getList(), tgt = findList(targetListId);

    if (!src.getBoard().equals(tgt.getBoard()))
        throw new InvalidOperationException("Cross-board moves not supported");

    synchronized (src) { // lock source list during removal
        src.getCards().remove(card);
        reorderPositions(src.getCards());
    }
    synchronized (tgt) { // lock target list during insertion
        card.setList(tgt);
        tgt.getCards().add(targetPosition, card);
        reorderPositions(tgt.getCards());
    }

    src.getBoard().emit(new Activity(ActivityType.CARD_MOVED, currentUser, card,
        "moved from " + src.getName() + " to " + tgt.getName()));
}

private void reorderPositions(List<Card> cards) {
    for (int i = 0; i < cards.size(); i++) cards.get(i).setPosition(i);
}
public double getChecklistCompletion(Card card) {
    int total = 0, done = 0;
    for (Checklist cl : card.getChecklists())
        for (ChecklistItem item : cl.getItems()) {
            total++;
            if (item.isCompleted()) done++;
        }
    return total == 0 ? 0 : (done * 100.0 / total);
}
07

Design Patterns Used

Name these explicitly — it's what separates a good answer from a great one.

Behavioral
Observer
Board emits Activity events. NotificationObserver and ActivityLogObserver react independently — adding new reactions requires zero changes to Board.
Structural
Composite
Board → BoardList → Card → Checklist → ChecklistItem forms a tree. "Archive board" cascades naturally through the hierarchy.
Creational
Builder
Card has many optional fields (dueDate, description, labels). Builder avoids a telescoping constructor and makes creation intent-revealing.
Behavioral
Command
Wrap moveCard, assignCard, etc. as Command objects to enable undo/redo and replay the activity log. Each command knows how to execute and reverse itself.
Behavioral
Strategy
NotificationStrategy: EmailNotification, PushNotification, InAppNotification — swap delivery channel without changing Board or observer logic.

Concurrency & Thread Safety

Most candidates skip this entirely — bring it up and you immediately stand out.

The core race condition: two users drag the same card simultaneously. Without synchronization, both threads read stale positions and produce corrupted ordering.
RiskLocationSeverityFix
Double-move same cardmoveCard()HIGHLock on source list + target list (acquire in consistent order to avoid deadlock)
Position corruption on reorderreorderPositions()HIGHLock at BoardList level during both remove and reorder
Concurrent activity log writesBoard.activityLogMEDUse CopyOnWriteArrayList — safe for concurrent appends, reads don't block
Stale card version on concurrent editCard fieldsMEDOptimistic locking: Card.version — compare-and-swap before applying edit
Observer notification during lockBoard.emit()LOWEmit outside the synchronized block — don't hold list lock while notifying observers

→ Production Trello uses event sourcing: all mutations are commands appended to a log; the board state is derived by replaying. This eliminates most concurrency issues at the model level.

09

Interview Tips

What separates a good answer from a great one.

1
Labels are Board-scoped, not Card-scoped. A common mistake is putting Label inside Card. In Trello, labels are defined on the Board and referenced by cards — so renaming a label updates it everywhere instantly.
2
Mention floating-point positions. Integer reindexing on every move is O(n) writes to the DB. Production systems use decimals: insert between 1.0 and 2.0 → position 1.5. No other cards need updating.
3
Activity log is append-only. Never update or delete activity records. It's an audit trail. Say this explicitly — it implies an immutable event store, which is a strong architectural signal.
4
The Observer decouples notification from business logic. Board.emit() doesn't know anything about email or push notifications — it just fires events. This is the key extensibility win.
5
Card.position ≠ BoardList.position. Both lists and cards have positions. Managing both hierarchies is a common gotcha — mention it to show you've thought through the data model.
