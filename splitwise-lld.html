02

Clarifying Questions

Ask these before designing — each answer changes which classes you need.

  • ?Can a non-group expense exist?Yes — two friends splitting a bill without a group. Makes Group optional on Expense.
  • ?Can the payer also owe money?Yes — if Alice pays ₹100 for 3 people (equal), Alice owes ₹0 but the others owe ₹33.33 each. The payer is excluded from owing themselves.
  • ?Is simplify-debts required or optional?It's an optimization. Ask if it's in scope — it adds significant complexity (greedy graph algorithm).
  • ?Should we support expense deletion or editing?Adds reverse-balance logic. Simplest: support delete by reversing balance deltas.
  • ?Currency — single or multi?For an interview, assume single currency. Multi-currency adds an FX conversion layer.
  • ?Integer or float amounts?Store in paise/cents (int) to avoid floating-point rounding — e.g., ₹100 = 10000 paise.

Key Flows

Walk through both flows step by step in the interview — shows you understand data flow, not just class structure.

Add Expense Flow

Caller provides expense details
paidBy, amount, participants list, splitType (EQUAL/EXACT/etc), and any extras (percentages, shares).
Strategy selected via factory
SplitStrategyFactory.get(splitType) returns the right SplitStrategy implementation.
Calculate splits
strategy.calculateSplits(amount, participants, extras) → List<ExpenseSplit>. Validate sum == total.
Update balance map
For each split: if split.user != paidBy → balances[split.user][paidBy] += split.amount.
Store & notify
Expense persisted. Observers (push notifications) triggered for all participants.

Settle Up Flow

User A settles with User B
Lookup net balance between A and B: balances[A][B] - balances[B][A].
Create settlement record
Settlement(from=A, to=B, amount=netOwed) stored for audit history.
Process payment (optional)
PaymentProcessor.process(amount) — UPI, card, etc. Can be skipped if offline/manual.
Zero out balances
balances[A][B] = 0 and balances[B][A] = 0. Net between A and B is now cleared.

Code

Focus on the logic that interviewers actually ask about — split calculation, balance update, and debt simplification.

ExpenseSplit hierarchy
abstract class ExpenseSplit {
    protected final User user;
    protected long amount; // in paise

    public abstract long getAmount();
    public User getUser() { return user; }
}
class EqualSplit extends ExpenseSplit {
    public EqualSplit(User user, long amount) {
        this.user = user;
        this.amount = amount; // set by EqualSplitStrategy
    }
    public long getAmount() { return amount; }
}

class EqualSplitStrategy implements SplitStrategy {
    public List<ExpenseSplit> calculateSplits(long total, List<User> participants, ...) {
        long each = total / participants.size();
        long remainder = total % participants.size();
        List<ExpenseSplit> splits = new ArrayList<>();
        for (int i = 0; i < participants.size(); i++) {
            // distribute remainder paise to first users
            splits.add(new EqualSplit(participants.get(i), each + (i < remainder ? 1 : 0)));
        }
        return splits;
    }
}
class PercentageSplit extends ExpenseSplit {
    private final double percentage;

    public PercentageSplit(User user, double percentage) {
        this.user = user;
        this.percentage = percentage;
    }
    public long getAmount() { return (long)(amount * percentage / 100); }
}

// validation in strategy: percentages must sum to 100
if (percentages.stream().mapToDouble(Double::doubleValue).sum() != 100.0)
    throw new InvalidSplitException("Percentages must sum to 100");
addExpense — balance update
public void addExpense(User paidBy, long amount,
        List<User> participants, SplitType type, Map<String,Object> extras) {

    SplitStrategy strategy = strategyFactory.get(type);
    List<ExpenseSplit> splits = strategy.calculateSplits(amount, participants, extras);

    // validate splits sum to total
    long total = splits.stream().mapToLong(ExpenseSplit::getAmount).sum();
    if (total != amount) throw new InvalidSplitException();

    Expense expense = new Expense(UUID.randomUUID().toString(), paidBy, amount, splits);

    // update balances: each participant owes paidBy their split amount
    for (ExpenseSplit split : splits) {
        if (!split.getUser().equals(paidBy)) {
            balances
                .computeIfAbsent(split.getUser(), k -> new ConcurrentHashMap<>())
                .merge(paidBy, split.getAmount(), Long::sum);
        }
    }
    expenses.add(expense);
    observers.forEach(o -> o.onExpenseAdded(expense));
}
public Settlement settleUp(User from, User to) {
    long owes = balances
        .getOrDefault(from, Collections.emptyMap())
        .getOrDefault(to, 0L);

    if (owes <= 0) throw new NoBalanceException("No debt from " + from + " to " + to);

    Settlement s = new Settlement(from, to, owes, LocalDateTime.now());

    // clear balance in both directions
    balances.get(from).put(to, 0L);
    balances.computeIfAbsent(to, k -> new HashMap<>()).put(from, 0L);

    settlements.add(s);
    return s;
}
Simplify Debts — greedy O(n log n)
public List<Settlement> simplifyDebts() {
    // Step 1: compute net position for each user (+ve = owed, -ve = owes)
    Map<User, Long> net = new HashMap<>();
    for (var outer : balances.entrySet()) {
        for (var inner : outer.getValue().entrySet()) {
            net.merge(outer.getKey(), -inner.getValue(), Long::sum);
            net.merge(inner.getKey(),  inner.getValue(), Long::sum);
        }
    }
    // Step 2: greedy — match max creditor with max debtor
    PriorityQueue<Map.Entry<User,Long>> creditors =
        new PriorityQueue<>((a,b) -> Long.compare(b.getValue(), a.getValue()));
    PriorityQueue<Map.Entry<User,Long>> debtors =
        new PriorityQueue<>((a,b) -> Long.compare(a.getValue(), b.getValue()));

    net.entrySet().forEach(e -> {
        if (e.getValue() > 0) creditors.add(e);
        else if (e.getValue() < 0) debtors.add(e);
    });

    List<Settlement> result = new ArrayList<>();
    while (!creditors.isEmpty() && !debtors.isEmpty()) {
        var creditor = creditors.poll();
        var debtor   = debtors.poll();
        long amt = Math.min(creditor.getValue(), -debtor.getValue());
        result.add(new Settlement(debtor.getKey(), creditor.getKey(), amt));
        // re-add remainder if any
        if (creditor.getValue() - amt > 0) creditors.add(Map.entry(creditor.getKey(), creditor.getValue() - amt));
        if (-debtor.getValue() - amt > 0) debtors.add(Map.entry(debtor.getKey(), debtor.getValue() + amt));
    }
    return result;
}

Design Patterns

Name these explicitly in the interview — interviewers want to hear the vocabulary.

Behavioral
Strategy
SplitStrategy interface — EqualSplitStrategy, ExactSplitStrategy, etc. Swap split logic without touching ExpenseManager.
Behavioral
Observer
ExpenseManager notifies listeners on expense added or settled — push notifications, audit log, UI refresh.
Structural
Facade
ExpenseManager is the single entry point. Hides strategy selection, balance map updates, and settlement complexity.
Creational
Factory
SplitStrategyFactory maps SplitType enum → SplitStrategy instance. Adding a new type = add one entry in the factory.

Concurrency

Bring this up unprompted — most candidates skip it entirely.

The core race condition: Two threads simultaneously adding expenses for the same user pair can corrupt the balance map — one read-modify-write overwrites the other.
RiskLocationSeverityFix
Balance map corruptionaddExpense() — balances.merge()HIGHConcurrentHashMap + atomic merge()
Stale balance read during settlesettleUp()HIGHSynchronized block on user-pair key (ordered to avoid deadlock)
expenses list concurrent modificationexpenses.add()MEDUse CopyOnWriteArrayList or synchronized list
Observer notification during mutationnotifyObservers()LOWNotify after balance update is complete, not during

→ For settleUp deadlock prevention: always acquire locks in user-id order (lower id first) so two threads settling A↔B never deadlock.

09

Interview Tips

What separates a good answer from a great one in the interview.

1
Store amounts as integers (paise/cents), not doubles. ₹100 = 10000 paise. Float arithmetic on ₹33.33 × 3 ≠ ₹100. This is a real production bug — interviewers love it when you catch it.
2
Normalize the balance key pair. Instead of balances[A][B] and balances[B][A] separately, always use the user with lower ID as the outer key. Halves memory and avoids inconsistency.
3
paidBy can also appear in splits. If Alice pays ₹100 for herself and two others (equal split), her share is ₹33. She does NOT owe herself — skip her in the balance update loop.
4
Simplify debts is optional — ask first. The greedy PriorityQueue approach minimizes transactions but is O(n log n). The interviewer may not want it — don't over-engineer unless asked.
5
Group is optional on Expense. Two friends splitting a dinner bill don't need a group. Keep Group nullable to avoid coupling all expenses to groups.
6
Validate before mutating. In addExpense, validate that splits sum to total BEFORE updating balances. Partial balance updates from invalid splits are very hard to roll back.
