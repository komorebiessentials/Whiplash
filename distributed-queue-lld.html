Functional

Functional

02

Clarifying Questions

These questions reveal whether the interviewer wants a simple queue or a Kafka-level system.

  • ?Single node or distributed across brokers?Single-node for LLD interview. Distributed adds leader election, replication, ISR — that's HLD territory.
  • ?At-least-once or exactly-once delivery?At-least-once is simpler (commit offset after processing). Exactly-once requires idempotent producers and two-phase commit.
  • ?Message retention: by time or by size?Both in Kafka. For interview: configurable retentionMs per topic. Messages older than retention are eligible for compaction.
  • ?Push or pull model for consumers?Pull (poll-based) — consumers read at their own pace. Push would require the broker to track consumer state and push capacity, which is harder.
  • ?How many partitions per topic?Configurable at topic creation. More partitions = more parallelism. In-interview: treat as fixed after creation.
  • ?What happens when a consumer leaves the group?GroupCoordinator triggers a rebalance — partitions reassigned among remaining consumers.

05

Key Flows

Three flows: produce, consume (poll + commit), and consumer group rebalance.

Produce Flow

Producer.send(topic, key, value)
Create Message(-1, key, value, now). Strategy has no offset yet — partition assigns it.
Select partition
strategy.selectPartition(topic, msg) → partitionId. RoundRobin for load balancing; KeyHash for ordering by key.
Append to partition log
offset = nextOffset.getAndIncrement(). writeLock. log.add(msg). unlock.
Return RecordMetadata
Return RecordMetadata(topicName, partitionId, offset) — producer confirms write.

Consume Flow (poll + commit)

Consumer.poll(topicName, partId, max)
tp = new TopicPartition(topic, partId). currentOffset = committedOffsets.getOrDefault(tp, 0).
Read from partition
partition.read(currentOffset, max) — returns messages from that offset onward. readLock held during scan.
Consumer processes messages
Application logic runs on messages. If crash here before commit → messages re-delivered on restart (at-least-once).
commitOffset(tp, newOffset)
committedOffsets.put(tp, newOffset). offsetManager.commit(groupId, tp, newOffset). Next poll starts from here.

Rebalance Flow (consumer joins/leaves group)

Consumer sends JoinGroup to GroupCoordinator
coordinator.onConsumerJoin(groupId, consumerId) triggered. Group membership changes.
Coordinator triggers rebalance
All consumers in group pause poll. Coordinator collects current membership list.
Partition assignment (round-robin)
allPartitions distributed across consumers: partition[i] → consumers[i % consumers.size()].
Consumers get new assignments
Each consumer receives its new partition list. Resume polling from last committed offset for each assigned partition.

Code

Focus on Partition (the core), PartitionStrategy, Producer, Consumer, and GroupCoordinator.

Message & Partition

class Message {
    private final long offset;
    private final String key;
    private final byte[] value;
    private final Instant timestamp;
    private final Map<String, String> headers;
    // constructor only — fully immutable, no setters

    public Message(long offset, String key, byte[] value,
                   Instant timestamp, Map<String,String> headers) {
        this.offset = offset;
        this.key = key;
        this.value = value.clone(); // defensive copy
        this.timestamp = timestamp;
        this.headers = Map.copyOf(headers);
    }
}
class Partition {
    private final List<Message> log = new ArrayList<>();
    private final AtomicLong nextOffset = new AtomicLong(0);
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

    public long append(String key, byte[] value, Map<String,String> headers) {
        long offset = nextOffset.getAndIncrement(); // atomic — no lock needed
        Message msg = new Message(offset, key, value, Instant.now(), headers);
        lock.writeLock().lock();
        try { log.add(msg); } finally { lock.writeLock().unlock(); }
        return offset;
    }

    public List<Message> read(long fromOffset, int maxCount) {
        lock.readLock().lock();
        try {
            return log.stream()
                .filter(m -> m.getOffset() >= fromOffset)
                .limit(maxCount)
                .collect(Collectors.toList());
        } finally { lock.readLock().unlock(); }
    }
}

PartitionStrategy & Producer

class RoundRobinStrategy implements PartitionStrategy {
    private final AtomicInteger counter = new AtomicInteger(0);
    public int selectPartition(Topic t, Message m) {
        return counter.getAndIncrement() % t.getPartitionCount();
    }
}

class KeyHashStrategy implements PartitionStrategy {
    public int selectPartition(Topic t, Message m) {
        if (m.getKey() == null)
            return new Random().nextInt(t.getPartitionCount());
        // same key → always same partition → in-order delivery per key
        return Math.abs(m.getKey().hashCode()) % t.getPartitionCount();
    }
}
class Producer {
    private final Broker broker;
    private PartitionStrategy strategy;

    public RecordMetadata send(String topicName, String key, byte[] value) {
        Topic topic = broker.getTopic(topicName);
        if (topic == null) throw new TopicNotFoundException(topicName);

        // temp message for partition selection (offset = -1 placeholder)
        Message probe = new Message(-1, key, value, Instant.now(), Map.of());
        int partId = strategy.selectPartition(topic, probe);

        Partition partition = topic.getPartition(partId);
        long offset = partition.append(key, value, Map.of());
        return new RecordMetadata(topicName, partId, offset);
    }
}
class Consumer {
    private final Map<TopicPartition, Long> committedOffsets =
        new ConcurrentHashMap<>();

    public List<Message> poll(String topicName, int partId, int maxRecords) {
        TopicPartition tp = new TopicPartition(topicName, partId);
        long fromOffset = committedOffsets.getOrDefault(tp, 0L);
        Partition partition = broker.getPartition(topicName, partId);
        return partition.read(fromOffset, maxRecords);
    }

    public void commitOffset(TopicPartition tp, long offset) {
        committedOffsets.put(tp, offset);
        offsetManager.commit(groupId, tp, offset); // durable
    }
}

GroupCoordinator — rebalance

class GroupCoordinator {
    private final Map<String, ConsumerGroup> groups = new ConcurrentHashMap<>();

    public void onConsumerJoin(String groupId, String consumerId) {
        ConsumerGroup group = groups.computeIfAbsent(groupId, ConsumerGroup::new);
        group.addConsumer(consumerId);
        triggerRebalance(group);
    }

    private void triggerRebalance(ConsumerGroup group) {
        List<Partition> allPartitions = getAllPartitionsForGroup(group);
        List<String> consumers = group.getActiveConsumers();

        Map<String, List<Partition>> assignments = new HashMap<>();
        for (int i = 0; i < allPartitions.size(); i++) {
            String consumer = consumers.get(i % consumers.size());
            assignments.computeIfAbsent(consumer, k -> new ArrayList<>())
                       .add(allPartitions.get(i));
        }
        group.applyAssignments(assignments);
        notifyConsumers(group, assignments); // tell each consumer its partitions
    }
}

Design Patterns Used

Name these in the interview — they show you understand the design decisions, not just the code.

Behavioral
Strategy
PartitionStrategy: RoundRobin, KeyHash, Sticky — Producer selects one; swap without changing producer or broker code.
Behavioral
Observer
GroupCoordinator observes consumer membership changes (join/leave) and triggers rebalance — producers are unaware.
Behavioral
Iterator
Partition log is read from an offset — consumers iterate the append-only log at their own pace using an explicit offset cursor.
Creational
Singleton / Registry
Broker acts as a registry of Topics. Could be a Singleton in single-node mode, or a registry of brokers in distributed mode.
Domain
Command / Event
Message is an immutable, time-stamped event record. The log is the source of truth — this is the Event Sourcing pattern.

Concurrency & Thread Safety

Partition append and read are the critical paths — concurrent producers and independent consumer groups are the key cases.

Key insight: Different ConsumerGroups can read the same partition simultaneously without coordination — they each track their own offset. ReadWriteLock lets multiple groups read concurrently while a producer holds the write lock only during append.
RiskLocationSeverityFix
Two producers get same offsetPartition.appendHIGHAtomicLong.getAndIncrement() — CAS-based, no lock needed for offset assignment
Read sees partial write to log listPartition.readHIGHReadWriteLock — write to log under write lock; readers wait until append completes
ConcurrentModification in groups mapGroupCoordinatorMEDConcurrentHashMap for groups; rebalance logic synchronized on the ConsumerGroup
Consumer polls partition mid-rebalanceConsumer.pollMEDConsumer checks assignment validity before polling; revoke partitions gracefully
Offset commit racing with rebalanceOffsetManagerLOWOffsetManager uses ConcurrentHashMap; commit is atomic per key

→ Note: AtomicLong for offset + ReadWriteLock for log access separates concerns: offset sequencing (CAS, no contention) vs log visibility (lock, minimal held time). This is why Kafka can handle millions of messages/second per partition.

Interview Tips

The conceptual clarity questions that interviewers love to ask.

1
The core insight: Kafka is a distributed commit log. Each partition is an append-only sequence of records. Consumers read at their own pace — the broker doesn't push, consumers pull. Say this in your first 30 seconds.
2
Topic (logical) vs Partition (physical). Clarify this early. A Topic is a name. A Partition is the actual storage unit. Most candidates conflate them — distinguishing yourself here is easy points.
3
Key → partition = ordering guarantee. KeyHashStrategy ensures all messages with the same key go to the same partition → ordered delivery per key. This is WHY messages have keys in Kafka.
4
At-least-once is the default — explain why. Commit offset AFTER processing = at-least-once (crash before commit → reprocess). Exactly-once requires idempotent producers AND transactions. Mention the trade-off explicitly.
5
ConsumerGroup parallelism cap = partition count. 3 partitions + 3 consumers = 1:1 assignment, full parallelism. 4th consumer sits idle. Adding more consumers than partitions gives zero benefit — say this.
