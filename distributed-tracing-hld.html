7. Database Design

Cassandra — Span Storage

Partition key = trace_id → all spans for a trace on the same partition → O(1) trace reconstruction. Cluster key = span_id. TTL on rows for auto-expiry (72 hours). Replication factor 3.

Elasticsearch — Search Index

Index per day: spans-2026-06-22. Fields indexed: service_name, operation_name, duration_ms, error, timestamp, tags (selective). Powers the search UI. Eventually consistent — async write from Kafka consumer.

Cassandra — Dependency Graph

Derived by aggregating parent/child service pairs from spans. Stored as time-bucketed adjacency list: (ts_bucket, parent_svc, child_svc) → call_count. Pre-aggregated hourly by Flink.

Redis — Tail Sampling Buffer

For tail-based sampling: buffer all spans in Redis with trace_id as key, TTL 30s. On trace completion (or timeout), apply sampling decision, then write selected traces to Cassandra + ES.

8. Key Design Decisions

Trace Context Propagation

The W3C traceparent header carries the trace across service boundaries:

traceparent: 00-{traceId-32hex}-{parentSpanId-16hex}-{flags-2hex}
Example: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01

// SDK automatically injects on outbound HTTP calls:
Span span = tracer.spanBuilder("GET /inventory").startSpan();
try (var scope = span.makeCurrent()) {
    W3CTraceContextPropagator.inject(context, httpRequest, setter);
    httpClient.send(httpRequest);
} finally {
    span.end();  // emits span to collector via OTLP
}

Head-Based vs Tail-Based Sampling

Head-BasedTail-Based
Decision pointAt root span (first service)After trace is complete (all spans received)
AccuracyCan't know if trace is interestingCan target errors / slow traces specifically
OverheadMinimal (random sample only)Must buffer all spans in memory (~30s window)
Partial tracesPossible if decision not propagatedNone — decision after all spans received
Use caseHigh-traffic, uniform sampling OKMust capture all errors & slow traces
ExamplesJaeger agent, ZipkinHoneycomb, AWS X-Ray (with reservoir)

Adaptive Sampling

Jaeger's adaptive sampling automatically adjusts the sampling rate per service+operation to maintain a target throughput (e.g., 100 traces/sec per operation). High-traffic operations like GET /health are sampled at 0.01%; critical low-traffic operations like POST /checkout at 100%. The Sampling Manager publishes updated rates periodically; SDKs poll and apply them.

Consistent-Hash Routing for Tail Sampling

Tail-based sampling needs all spans from a trace to arrive at the same collector instance. Use consistent hashing on trace_id to route spans to the same collector. If a collector crashes, the ring reroutes — some traces may have partial spans (acceptable).

9. Scaling & Bottlenecks

Collector Tier

Stateless for head-based sampling — scale horizontally. Stateful for tail-based — use consistent hashing. Each instance handles ~10K spans/sec before CPU becomes bottleneck.

Kafka Buffer

Decouples collector from storage. Handles ingestion spikes. Partitioned by trace_id for ordered delivery to processors. 7-day retention for replay.

Cassandra Write Path

Excels at high write throughput. All spans for a trace on one partition = fast reads. Batch writes from processor. TTL on data avoids manual cleanup.

Elasticsearch Indexing

Async write from Kafka consumer. ES may lag Cassandra by seconds. Shard by day — most searches are time-bounded. Don't index all tags (cardinality explosion).

Query Service

Cassandra for traceId lookup (single partition read). ES for search by service/duration/error. Redis cache for recently queried traces. Stateless, horizontally scalable.

Storage Tiering

Hot (0–72h): Cassandra full-fidelity. Cold (72h–1yr): S3 Parquet sampled traces. Dependency graph pre-aggregated with longer TTL (1yr).

10. Self-Check

+
How does trace context cross service boundaries?
Propagation
The W3C traceparent header is injected by the SDK on every outbound call (HTTP, gRPC, Kafka message headers). The header carries: version (00), traceId (16 bytes hex), parentSpanId (8 bytes hex of the caller's span), and flags (01 = sampled). The receiving service extracts it and creates a child span with the same trace_id and the caller's span_id as parent_span_id. This creates the parent-child relationship that allows reconstructing the full call graph. OpenTelemetry's SDK handles injection/extraction automatically — the developer just wraps code in a span context.
+
What is the key trade-off between head-based and tail-based sampling?
Sampling
Head-based sampling makes the decision at the root span (first service) before the trace is complete. It's simple and low-overhead but blind — you randomly sample and may miss all examples of a rare slow request or a specific error pattern. Tail-based sampling buffers all spans in memory (Redis, ~30s window) and makes an informed sampling decision after the trace is complete — capturing 100% of errors and all traces exceeding a latency threshold. The cost: significant memory overhead per collector, and consistent routing of all trace_id spans to the same collector instance (via consistent hashing).
+
Why is Cassandra's partition key set to trace_id, not service_name or timestamp?
Storage
All spans for a trace must be co-located on the same Cassandra partition for fast trace reconstruction. A single trace query (GET /api/traces/{traceId}) should be a single partition read — O(1), hitting a single Cassandra node. If you partition by service_name, the 10 spans of one trace are scattered across nodes; reconstructing the trace requires N queries across N nodes. If you partition by timestamp, hot partitions form on the current-time shard. trace_id hashes uniformly across nodes and collocates all related spans. Elasticsearch handles the search-by-service use case separately.
+
Why is 100% tracing not feasible at high scale?
Scale
At 1M req/sec with 10 spans/request, 100% tracing = 10M spans/sec. At ~1 KB/span = 10 GB/sec of data to ingest, store, and index. That requires ~864 TB of storage per day, plus the compute to index it all in Elasticsearch. The overhead on each instrumented service also compounds — every outbound call adds a traceparent header, each span adds serialization cost. 1% sampling = 100K spans/sec = 100 MB/sec = manageable, while still providing statistically representative coverage for performance debugging. Adaptive sampling ensures critical low-volume operations (e.g., checkout) are still sampled at 100%.
+
What is OpenTelemetry and why does it matter architecturally?
Standards
OpenTelemetry (OTEL) is a CNCF-standard vendor-neutral instrumentation framework covering traces, metrics, and logs. The key architectural value: instrument your services once with the OTEL SDK emitting OTLP (OpenTelemetry Protocol), then switch backends (Jaeger → Datadog → Honeycomb) by changing only the exporter configuration — no code changes. Without OTEL, every backend switch requires re-instrumentation of every service. The OTEL Collector also acts as a pipeline (receive, transform, sample, export to multiple backends simultaneously) that decouples your services from the choice of storage backend.

11. Interview Tips

"How does trace context cross service boundaries?"

W3C traceparent header injected by SDK on every outbound call. Receiver extracts it and creates a child span with the same trace_id and caller's span_id as parent_span_id.

"Why Cassandra partition key = trace_id?"

All spans for a trace must be co-located for fast reconstruction. Single trace read = single partition read (one Cassandra node). Partitioning by service would scatter spans.

"Head-based vs tail-based — key trade-off?"

Head-based: simple, low-overhead, but blind (may miss all slow request examples). Tail-based: captures all errors/slow traces, but requires buffering all spans + consistent routing.

"Why not 100% tracing?"

10M spans/sec at 1 KB each = 10 GB/sec = 864 TB/day. Cost prohibitive. 1% sampling → 100 MB/sec = manageable while still giving statistical coverage.

"How do you correlate traces with logs and metrics?"

Inject trace_id into every log line (MDC in Java). In Grafana: high-latency metric spike → query traces in that time window → jump to logs with that trace_id.

"What is OpenTelemetry?"

Vendor-neutral CNCF standard. Instrument once with OTLP; switch backends by changing exporter config. Avoids lock-in at instrumentation layer. OTEL Collector provides pipeline flexibility.
