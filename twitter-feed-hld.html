07

The Fan-out Problem

When a user posts a tweet, how do we get it into 300M+ followers' home timelines efficiently? This is the core design challenge.

Strategy A
Fan-out on Write (Push)
On tweet creation, push tweet_id to every follower's Redis sorted set via Kafka consumers.
✓ Fast timeline reads (O(1) Redis lookup)
✗ Celebrity problem: 10M followers = 10M Redis writes per tweet
Strategy B
Fan-out on Read (Pull)
On timeline request, fetch recent tweets from all followed users, merge sort by timestamp.
✓ No write amplification; celebrity tweets handled naturally
✗ Slow reads if following 5000 users; N DB queries per request
Strategy C — Recommended
Hybrid Approach
Push for normal users (<10K followers). Pull for celebrities (>10K followers). Merge both sets on timeline read.
✓ Best of both: fast reads, controlled write amplification
✗ Complexity: need to identify celebrities, merge at read time

    void process(TweetEvent event) {
        Long authorId = event.getAuthorId();
        // Only push if author is not a celebrity
        if (userService.getFollowerCount(authorId) < 10_000) {
            List<Long> followers = followService.getFollowerIds(authorId);

                redis.zadd("timeline:" + followerId, score, tweetId);
                redis.zremrangeByRank("timeline:" + followerId, 0, -801);
            }
        }

08

Scaling & Bottlenecks

🎯
Kafka for Async Fan-out
Tweet creation publishes to Kafka. Multiple consumer groups fan out asynchronously — timeline workers, notification workers, analytics workers. Decouples tweet write from side effects.
⚡
Redis Sorted Sets — Timeline Cache
1.9 TB of timeline data across Redis cluster. ZREVRANGE is O(log N + M). Keep only 800 entries per user. Evict LRU users (inactive for 30 days) from cache.
🗄️
Cassandra for Tweets
Partition by user_id, cluster by tweet_id desc. All tweets by one user on same partition — fast range scans. Replication factor 3 for HA.
🌐
CDN + S3 for Media
150 TB/day of media. Store originals in S3. Transcode to multiple resolutions. Serve via CloudFront CDN. Client uploads directly to S3 presigned URL.

Self Check

Answer before revealing. These are real interview questions.

What is fan-out on write vs fan-out on read? Which does Twitter use?Core Design+

Fan-out on write: when a user posts, immediately push tweet ID to all followers' timeline caches. Pro: O(1) read (just ZRANGE). Con: O(followers) write cost — 150M follower account = 150M Redis writes per tweet.

Fan-out on read: don't pre-compute; at read time, fetch tweets from each followee and merge. Pro: O(1) write. Con: O(followees) read cost — following 1000 accounts = 1000 Cassandra reads per timeline load.

Twitter uses hybrid: fan-out on write for regular users (<1M followers), fan-out on read for celebrities. At read time, merge celebrity tweets in. This balances write amplification vs read latency.

How do you handle the 'celebrity problem' at 150M followers?Scale Challenge+

Pure fan-out on write is infeasible: a single Beyonce tweet triggers 150M Redis ZADD operations. Memory: each timeline entry is ~48 bytes × 150M = 7.2 GB per celebrity tweet just for fan-out. Time: even at 1M ops/sec, 150M operations = 150 seconds.

Solution: Celebrity threshold (~1M followers). Above threshold: skip write fan-out. Instead, each celebrity's recent tweets are cached separately. At read time, the Feed API fetches the last N tweets from each celebrity you follow and merges into your timeline. This is called "pull on demand" or "lazy fan-out."

What does the Redis timeline store — tweets or tweet IDs?Data Model+

Tweet IDs only. Storing full tweet objects in every follower's Redis list would use enormous memory (a tweet can be 500 bytes × 500M users = 250 GB just for Redis). Instead, store only the tweet_id (8 bytes) in a sorted set (score = timestamp). Then hydrate by multi-fetching tweet content from Cassandra at read time.

This is the "tweet ID fan-out" pattern. Cassandra lookup by tweet_id is a primary key lookup — extremely fast and parallelizable.

A user posts a tweet. Walk through the complete write path.Write Path+

(1) POST /tweet → API server validates, creates tweet object, assigns Snowflake ID. (2) Write tweet to Cassandra (user_id partition). (3) Publish TweetCreatedEvent to Kafka. (4) Fan-out workers consume from Kafka: for each follower (fetched from User Graph Service), ZADD timeline:{followerId} {tweetId} {score=timestamp}. (5) Trim old entries: ZREMRANGEBYRANK timeline:{followerId} 0 -801 (keep last 800). (6) Push real-time notification via WebSocket if follower is online. Celebrity accounts skip steps 4-5.

How does the Trending Topics feature work?Feature Design+

Trending requires counting hashtag frequency in near-real-time. Naive: SELECT hashtag, COUNT(*) GROUP BY hashtag WHERE created_at > NOW()-1h — too slow on 500M tweets/day.

Production: (1) All tweets published to Kafka. (2) A Flink/Spark Streaming job counts hashtag occurrences in tumbling 1-minute windows with a 1-hour lookback. (3) Top-K hashtags computed using a min-heap (keep top 50). (4) Results written to Redis with 1-minute TTL. (5) API reads from Redis. This gives trending updated every minute with sub-second reads.
