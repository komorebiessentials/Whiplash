01 What Is It

Observability is the ability to understand the internal state of a system from its external outputs — without having to deploy new code or SSH into a server. A system is observable if you can answer arbitrary questions about its behavior purely from the data it emits.

Monitoring vs Observability: Monitoring tells you when something is wrong (known unknowns — dashboards, alerts). Observability lets you understand why it's wrong (unknown unknowns — you can explore any question from the data). Monitoring is a subset of observability.

📊
Pillar 1

Metrics

Aggregated numeric time-series. Cheap to store. Great for alerting and dashboards.

📋
Pillar 2

Logs

Structured event records. High detail, high cost. Essential for debugging specific events.

🔗
Pillar 3

Traces

Request flows across services. Shows exactly where time was spent in distributed calls.

02 The Problem It Solves

In a monolith, you can attach a debugger or add a print statement and redeploy. In a distributed system with 50 services, a single user request may touch 15 services. A user reports "it's slow." Without observability:

  • You don't know which of 15 services is slow
  • You don't know if it's the DB, a downstream API, or GC pauses
  • You can't reproduce it because it only happens at peak load
  • By the time you SSH in, the incident is over

With full observability: traces show the slow DB query, metrics show the query rate spike, logs show the specific query that timed out. MTTR drops from hours to minutes.

03 How It Works

The OpenTelemetry SDK instruments your service once. The OTel Collector receives all signals and exports to multiple backends. The key insight: a single trace_id propagated via HTTP headers ties together the metric spike, the trace showing which span was slow, and the log line with the error detail.

04 The Three Pillars + SLI/SLO

1. Metrics (Prometheus / Grafana)

Metric Types

  • Counter: monotonically increasing (http_requests_total). Use rate() to get per-second rate.
  • Gauge: can go up or down (jvm_memory_used_bytes, active_connections).
  • Histogram: samples in configurable buckets (http_request_duration_seconds_bucket). Enables histogram_quantile(0.99, ...) across multiple instances.
  • Summary: pre-computed quantiles on client. Can't aggregate across replicas — prefer Histogram.

Cardinality explosion: Each unique combination of label values is a separate time series. Labels like user_id or session_id create millions of series and kill Prometheus. Rule: labels should have bounded, low-cardinality values.

2. Logs (ELK / Loki)

Structured vs Unstructured

Always emit structured JSON logs. {"level":"ERROR","trace_id":"abc123","msg":"DB timeout","duration_ms":5000} is queryable. Plain text strings require fragile regex parsing.

Correlation ID: Inject trace_id into MDC at request start. Every log line in that request thread automatically includes it → grep across 50 services for one request.

ELK vs Loki: Elasticsearch indexes every field — expensive but powerful full-text search. Loki stores logs as compressed chunks indexed only by labels (like Prometheus) — 10x cheaper, but log content is not indexed.

3. Distributed Tracing (OpenTelemetry / Jaeger)

Trace Anatomy

A Trace is a tree of Spans. Each span represents one operation: an HTTP call, a DB query, a cache lookup. Spans have: start_time, duration, tags (key-value metadata), events (log lines within the span), status (OK/ERROR), and parent_span_id.

Propagation: The W3C traceparent header carries trace_id + span_id through every HTTP call. Your service reads it, creates a child span, and passes it to downstream calls.

Sampling strategies: Head-based (decide at request start — simple but misses rare errors). Tail-based (buffer all spans, decide after trace completes — catch errors and slow traces, but requires a stateful buffer like Tempo).

SLI / SLO / SLA / Error Budget

SRE Reliability Framework

SLI
Service Level Indicator — the metric you measure. E.g., "fraction of requests with latency <200ms" or "error rate".
SLO
Target you commit to internally. E.g., "99.9% of requests succeed". Stricter than SLA — violation = stop feature work.
SLA
Contractual commitment with customers. Usually looser than SLO. Breach = credits/refunds.
Error Budget
SLO=99.9% → budget=0.1% of time = 43.8 min/month. Shared between dev and SRE. When exhausted → freeze risky deploys.

Burn rate alerting: A 1x burn rate means you'll exhaust the budget at exactly the end of the period. Alert at 2x burn rate for a fast (1-hour) alert; 14.4x for a very fast (5-minute) alert. This avoids alert fatigue from transient blips.

06 Code Examples

// Micrometer metrics (Spring Boot auto-configures Prometheus)
@Component
public class OrderService {
    private final Counter orderCounter;
    private final Timer checkoutTimer;
    private final AtomicLong pendingOrders = new AtomicLong(0);

    public OrderService(MeterRegistry registry) {
        // Counter: total orders by status
        orderCounter = Counter.builder("orders.total")
            .tag("status", "created")
            .register(registry);

        // Timer: checkout duration histogram
        checkoutTimer = Timer.builder("checkout.duration")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);

        // Gauge: current pending orders
        Gauge.builder("orders.pending", pendingOrders, AtomicLong::get)
            .register(registry);
    }

    public Order checkout(Cart cart) {
        return checkoutTimer.record(() -> {
            pendingOrders.incrementAndGet();
            try {
                Order order = createOrder(cart);
                orderCounter.increment();
                return order;
            } finally { pendingOrders.decrementAndGet(); }
        });
    }
}

// PromQL: p99 checkout latency over 5 min
// histogram_quantile(0.99, rate(checkout_duration_seconds_bucket[5m]))
// Logback JSON + MDC for correlation IDs

// Filter: inject trace context into MDC on every request
@Component
public class TraceContextFilter implements Filter {
    public void doFilter(ServletRequest req, ...) {
        String traceId = extractTraceId(req)  // from traceparent header
            .orElse(UUID.randomUUID().toString());
        MDC.put("trace_id", traceId);
        MDC.put("service", "order-service");
        try { chain.doFilter(req, res); }
        finally { MDC.clear(); }
    }
}

// Usage — MDC fields auto-appear in every log line
private static final Logger log = LoggerFactory.getLogger(OrderService.class);

log.info("Order created", Map.of(
    "order_id", order.getId(),
    "user_id", userId,
    "amount_cents", amount  // always integers, not floats
));

// JSON output:
// {"level":"INFO","trace_id":"abc123","service":"order-service",
//  "order_id":"ord-456","user_id":"u-789","amount_cents":4999,
//  "msg":"Order created","timestamp":"2024-01-15T10:30:00.123Z"}
// OpenTelemetry Java SDK — manual instrumentation
private static final Tracer tracer =
    GlobalOpenTelemetry.getTracer("order-service", "1.0.0");

public Order processOrder(Cart cart) {
    // Start span (child of incoming traceparent)
    Span span = tracer.spanBuilder("processOrder")
        .setAttribute("cart.item_count", cart.size())
        .setAttribute("user.id", cart.getUserId())
        .startSpan();

    try (Scope scope = span.makeCurrent()) {
        Order order = createOrder(cart);

        // Add event (annotate timeline)
        span.addEvent("inventory_reserved");

        // Downstream call — context propagates automatically
        paymentClient.charge(order);  // OTel injects traceparent header

        span.setStatus(StatusCode.OK);
        return order;

    } catch (Exception e) {
        span.recordException(e);       // attaches stack trace to span
        span.setStatus(StatusCode.ERROR, e.getMessage());
        throw e;
    } finally { span.end(); }
}
# Prometheus alerting rules for SLO-based alerts

# SLI: success rate = (total - errors) / total
# SLO: 99.9% success rate

groups:
  - name: slo-alerts
    rules:
      # 1h burn rate > 2x → consume budget 2x fast
      - alert: SLOBurnRateFast
        expr: |
          (
            rate(http_requests_total{code=~"5.."}[1h])
            /
            rate(http_requests_total[1h])
          ) > 0.002   # 0.1% budget × 2x burn rate
        annotations:
          summary: "SLO burn rate >2x for 1h"

      # Missing metric alert — alert if no data
      - alert: MetricsMissing
        expr: absent(http_requests_total) == 1
        for: 5m

      # p99 latency SLO: 99th percentile < 500ms
      - alert: HighP99Latency
        expr: |
          histogram_quantile(0.99,
            rate(http_request_duration_seconds_bucket[5m])
          ) > 0.5
        for: 2m

07 Real-World Usage

Google SRE

Origin of SLI/SLO

The SRE book coined error budgets. Google runs SLOs on every service. Error budget depletion triggers feature freeze.

OpenTelemetry

CNCF Standard

Vendor-neutral SDK. Instrument once, export to Datadog, Jaeger, Honeycomb, or any OTLP-compatible backend.

Honeycomb

Event-Based Observability

Stores every event (not pre-aggregated metrics). Query arbitrary high-cardinality dimensions after the fact. Invented by Charity Majors.

Datadog

All-in-One

Metrics, logs, traces in one platform with automatic correlation. APM traces linked to logs via trace_id injection.

08 Interview Tips

🔍
"What's the difference between monitoring and observability?"
Monitoring = known unknowns. You define dashboards and alerts for failures you can predict. Observability = unknown unknowns. You can ask arbitrary questions about system behavior from the telemetry data, without deploying new instrumentation.
🐌
"How would you debug high p99 latency in production?"
1) Check distributed traces for the slow span — is it DB, external API, or serialization? 2) Check metrics for DB connection pool saturation, GC pause time, thread pool queue depth at the same timestamp. 3) Check structured logs for timeout/retry messages correlated by trace_id.
📈
"What is cardinality in Prometheus and why does it matter?"
Each unique label value combination creates a separate time series. A label with 1M unique user IDs × 10 metrics = 10M time series. Prometheus stores all active series in RAM. Cardinality explosion causes OOM crashes. Rule: never use user_id, request_id, or URLs as label values.
💰
"Why use error budgets instead of just setting SLAs?"
Error budgets quantify reliability as a shared resource. Devs want to ship features (consume budget), SREs want stability (preserve budget). When budget is healthy, move fast. When exhausted, freeze risky deploys. This creates aligned incentives instead of dev vs ops conflict.
🎯
"What is tail-based sampling in tracing?"
Head-based: decide to sample at request start (e.g., 1%). Cheap but misses rare errors. Tail-based: buffer all spans from a trace, decide after the trace completes — capture 100% of errors and slow traces. Requires a stateful collector (Grafana Tempo or Jaeger with adaptive sampling). Much more useful for debugging.
