CAP Theorem

A distributed system can only guarantee 2 of 3 properties. Understanding this forces explicit trade-off decisions in every system design interview.

What Is ItThe ProblemHow It Works CP vs APPACELCCode Real WorldInterview Tips
01

What Is It

Proposed by Eric Brewer (2000), proved by Gilbert & Lynch (2002). Every distributed data store must choose which two of three guarantees to provide.

C — Consistency
Every read sees the latest write
Every read receives the most recent write or an error. No stale reads. All nodes see the same data at the same time.
A — Availability
Every request gets a response
Every request (to a non-failing node) receives a response — not an error. The response may be stale. The system never returns an error just to preserve consistency.
P — Partition Tolerance
System continues despite network splits
The system continues operating even when an arbitrary number of messages between nodes are dropped or delayed. Networks DO partition — this is not optional in any real distributed system.
Key insight: P is not really a choice. Networks partition. You must decide: when a partition happens, do you return an error (C over A) or serve potentially stale data (A over C)? The real choice is CP or AP.
02

The Problem It Solves

Before CAP, engineers assumed they could build systems that were always consistent AND always available AND fault-tolerant. CAP proved this is impossible.

Without CAP thinking

  • Engineers over-promise: "our DB is always consistent AND never goes down"
  • Network partition happens → forced into an undocumented, surprising failure mode
  • Users experience worse outcomes than if the trade-off was planned

With CAP thinking

  • Explicit decision: "we choose AP because stale data is better than an outage for this use case"
  • Partition happens → system behaves as designed, users see stale data instead of errors
  • Clients and SLAs are written around the actual guarantees
03

How It Works

The partition scenario: Node A and Node B can no longer communicate.

Choose C (over A): Node B refuses to serve the read. Returns error or blocks until partition heals. Data is always correct — but users see failures.
Choose A (over C): Node B serves stale data (val=10). Users get a response — but it may be wrong. Nodes reconcile when partition heals.

PACELC — Beyond CAP

CAP only describes behavior during partitions. PACELC also asks: even when there's no partition, what's your trade-off between latency and consistency?

PACELC: If Partition → choose Availability or Consistency. Else (normal operation) → choose Latency or Consistency.

Code

Illustrating CP and AP behaviors in a simple two-node replication scenario.

// CP — refuse reads/writes if cannot reach quorum
class CPDataStore {
    private final List<Node> replicas;
    private final int quorum; // majority = replicas.size()/2 + 1

    public String read(String key) {
        int reachable = countReachable();
        if (reachable < quorum)
            throw new QuorumUnavailableException("Cannot guarantee consistency");
        return primary.read(key); // read from primary (always fresh)
    }

    public void write(String key, String value) {
        int acks = 0;
        for (Node node : replicas) {
            try { node.write(key, value); acks++; } catch (Exception e) {}
        }
        if (acks < quorum)
            throw new WriteFailedException("Could not reach quorum");
    }
}
// AP — always serve, accept stale reads
class APDataStore {
    private final List<Node> replicas;

    public String read(String key) {
        // Try any replica — return first response (may be stale)
        for (Node node : replicas) {
            try { return node.read(key); } catch (Exception e) {}
        }
        throw new AllNodesDownException(); // only fails if ALL nodes down
    }

    public void write(String key, String value) {
        // Write to any available node — replicate async (eventual consistency)
        for (Node node : replicas) {
            try {
                node.write(key, value); // fire-and-forget on others
                asyncReplicate(key, value, replicas);
                return;
            } catch (Exception e) {}
        }
    }
}
// Eventual consistency reconciliation — Last Write Wins (LWW)
class VersionedValue {
    String value;
    long timestamp; // wall clock or logical clock (Lamport)
}

class ReplicationHandler {
    // When partition heals, nodes exchange their versions
    public VersionedValue reconcile(VersionedValue local, VersionedValue remote) {
        // Last Write Wins (LWW) — higher timestamp wins
        return remote.timestamp > local.timestamp ? remote : local;
    }
}

Real-World Usage

Banking
CP always
Never show wrong balance. Outage during partition is better than incorrect account state.
Social Media
AP preferred
Seeing a post 2 seconds late is fine. High availability matters more than perfect consistency.
E-Commerce Cart
AP (Amazon Dynamo)
Amazon chose AP for shopping carts — a lost item is worse than a duplicate item (fixable at checkout).
Distributed Locks
CP (Zookeeper)
A distributed lock that gives two nodes permission simultaneously is worse than locking being unavailable.
08

Interview Tips

1
Never say "we want all 3." This signals you don't understand CAP. P is not optional — networks partition. Say "we choose CP because X" or "we choose AP because Y."
2
CAP-C ≠ ACID-C. CAP Consistency = linearizability (reads see the latest write globally). ACID Consistency = integrity constraints (foreign keys, etc.). They're different things — interviewers test this.
3
Eventual consistency is AP with a guarantee. "Eventually consistent" means nodes WILL converge to the same state — it's not "no consistency." The window is typically milliseconds to seconds.
4
Mention PACELC for SDE III level. "CAP only covers partition behavior. PACELC also asks about latency vs consistency in normal operation — Spanner is PC/EC, DynamoDB is PA/EL."
5
Cassandra is tunable. It's AP by default but lets you tune with ConsistencyLevel.QUORUM to get CP-like behavior per-query. Mention this to show nuance — real systems don't always fit neatly.
6
Always connect to the use case. "For a payment service I'd choose CP — a failed transaction is better than a duplicate charge. For a news feed I'd choose AP — a 5-second delay in seeing a post is acceptable."
