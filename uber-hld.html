07

Key Design Decisions

Geospatial Indexing

  • Geohash: Earth divided into grid cells (5-7 char = 4km sq)
  • Nearby cells share string prefix -- prefix scan finds neighbours
  • Redis GEOADD uses sorted set with geo-encoded score
  • GEORADIUS returns drivers within N km in O(N + log M)
  • Alternative: quadtree (dynamic cells), Google S2 (spherical)
  • Redis chosen: sub-millisecond lookups, 320 MB fits in RAM

Driver Matching Algorithm

  • Find all available drivers within 2km via GEORADIUS
  • Rank by ETA (road distance, not straight line) via Maps API
  • Offer to nearest driver with 10s timeout
  • If declined, try next -- extend radius to 5km after 3 attempts
  • Mark driver as "on offer" in Redis (SETNX) to prevent double-send
  • Repeated declines affect match priority scoring

Location Service Architecture

  • Drivers maintain persistent WebSocket to Location Server
  • Consistent hashing routes each driver to specific server
  • Server fail: driver reconnects to hash ring neighbor
  • Each server holds in-memory map: driverId to WS connection
  • Location updates batched: write to Redis every 5s
  • 10K drivers per instance -- 500 instances for 5M drivers

Trip State Machine

  • REQUESTED: Rider confirmed, surge acknowledged
  • MATCHED: Driver accepted; marked unavailable in Redis
  • EN_ROUTE: Driver heading to pickup
  • IN_PROGRESS: Rider in car, trip underway
  • COMPLETED: Fare calculated; Kafka fires payment + ratings
  • CANCELLED: Either party cancels; driver re-marked available
// Driver matching pseudocode
public Driver matchDriver(RideRequest request) {
    List candidates = redis.geoRadius(
        "drivers:active", request.pickup, 2, KM, 20
    );
    return candidates.stream()
        .filter(d -> d.status == AVAILABLE)
        .sorted(Comparator.comparingInt(d ->
            mapsApi.getETA(d.location, request.pickup)))
        .findFirst()
        .map(d -> {
            redis.setex("offer:" + d.id, 10, request.id); // 10s lock
            notificationService.sendOffer(d, request);
            return d;
        }).orElse(null);
}

Scaling and Bottlenecks

📍

Redis Geo Cluster

Partition by geographic region (city-level shards). Prevents hotspot on a single node. 320 MB holds all 5M driver locations -- sharding is for write throughput, not capacity.

🔗

Consistent Hashing

Route each driver to a specific WebSocket Location Server. Server failure only disconnects its slice of drivers -- they reconnect to the hash ring neighbor. No thundering herd.

⚡

Kafka Async

Trip completed -- Kafka event -- async payment processing, rating requests, driver stats updates. Decouples ride completion from side effects.

🗺

Maps API Caching

Pre-compute ETA for popular origin-destination pairs. Cache road graph locally. External Maps API only for novel routes (rate-limited by provider).

💳

Payment Idempotency

Each trip charge has an idempotency key (tripId + version). Retries on network failure won't double-charge. Stripe deduplicates on their side too.

📊

Read Replicas

PostgreSQL read replicas for ride history and profiles. Write path (trips, payments) hits primary only. Reads scale horizontally across replicas.

Self Check

Common Uber design interview follow-up questions. Answer before expanding.

Why store driver locations in Redis instead of PostgreSQL? Geospatial +

PostgreSQL with PostGIS can handle geospatial queries, but at 1M writes/sec it would require extreme sharding and still struggle with p99 latency. Redis GEOADD stores coordinates as sorted set scores and executes GEORADIUS in O(N + log M) -- sub-millisecond. More importantly, driver locations are ephemeral: they don't need durability, ACID transactions, or history. Redis is purpose-built for this: in-memory, no disk I/O. 320 MB holds all 5M drivers. If Redis restarts, drivers reconnect and re-register -- no meaningful data loss.

What prevents two matching services from double-assigning the same driver? Concurrency +

The Matching Service uses Redis SETNX (set if not exists) with a 10-second TTL on key offer:{driverId} before sending the offer notification. If two Matching Service instances race to offer the same driver, the second SETNX returns 0 (failure) and that instance skips to the next candidate driver. This is a distributed lock. When the offer expires (10s timeout) or driver declines, both the Redis key and driver status are cleared -- making the driver available for the next request.

How does consistent hashing help with the Location Service? Distributed Systems +

With 500 Location Server instances (10K drivers each), you need to route reconnects to the same server that holds the driver's WebSocket. Consistent hashing maps hash(driverId) mod ring to a specific server node. When a server fails, only the approximately 10K drivers on that node reconnect -- they get re-hashed to the neighboring node on the ring. Without consistent hashing, any server failure could cause all 5M drivers to randomly distribute across servers, destroying all in-memory connection state. The ring minimizes disruption to 1/N of drivers per failure.

How is surge pricing computed and stored at scale? Algorithm +

A Surge Service runs every 60 seconds per city zone (geohash cell). It reads two Redis counters: pending ride requests in the last 5 minutes for the zone, and available drivers in the zone. surge = min(3.0, demand / supply). Results are stored as surge:{geohash} with 90-second TTL. When a rider requests a ride, Trip Service reads the pickup geohash's surge value from Redis (single sub-ms read). The Surge Service is stateless -- multiple instances produce the same result (idempotent). No coordination needed; last-write-wins on Redis.

How do you handle payment failure after a trip completes? Reliability +

Trip completion publishes a Kafka event with idempotency key (tripId + version). The Payment Consumer calls Stripe with this key -- Stripe deduplicates on their end. If Stripe call fails, the consumer does NOT commit the Kafka offset, so the message is redelivered. Exponential backoff: 1s, 2s, 4s, 8s retry. After 5 retries, the event moves to a Dead Letter Queue (DLQ) Kafka topic. A DLQ consumer alerts ops and sets trip status to PAYMENT_FAILED. The rider is notified and asked to resolve manually. Crucially, the trip stays COMPLETED -- payment failure never un-completes a trip.

Interview Tips

🌍
Geospatial indexing is the crux. Explain geohash (Earth divided into grid cells, nearby cells share string prefix) and why Redis GEOADD is the right tool -- sub-millisecond lookups, 320 MB fits in RAM for 5M drivers.
🤝
Matching is a two-sided marketplace. Don't just say "find nearest driver." Discuss offer timeout (10s), driver declining (try next), radius expansion, and how repeated declines affect match priority. This shows operational reality.
🔄
Location data is ephemeral. Driver locations live only in Redis -- no database persistence. Only trip data (start/end coordinates) persists. Common trick question: "how do you store location history?" -- you don't, except for completed trips.
💰
Surge pricing = supply-demand signal. Frame it as balancing supply and demand in a geohash zone, not just revenue extraction. Surge incentivizes drivers toward demand zones. Computed per zone every 60s from request/driver ratio.
📐
Estimation anchor. 5M drivers x 1 update/5s = 1M location writes/sec. This is the dominant load -- it drives the Redis-based architecture. If you used SQL for live driver locations, it would melt at this QPS. Lead with this calculation.
