Requirements

Scope precisely — 2048 seems simple but has several design decisions.

Functional

4×4 grid with tiles containing powers of 2

Functional

Slide in 4 directions: UP, DOWN, LEFT, RIGHT

Functional

Equal adjacent tiles merge into their sum (once per move)

Functional

Spawn tile (2 or 4) after each valid (board-changing) move

Functional

Win: 2048 tile created. Over: no valid moves remain.

Functional

Score = sum of all merged tile values

Non-Functional

Testable — injectable Random for deterministic spawning

Non-Functional

Extensible — configurable board size and win target

02

Clarifying Questions

Each answer shapes a design decision.

  • ?Does winning end the game or let the player continue?Standard: player can keep playing after hitting 2048. Affects GameState — distinguish WON-but-continuing from OVER.
  • ?What if a slide doesn't change the board?No tile is spawned. Only spawn after a valid (board-changing) move. Critical edge case.
  • ?New tile probabilities?Standard: 90% chance of 2, 10% chance of 4. Confirm so spawn logic is explicit.
  • ?Can a tile merge twice in one move?No — [2,2,2,2] → [4,4,0,0], not [8,0,0,0]. Each tile participates in at most one merge per move.

05

Key Flows

makeMove() is the complete game loop — walk through it step by step in the interview.

1. Validate state
If state == OVER, reject. If state == WON and continue-after-win is enabled, allow.
2. board.slide(direction) → boolean
Transform orientation (transpose/reverse), apply slideLine() to each row, reverse transform. Returns true if any tile moved or merged.
3. If no change — skip spawn
slide() returns false → board unchanged. Do NOT spawn a tile. Return current state unchanged.
4. board.spawnTile()
Pick random empty cell. Place 2 (90%) or 4 (10%). Board guaranteed to have space since slide() changed it.
5. Determine new state
board.hasWinTile(2048) → WON. !board.hasValidMoves() → OVER. Otherwise ACTIVE.
slideLine example — LEFT:
Input: [2, 0, 2, 4]
Step 1 compact (remove zeros): [2, 2, 4]
Step 2 merge equals: [4, 4] — score += 4, index i skipped
Step 3 pad to length 4: [4, 4, 0, 0]
06

Code

The entire game is slideLine() + direction transforms. Get these right and everything else is glue.

Core algorithm — slideLine, spawnTile, hasValidMoves
// [2,0,2,4] → LEFT → [4,4,0,0], score += 4
private int[] slideLine(int[] line) {
    // compact — remove zeros
    int[] c = Arrays.stream(line).filter(x -> x != 0).toArray();
    int[] result = new int[line.length];
    int idx = 0;
    for (int i = 0; i < c.length; i++) {
        if (i + 1 < c.length && c[i] == c[i + 1]) {
            result[idx++] = c[i] * 2;
            score += c[i] * 2;
            i++; // skip merged tile — no double merge
        } else {
            result[idx++] = c[i];
        }
    }
    return result; // remaining positions are 0
}
public void spawnTile() {
    List<int[]> empty = new ArrayList<>();
    for (int r = 0; r < size; r++)
        for (int c = 0; c < size; c++)
            if (grid[r][c] == 0) empty.add(new int[]{r, c});
    if (empty.isEmpty()) return;
    int[] cell = empty.get(random.nextInt(empty.size()));
    // 10% chance of 4, 90% chance of 2
    grid[cell[0]][cell[1]] = (random.nextInt(10) == 0) ? 4 : 2;
}
public boolean hasValidMoves() {
    // Part 1: any empty cell = moves exist
    for (int r = 0; r < size; r++)
        for (int c = 0; c < size; c++)
            if (grid[r][c] == 0) return true;
    // Part 2: any adjacent equal values (horizontal or vertical)
    for (int r = 0; r < size; r++) {
        for (int c = 0; c < size; c++) {
            if (c + 1 < size && grid[r][c] == grid[r][c+1]) return true;
            if (r + 1 < size && grid[r][c] == grid[r+1][c]) return true;
        }
    }
    return false;
}
Direction handling + makeMove
public boolean slide(Direction dir) {
    // All directions reduced to "slide LEFT" via transforms:
    // LEFT  = slideLeft (no transform)
    // RIGHT = reverseRows → slideLeft → reverseRows
    // UP    = transpose → slideLeft → transpose
    // DOWN  = transpose → reverseRows → slideLeft → reverseRows → transpose

    if (dir == Direction.UP   || dir == Direction.DOWN)  transpose();
    if (dir == Direction.RIGHT || dir == Direction.DOWN) reverseRows();

    boolean changed = false;
    for (int r = 0; r < size; r++) {
        int[] newRow = slideLine(grid[r]);
        if (!Arrays.equals(grid[r], newRow)) changed = true;
        grid[r] = newRow;
    }

    if (dir == Direction.RIGHT || dir == Direction.DOWN) reverseRows();
    if (dir == Direction.UP   || dir == Direction.DOWN)  transpose();
    return changed;
}
public GameState makeMove(Direction dir) {
    if (state == GameState.OVER) return state;

    boolean moved = board.slide(dir);
    if (!moved) return state; // no change — no spawn, no state change

    board.spawnTile();

    if (board.hasWinTile(winTarget))  state = GameState.WON;
    else if (!board.hasValidMoves()) state = GameState.OVER;
    else                                state = GameState.ACTIVE;
    return state;
}

Design Patterns

Name these in the interview — vocabulary signals senior-level thinking.

Behavioral
State
GameState (ACTIVE/WON/OVER) — makeMove() guards on state. Win/over transitions happen exactly once.
Behavioral
Command
Direction as a command object. Can wrap in MoveCommand(direction, boardBefore) to support undo.
Behavioral
Template Method
slide() always calls slideLine() per row. Direction handling via transpose/reverse is the invariant template — the line algorithm is the varying step.
Behavioral
Observer
Game notifies UI on each state/grid change. Decouples rendering from game logic — good for web/mobile frontends.

Concurrency

Single-player game — one edge case in mobile/web contexts.

The one real risk: Rapid swipes on a touchscreen can trigger two makeMove() calls simultaneously — the second executes on a partially-updated grid.

09

Interview Tips

SDE I problem — the differentiator is algorithmic clarity and clean code.

1
slideLine() is the whole problem. Implement it first, test with [2,0,2,4]→[4,4,0,0] and [2,2,2,2]→[4,4,0,0]. If this works, slide() is just transforms around it.
2
The transpose trick = DRY code. Rather than 4 separate slide implementations, transpose/reverse to reduce everything to "slide left." Interviewers notice this — it's the elegant approach.
3
No tile merges twice in one move. [2,2,2,2]→[4,4,0,0] not [8,0,0,0]. The `i++` skip after a merge is what enforces this rule.
4
Only spawn after a board-changing move. slide() returns false → no spawn. Standard game rule that candidates often miss.
5
hasValidMoves() has two parts. (1) Any empty cell. (2) Any adjacent equal tiles. Missing part 2 is the most common bug — a full board with [2,2] horizontally still has moves.
