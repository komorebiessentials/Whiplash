Section 07

Key Design Decisions

At-Least-Once Delivery with Idempotency
Kafka guarantees at-least-once delivery. Workers only commit offset after the external provider confirms receipt. On failure, the message is reprocessed — so workers must be idempotent.

Each notification gets an idempotency_key = SHA-256(userId + templateId + timestamp_bucket). Before sending, the worker checks Redis: if key exists → skip. Key expires after 24h. This prevents double-sends even on consumer restart.
Priority Queues
Kafka topics split by priority:
  • push.critical — OTPs, security alerts → dedicated consumer group, max consumers
  • push.normal — social likes, marketing → shared consumers, may lag under load
  • email, sms — separate topics, rate-limited by provider capacity
Critical notifications get dedicated consumer instances so marketing bursts never delay OTPs.
Retry & Dead-Letter Queue
On provider failure: exponential backoff (1s → 2s → 4s → 8s, max 3 retries). After exhausting retries, message moves to a dead-letter queue (DLQ) Kafka topic. A separate DLQ consumer alerts ops and stores failure reason in the notifications table. Permanent failures (invalid token, unsubscribed) are not retried — detected via 410/403 provider responses.
Scheduled Notifications
Scheduled notifications are stored in a PostgreSQL table with scheduled_at timestamp. A cron worker runs every minute, queries WHERE scheduled_at <= NOW() AND status = 'pending', and publishes to Kafka. Uses row-level locking (SELECT FOR UPDATE SKIP LOCKED) to safely distribute work across multiple cron instances.
User Rate Limiting
Prevent notification fatigue: Redis counter per user per channel per hour. Worker checks INCR user:{id}:push:hour before sending — if count > 5, drop to low-priority queue or skip marketing notifications. Transactional (OTP, order updates) always bypass rate limits.
Section 08

Scaling & Bottlenecks

Kafka Partitioning

Partition by userId → ordered delivery per user. 100 partitions × 10 consumers per topic = 1000 parallel deliveries. Add partitions as throughput grows.

Provider-Specific Workers

Separate worker pools for APNS, FCM, SendGrid, Twilio — each has different rate limits (FCM: 600K/sec, Twilio: much lower). Scale each pool independently.

Stateless Notification Service

The notification service only validates preferences and publishes to Kafka — no state. Scales horizontally behind a load balancer with no session affinity needed.

Delivery Receipt Tracking

FCM/APNS return delivery receipts asynchronously. Worker publishes receipt events to a Kafka "receipts" topic → async consumer updates Cassandra status. Never block on DB writes in the hot path.

Device Token Cleanup

On 410 Gone from APNS/FCM (invalid token), worker publishes to "token-cleanup" topic. Async consumer deletes from device_tokens table. Prevents wasted sends to uninstalled apps.

Preference Cache

User preferences are read on every notification send. Cache in Redis with TTL 1 hour to avoid PostgreSQL hits. Invalidate on preference update.

Section 09

Self-Check

Test your understanding. Click to reveal answers.

How do you prevent notification spam — 100 notifications to one user in an hour? Rate Limiting +

Three layers: (1) Per-user sliding window counter in Redis: key 'rl:{userId}:{hour}', max 10 notifications/hour (transactional exempt — 2FA, password reset bypass this). (2) Per-notification-type deduplication: idempotency key = hash(userId + eventType + entityId + day). Store in Redis with 24h TTL — same event on same day sent only once. (3) Digest batching for low-priority: hold 'new follower' notifications and batch them into '20 new followers today' daily digest. Kafka consumer buffers low-priority messages and flushes at scheduled times.

How do you handle a provider (SES, FCM) being down? Resilience +

Retry with exponential backoff: failure → retry 1s, 2s, 4s, 8s, 16s (5 retries). After 5 failures: move to a dead letter queue (DLQ) for human inspection. For critical notifications (2FA): maintain a secondary provider (SendGrid as backup if SES is down, AWS SNS as backup if FCM fails). Both providers have different carrier/infrastructure relationships. Circuit breaker: if >10% of requests to a provider fail in a 1-minute window, stop sending, switch to backup, alert on-call. Store all delivery attempts in a notifications_log table for audit.

Email vs SMS vs Push — how do you choose channels? Channel Strategy +

Decision factors: (1) User explicit preference — they opted in or out in settings. (2) Notification urgency: 2FA → SMS (high trust, time-critical); order updates → email (rich content, archivable); real-time alerts → push (instant, no cost). (3) User online status: if user is currently in the app, show an in-app banner instead of push (avoid duplicate). (4) Cost: push = $0. Email ≈ $0.0001/msg. SMS ≈ $0.0075/msg. A company sending 10M SMS/day pays $75K/day — rate-limit aggressively. (5) Opt-in rate: push requires notification permission (~60% opt-in on iOS). Email is universal.

How do you handle 50 languages and different date formats? i18n +

Template system with locale variants: template key = 'order_shipped' + locale 'es-MX'. Template DB stores one row per (template_name, locale). Missing locale falls back to language default, then to 'en-US'. User's locale stored in profile. Date/time: always store in UTC in DB. Format in templates using the user's timezone: 'Jan 15' in en-US, '15 jan.' in fr-FR, '15. Jan.' in de-DE. ICU MessageFormat handles plurals which vary per language: '1 item' vs '2 items' in English, but Russian has 4 plural forms.

A critical 2FA SMS fails to deliver. What's the fallback? Critical Path +

2FA is blocking — user cannot log in. SLA: <10 seconds. Flow: (1) Attempt Twilio primary. (2) No delivery receipt within 5s → simultaneously try AWS SNS (second provider). (3) On both failures: expose 'Resend code' button to user (rate-limited: once per 30s). (4) Escalate on-call alert if SMS failure rate > 0.1% in any 5-minute window. (5) TOTP backup: user should have been asked to set up an authenticator app during 2FA enrollment. Never rely solely on SMS for auth. Twilio and SNS use different carrier agreements — a carrier outage affecting one typically doesn't affect the other.

Section 10

Interview Tips

🔌
Channel abstraction: Design the worker interface so adding a new channel (WhatsApp, Slack) requires only a new consumer class implementing send(notification) — no changes to the core pipeline. This is the Open/Closed principle applied to system design.
⚠️
Preferences at fan-out time: Always check user preferences inside the worker, not just at API ingestion. A preference might change between when the notification was enqueued and when it's delivered — checking late is safer.
🔑
Token management: Device tokens expire when users reinstall the app. APNS returns 410 Gone for stale tokens. Your system must handle this to avoid wasting sends and avoid hitting provider rate limits with dead tokens.
🔁
Idempotency key: This is the interviewer's follow-up question — "what if Kafka redelivers a message?" Have a clear answer: idempotency_key checked in Redis before sending prevents duplicates even under failure and restart scenarios.
📊
Metrics to monitor: Delivery success rate per channel, p99 latency from send to delivery, DLQ depth (alert if growing), per-provider error rates, token invalidation rate. These show system health and provider issues.
