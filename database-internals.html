1. What Is It

Database internals refers to the implementation machinery underneath SQL — how data is physically stored on disk, how indexes accelerate queries, how transactions remain consistent under concurrent access, and how the database survives crashes. For a senior engineer, understanding this layer is what separates someone who can use a database from someone who can reason about it under load.

2. The Problem It Solves

Disk I/O is 100,000× slower than RAM access. A naive database that stored every row in a flat file would require scanning the entire file for every query — O(N) per read. Databases solve this with layered machinery:

The core tension: writes want sequential I/O (fast on disk); reads want random access to any record (fast with indexes). Every storage engine design is a tradeoff between these two imperatives.

On top of storage, concurrent users create a second problem: how do you prevent one transaction from reading half-written data from another? MVCC (Multi-Version Concurrency Control) solves this by keeping multiple versions of a row, so readers always see a consistent snapshot without blocking writers.

Finally, crashes happen. WAL (Write-Ahead Log) ensures durability: all changes are recorded to a sequential log before being applied to data pages. On restart, the database replays the log to recover.

3. How It Works

B-tree and LSM tree are the two dominant storage engine architectures. They make opposite tradeoffs.

MVCC — Snapshot Isolation Without Locks

When a transaction starts, PostgreSQL assigns it a transaction ID (xid). Every row stores the xmin (xid that created it) and xmax (xid that deleted it). A reader only sees rows where xmin is committed and less than its own xid, and xmax is either null or not yet committed. This means readers never need locks — they simply ignore versions written after their snapshot time.

Write-Ahead Log (WAL)

Before any data page is modified, the change is appended to the WAL (a sequential, append-only file). On crash, PostgreSQL replays the WAL from the last checkpoint to redo committed changes, then undoes uncommitted ones. Sequential log writes are far faster than random page writes, which is why WAL enables both durability and high write throughput.

4. Storage Engines, Indexing & Transactions

Storage Engines

B-Tree
InnoDB, PostgreSQL
Balanced tree of fixed-size pages (4–16KB). In-place updates — good for reads, random writes cause expensive page splits. Self-balancing: O(log N) for reads and writes. Leaf nodes in a B+ tree form a linked list for range scans.
LSM Tree
Cassandra, RocksDB, LevelDB
Writes go to in-memory MemTable (red-black tree) + WAL. Periodically flushed to sorted SSTables on disk. Background compaction merges levels. Sequential writes = very fast. Reads may check multiple SSTables (use bloom filters to skip). High write amplification on compaction.
Heap Files
Simple row storage
Unordered rows written sequentially. Fast inserts, slow reads (full scan without index). PostgreSQL uses heap files + separate B-tree index files. Updates write new version to heap; VACUUM cleans dead tuples.

Index Types

B-Tree Index
Most common — range + equality
Supports =, <, >, BETWEEN, LIKE 'prefix%', ORDER BY. Default index type. Maintained on every write — adds overhead to INSERT/UPDATE/DELETE.
Hash Index
Equality only — O(1)
Only useful for exact equality (=). No range scans. Used in Redis, and InnoDB adaptive hash index (automatic, in-memory). PostgreSQL supports explicit hash indexes but rarely used.
Composite Index
Column order matters critically
INDEX (a, b, c) can satisfy queries on (a), (a,b), (a,b,c) — but NOT (b) or (c) alone. Leftmost prefix rule. Think of it as a phone book sorted by last name then first name: you can look up "Smith" or "Smith, John" but not "John" alone.
Covering Index
Eliminates table lookup entirely
If the index contains ALL columns the query needs (SELECT + WHERE + ORDER BY), the DB can answer the query from the index alone — no heap/row access. Can be 5–10× faster for high-selectivity queries. Create with INCLUDE columns in PostgreSQL.

Query Optimization — Join Algorithms

Nested Loop Join
Best for small outer tables
For each row in outer table, scan inner table. O(N×M). Good when outer is small and inner has an index. Bad for large unindexed joins.
Hash Join
Large equality joins
Build hash table from smaller relation. Probe with larger relation. O(N+M). Requires equality condition. Memory-intensive — spills to disk if table doesn't fit.
Merge Join
Pre-sorted inputs
Sort both sides, then merge. O(N log N + M log M). Efficient when both sides are already sorted (e.g., indexed columns). Optimal for range joins.

6. Code Examples

-- Range scan on B-tree index: efficient (index range scan)
EXPLAIN ANALYZE
SELECT id, name, email
FROM users
WHERE created_at BETWEEN '2024-01-01' AND '2024-06-01';
-- → Index Scan using idx_users_created_at
--    Index Cond: (created_at >= '2024-01-01' AND created_at <= '2024-06-01')
--    Rows: 45231  Actual Rows: 44890  Time: 23ms

-- Same query on Cassandra (LSM) — must specify partition key!
-- Cassandra stores data in SSTables partitioned by partition key.
-- Range scan WITHOUT partition key = full cluster scan (avoid!)
SELECT * FROM user_events
WHERE user_id = 12345           -- partition key (required)
  AND event_time > '2024-01-01'  -- clustering key (range OK)
LIMIT 100;

-- LSM bloom filter check: before reading an SSTable,
-- check the bloom filter. If it says "no", skip the file entirely.
-- Reduces read amplification from O(levels) to ~O(1) for non-existent keys.

-- Session 1: begin transaction, don't commit yet
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- PostgreSQL creates new row version: xmin=txn_101, xmax=NULL
-- Old version: xmin=txn_50, xmax=txn_101 (visible to concurrent readers)

-- Session 2: concurrent read — sees OLD value due to MVCC
BEGIN;
SELECT balance FROM accounts WHERE id = 1;
-- Returns original balance — Session 2's snapshot predates txn_101
-- No lock needed! Readers never block writers in MVCC.

-- Session 1 commits
COMMIT; -- txn_101 marked as committed in pg_xact

-- Session 2 still sees old value (Repeatable Read) or new (Read Committed)
SELECT balance FROM accounts WHERE id = 1;
-- Read Committed: sees 900 (new committed value)
-- Repeatable Read: sees 1000 (snapshot from txn start)

-- Dead tuples accumulate from MVCC — VACUUM reclaims them
VACUUM ANALYZE accounts;
-- Removes versions where xmax is committed and no live txn needs them

-- Composite index: LEFT-PREFIX rule matters
CREATE INDEX idx_orders_composite
  ON orders (user_id, status, created_at);

-- Uses index (leftmost prefix matches)
SELECT * FROM orders WHERE user_id = 42;
SELECT * FROM orders WHERE user_id = 42 AND status = 'PAID';
SELECT * FROM orders WHERE user_id = 42 AND status = 'PAID' AND created_at > '2024-01';

-- Does NOT use index (missing leftmost column)
SELECT * FROM orders WHERE status = 'PAID'; -- full table scan!

-- Covering index: no heap access needed (index-only scan)
CREATE INDEX idx_users_covering
  ON users (email)
  INCLUDE (id, name, created_at);  -- PostgreSQL 11+ INCLUDE syntax

-- This query hits ONLY the index:
SELECT id, name, created_at FROM users WHERE email = 'user@example.com';
-- EXPLAIN shows: Index Only Scan using idx_users_covering
-- Heap Fetches: 0  ← never touches the actual table rows!

-- Index selectivity: avoid indexing low-cardinality columns
CREATE INDEX idx_bad ON orders (status);   -- BAD: 4 values, low selectivity
CREATE INDEX idx_good ON orders (order_id); -- GOOD: unique, high selectivity

// WAL crash recovery — pseudocode showing PostgreSQL recovery logic

class WALRecovery {

  void recover() {
    // 1. Find last checkpoint (written periodically, ~every 5 min)
    Checkpoint checkpoint = readLastCheckpoint();
    long replayFrom = checkpoint.walPosition;

    Set<Long> committedTxns = new HashSet<>();
    Set<Long> uncommittedTxns = new HashSet<>();

    // 2. REDO PASS — replay all WAL records from checkpoint forward
    for (WALRecord record : readWAL(replayFrom)) {
      if (record.type == COMMIT) {
        committedTxns.add(record.txnId);
      } else if (record.type == ABORT) {
        uncommittedTxns.remove(record.txnId);
      } else {
        // Apply the data change to the page
        applyChange(record);
        uncommittedTxns.add(record.txnId);
      }
    }

    // 3. UNDO PASS — roll back uncommitted transactions
    for (long txnId : uncommittedTxns) {
      // Walk undo log backwards, reverse each change
      undoTransaction(txnId);
    }

    // DB is now in consistent state — open for connections
  }
}

// WAL write flow (called on every DML statement):
// 1. Generate WAL record (old + new values)
// 2. Append to WAL buffer in memory
// 3. On COMMIT: fsync WAL buffer to disk (synchronous_commit=on)
// 4. Only THEN acknowledge commit to client
// WAL is ALWAYS written before data pages — that's the "write-ahead" guarantee

7. Real-World Usage

PostgreSQL
B-Tree + MVCC + WAL
Heap storage + B-tree indexes. MVCC with version chains in heap. WAL for crash recovery. VACUUM to reclaim dead tuples. Extensible index types: GIN (full-text), GiST (geometry), BRIN (time-series ranges).
MySQL InnoDB
Clustered B-Tree + Doublewrite
Primary key IS the clustered index — rows stored in PK order. Doublewrite buffer prevents torn page writes. Redo log (WAL) + undo log. Gap locks to prevent phantom reads at Repeatable Read level.
Cassandra
LSM + SSTables + Compaction
Pure LSM tree. All writes go to MemTable + CommitLog (WAL). Bloom filters per SSTable reduce read amplification. Compaction strategies: STCS (write-heavy), LCS (read-heavy, space-efficient), TWCS (time-series).
RocksDB
LSM engine — embedded
Used as embedded storage in Kafka, TiKV, MyRocks (MySQL + RocksDB). Tunable compaction, block cache, bloom filter per level. Write-optimized — ideal for event logs and KV stores.
MongoDB WiredTiger
B-Tree + MVCC
B-tree storage engine with document-level concurrency. MVCC snapshot isolation. Journal (WAL) for durability. Compression (snappy/zstd) on data blocks by default.

8. Interview Tips

💡
"Why is Cassandra fast for writes?" LSM tree — writes go to in-memory MemTable and sequential WAL. No random disk I/O on the write path. Compaction happens in the background, not blocking writes.
💡
"Why doesn't a SELECT block an UPDATE in PostgreSQL?" MVCC — the reader sees the row version from its snapshot timestamp. The writer creates a new version. They operate on different row versions simultaneously.
💡
"What causes index bloat in PostgreSQL?" MVCC dead tuples. When a row is updated, the old version isn't immediately deleted — it stays in the heap until VACUUM runs. Indexes still point to both old and new versions. autovacuum handles this, but high-write tables may need manual tuning.
💡
"When NOT to add an index?" (1) Low-cardinality columns (e.g., boolean, status with 3 values) — the planner may prefer a seq scan. (2) Write-heavy tables — every index costs an additional write on INSERT/UPDATE/DELETE. (3) Small tables — seq scan is faster than index overhead.
💡
"What is write amplification and why does it matter?" LSM compaction rewrites data multiple times as SSTables move through levels. A single user write can cause 10–30 actual disk writes. On SSDs with finite write cycles, this is a real operational concern. RocksDB tuning (compaction strategy, level sizes) significantly affects this.
💡
"Explain the N+1 query problem." Loading 100 orders then firing a separate SELECT for each order's user = 101 queries. Fix: JOIN in one query, or batch load (SELECT ... WHERE user_id IN (...)). ORMs are notorious for this — always check generated SQL in production.
