07

Key Design Decisions

The core challenge: how do we generate globally unique, short, collision-free codes at 1200 writes/sec?

Option 1
MD5 Hash + Truncate
MD5(longUrl) → take first 8 chars. Fast. But collision-prone: two different URLs could produce the same prefix. Requires collision detection + retry loop on every write.
Option 2 — Recommended
Snowflake ID → Base62
Generate a unique 64-bit Snowflake ID → encode to Base62 string. Globally unique by construction. 8 chars covers 218 trillion URLs. No collision possible.
Option 3
Pre-generated Code Pool
Pre-generate short codes offline, store in an available_codes table. Pop one on each write request. Zero collision risk. Slow startup; complex ops.

private static final String BASE62 = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";

public String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(BASE62.charAt((int) (id % 62)));
        id /= 62;
    }
    return sb.reverse().toString(); // e.g. 1234567 → "5ban"
}

// Snowflake ID: timestamp(41) | machineId(10) | sequence(12)
public long nextId() {
    long timestamp = System.currentTimeMillis() - EPOCH;
    return (timestamp << 22) | (machineId << 12) | sequenceNum++;
}
// Redirect handler — optimized for <10ms p99
public ResponseEntity<?> redirect(String shortCode) {
    // 1. Check Redis cache first (~0.5ms)
    String longUrl = redis.get("url:" + shortCode);

    if (longUrl == null) {
        // 2. Cache miss → query read replica (~2–5ms)
        longUrl = urlRepo.findLongUrl(shortCode);
        if (longUrl == null) {
            return ResponseEntity.notFound().build();
        }
        // 3. Write-through: populate cache
        redis.setex("url:" + shortCode, 86400, longUrl); // TTL 1 day
    }

    // 4. Fire-and-forget analytics — does NOT block response
    eventBus.publishAsync(new ClickEvent(shortCode, request));

    return ResponseEntity.status(302)
        .header("Location", longUrl).build();
}
08

Scaling & Bottlenecks

⚡
Redis Cache — Handle 120K reads/sec
Cache shortCode→longUrl mappings with 24h TTL. 80% of traffic hits hot URLs. With 25GB RAM, cache 50M entries. Redis cluster for HA. Use SETNX to prevent cache stampede.
🌐
CDN for 301 Redirects — Zero-latency hot URLs
301 responses are cacheable by browsers and CDN edge nodes. Popular short links (e.g., in viral tweets) get resolved at CDN edge with ~0ms server involvement.
🗄️
Read Replicas — Scale DB reads
Redirect lookups (cache misses) go to read replicas. Writes (URL creation) go to primary. 3–5 replicas handle 120K QPS with 95%+ cache hit rate meaning only ~6K QPS to DB.
🔀
Horizontal Sharding — Beyond 1B URLs
Shard by hash(shortCode) if URL count exceeds single-node capacity. Consistent hashing ensures even distribution and minimal resharding on node addition.
🧹
Expiration Cleanup — Lazy + Cron
Lazy deletion: check expires_at on every redirect read. Background cron: DELETE FROM urls WHERE expires_at < NOW() — run nightly during low traffic. Index on expires_at.
09

Self Check

Answer before revealing. These are real interview questions.

Why do we need Redis? Can't we just query PostgreSQL for every redirect?Most Asked +

We have 120,000 reads/sec. Even with 5 read replicas, that's 24,000 QPS per replica — right at the PostgreSQL limit, with no headroom for bursts or slow queries.

Redis handles 100,000+ ops/sec per node at ~0.5ms latency (vs PostgreSQL's 2–5ms for an indexed lookup). By caching shortCode → longUrl with 24h TTL, 80–90% of reads never reach the database. The DB only sees cache misses — roughly 6–12K QPS — leaving plenty of headroom.

The other reason is the 10ms p99 requirement. Cache-miss path: ~0.5ms Redis + ~3ms DB = ~5ms. Cache-hit path: ~0.5ms total. Without Redis, every request costs 3–5ms just for the DB, and bursty load pushes you past 10ms easily.

What happens when Redis goes down?Failure Mode +

Redirects degrade but don't fail. All 120K QPS falls through to the DB read replicas. Latency jumps from ~1ms to ~5ms. Under high load, DB becomes the bottleneck.

Mitigations: (1) Redis Cluster or Sentinel for automatic failover — a replica becomes primary within seconds. (2) Circuit breaker on the cache client: if Redis times out, skip to DB immediately rather than waiting per-request. (3) On restart, the cache is cold — implement a gradual warm-up strategy: pre-populate hot keys from the DB on startup, or add a short-circuit randomised sleep to spread DB load ("thundering herd" prevention).

Why use Snowflake ID + Base62 instead of MD5-hashing the URL?Core Algorithm +

With MD5 truncation: shortCode = MD5(longUrl)[0:8], collisions are a real problem. With 500M URLs and only 4 billion possible 8-hex-char codes (16⁸), you hit ~6% collision probability by the birthday paradox. Every collision requires a retry — a different prefix length, append a counter, etc. This adds latency and complexity to every write.

Snowflake IDs are unique by construction: timestamp(41 bits) | machineId(10 bits) | sequence(12 bits). Two requests in the same millisecond on the same machine get different sequence numbers. Two machines get different machine IDs. Globally unique, no coordination needed, no collisions. Base62 encoding then maps these to human-readable strings (62⁸ ≈ 218 trillion capacity).

Bonus: Snowflake IDs are monotonically increasing — B-tree index inserts are always at the right end, avoiding the "random write" fragmentation problem UUID primary keys cause.

Two users request the same custom alias simultaneously. What happens?Race Condition +

Both requests hit separate stateless API server instances. Both try to INSERT INTO urls (short_code, ...) VALUES ('my-brand', ...) at roughly the same time.

PostgreSQL's UNIQUE constraint on short_code is the serialization point. Postgres acquires a row-level lock on the unique index page. One INSERT wins. The other gets a constraint violation (error code 23505). The API server catches this specific exception and returns HTTP 409 Conflict with "Alias 'my-brand' is already taken."

No application-level locking or distributed lock needed — the database handles it atomically. This is one reason to prefer PostgreSQL over a NoSQL store that lacks atomic unique guarantees.

How do you count 120,000 clicks/sec without thrashing the database?Scale Challenge +

Naive: UPDATE urls SET click_count = click_count + 1 WHERE short_code = 'abc'. At 120K QPS for a hot link, this causes constant row-level lock contention. The single row becomes a serialization bottleneck — updates queue behind each other and the DB is overwhelmed.

Solution 1 — Redis INCR: INCR click:abc123 is an atomic O(1) in-memory operation with no locking. Redis handles 100K ops/sec. A background job flushes counts to PostgreSQL every 5 minutes: UPDATE urls SET click_count = click_count + ? WHERE short_code = ?. You lose up to 5 min of data if Redis crashes — acceptable for analytics.

Solution 2 — Kafka event stream: On each redirect, publish a ClickEvent to Kafka asynchronously (fire-and-forget, doesn't block the response). A consumer batch-processes into ClickHouse or a data warehouse. This supports rich analytics (country, device, referrer, time series) completely decoupled from the hot path. This is what production analytics-focused URL shorteners use.
