02

Clarifying Questions

Ask these in an interview — it shows you understand the problem space, not just the surface.

  • ? How is "nearest" defined? Nearest by floor first, then by spot number? Or straight-line distance? Affects the spot-search algorithm.
  • ? How is pricing structured? Flat rate per hour? First hour free? Differential by vehicle type? Defines the pricing strategy interface.
  • ? What payment methods? Cash, card, UPI? This determines whether we need a payment gateway abstraction.
  • ? Multiple vehicles per spot? Can two motorcycles share a large spot? Simplest answer: no, 1 vehicle per spot.
  • ? Reserved spots? Monthly passes, handicapped spots? Adds a reservation concept to the model.
  • ? Persistence required? In-memory is fine for an interview; note where you'd add a DB layer if asked.

05

Key Flows

Walk through these step-by-step in your interview — it shows you understand data flow, not just class diagrams.

Entry Flow

Vehicle arrives at entry gate
System reads license plate. Creates a Vehicle object with type.
Search for nearest available spot
ParkingLot iterates floors in order. Each floor scans spots for one where isAvailableFor(vehicle) returns true.
Reserve the spot (thread-safe)
Spot is marked occupied atomically. Vehicle reference stored on spot.
Issue ticket
ParkingTicket created with UUID, entryTime=now, spot reference. Stored in activeTickets map. Printed/displayed to user.
No spot available?
Throw ParkingLotFullException. Gate stays closed.

Exit Flow

Vehicle presents ticket
Ticket ID scanned. Lookup in activeTickets map to retrieve ticket.
Calculate fee
Set exitTime = now. Delegate to pricingStrategy.calculateFee(ticket).
Process payment
Create Payment object. Call paymentProcessor.process(amount). Mark ticket PAID on success.
Free the spot
spot.unpark() clears vehicle reference and sets isOccupied=false. Remove from activeTickets. Gate opens.
Payment fails?
Spot stays occupied. Ticket remains active. Ticket marked FAILED.

Code

Key implementations — focus on these in the interview. Don't write getters/setters, focus on the logic.

Enums & Vehicle hierarchy

enum VehicleType { MOTORCYCLE, CAR, TRUCK }
enum SpotSize    { SMALL, MEDIUM, LARGE }
enum PaymentStatus { PENDING, PAID, FAILED }
abstract class Vehicle {
    protected final String licensePlate;
    protected final VehicleType type;

    public abstract boolean canFitIn(SpotSize size);
}

class Motorcycle extends Vehicle {
    public boolean canFitIn(SpotSize size) { return true; }
}

class Car extends Vehicle {
    public boolean canFitIn(SpotSize size) {
        return size == SpotSize.MEDIUM || size == SpotSize.LARGE;
    }
}

class Truck extends Vehicle {
    public boolean canFitIn(SpotSize size) {
        return size == SpotSize.LARGE;
    }
}

ParkingSpot — thread-safe park/unpark

class ParkingSpot {
    private final String spotId;
    private final SpotSize size;
    private volatile boolean isOccupied = false;
    private Vehicle vehicle;

    public boolean isAvailableFor(Vehicle v) {
        return !isOccupied && v.canFitIn(size);
    }
}
// synchronized so two threads can't double-book the same spot
public synchronized void park(Vehicle v) {
    if (isOccupied) throw new SpotAlreadyOccupiedException();
    isOccupied = true;
    vehicle = v;
}

public synchronized void unpark() {
    isOccupied = false;
    vehicle = null;
}

ParkingLot — Singleton + entry/exit

class ParkingLot {
    private static volatile ParkingLot instance;
    private final List<ParkingFloor> floors;
    private final Map<String, ParkingTicket> activeTickets;
    private PricingStrategy pricingStrategy;

    private ParkingLot() {
        floors = new ArrayList<>();
        activeTickets = new ConcurrentHashMap<>();
    }

    // Double-checked locking
    public static ParkingLot getInstance() {
        if (instance == null) {
            synchronized (ParkingLot.class) {
                if (instance == null) instance = new ParkingLot();
            }
        }
        return instance;
    }
}
public ParkingTicket parkVehicle(Vehicle vehicle) {
    for (ParkingFloor floor : floors) {
        Optional<ParkingSpot> spot = floor.findSpot(vehicle);
        if (spot.isPresent()) {
            spot.get().park(vehicle);    // atomic
            ParkingTicket ticket = new ParkingTicket(
                UUID.randomUUID().toString(),
                vehicle,
                spot.get(),
                LocalDateTime.now()
            );
            activeTickets.put(ticket.getId(), ticket);
            return ticket;
        }
    }
    throw new ParkingLotFullException("No spot available for " + vehicle.getType());
}
public Payment unparkVehicle(String ticketId) {
    ParkingTicket ticket = activeTickets.get(ticketId);
    if (ticket == null) throw new InvalidTicketException();

    ticket.setExitTime(LocalDateTime.now());
    double fee = pricingStrategy.calculateFee(ticket);
    ticket.setFee(fee);

    Payment payment = new Payment(ticket, fee, paymentProcessor);
    if (payment.process()) {
        ticket.setStatus(PaymentStatus.PAID);
        ticket.getSpot().unpark();
        activeTickets.remove(ticketId);
    } else {
        ticket.setStatus(PaymentStatus.FAILED);
    }
    return payment;
}

Pricing Strategy

interface PricingStrategy {
    double calculateFee(ParkingTicket ticket);
}
class HourlyPricingStrategy implements PricingStrategy {
    private final Map<VehicleType, Double> rates = Map.of(
        VehicleType.MOTORCYCLE, 20.0,
        VehicleType.CAR,        40.0,
        VehicleType.TRUCK,      80.0
    );

    public double calculateFee(ParkingTicket ticket) {
        long hours = ChronoUnit.HOURS.between(
            ticket.getEntryTime(), ticket.getExitTime()
        );
        double rate = rates.get(ticket.getVehicle().getType());
        return Math.max(1, hours) * rate; // minimum 1 hour
    }
}
class TieredPricingStrategy implements PricingStrategy {
    public double calculateFee(ParkingTicket ticket) {
        long mins = ChronoUnit.MINUTES.between(
            ticket.getEntryTime(), ticket.getExitTime()
        );
        if (mins <= 30)  return 0;    // grace period
        if (mins <= 60)  return 30;
        if (mins <= 120) return 50;
        return 50 + ((mins - 120) / 60.0) * 20; // ₹20/hr after 2h
    }
}

Design Patterns Used

Name these explicitly in the interview — interviewers want to hear the vocabulary.

Creational
Singleton
ParkingLot is a single shared instance. Use double-checked locking with volatile for thread safety.
Behavioral
Strategy
PricingStrategy interface lets you swap HourlyPricing vs TieredPricing without touching the lot code.
Structural
Composite
ParkingLot → ParkingFloor → ParkingSpot forms a tree. You can query availability at any level uniformly.
Creational
Factory
VehicleFactory creates the right Vehicle subclass from a type enum, keeping creation logic in one place.
Behavioral
Template Method
Abstract Vehicle defines the contract; each subclass implements canFitIn() — the algorithm skeleton is fixed, the step varies.
Structural
Facade
ParkingLot is a facade: clients call parkVehicle/unparkVehicle without knowing about floors, spots, or pricing internals.

Concurrency & Thread Safety

Most candidates skip this. Bring it up — it shows senior-level thinking.

The core race condition: two threads find the same spot available simultaneously and both try to park. Without synchronization, both succeed — double booking.
RiskLocationSeverityFix
Double-book a spot ParkingSpot.park() HIGH synchronized on the spot object
Stale singleton instance ParkingLot.getInstance() HIGH volatile + double-checked locking
ConcurrentModification on activeTickets ParkingLot MED Use ConcurrentHashMap
Floor-level spot search races ParkingFloor.findSpot() MED Lock at spot level, not floor level — better throughput
Fee calculation on stale exitTime ParkingTicket LOW Set exitTime once atomically before calculating

Advanced: ReentrantLock per spot (better throughput)

class ParkingSpot {
    private final ReentrantLock lock = new ReentrantLock();

    public boolean tryPark(Vehicle v) {
        if (lock.tryLock()) {         // non-blocking attempt
            try {
                if (!isOccupied && v.canFitIn(size)) {
                    isOccupied = true;
                    vehicle = v;
                    return true;
                }
            } finally {
                lock.unlock();
            }
        }
        return false;
    }
}

→ tryLock() avoids blocking the whole floor while one spot is being parked. Each spot locks independently = O(1) contention per spot.

09

Interview Tips

What separates a good answer from a great one.

1
Start with requirements, not code. Spend 3-5 minutes asking clarifying questions. Interviewers score this as signal that you don't jump to implementation before you understand the problem.
2
Name your design patterns out loud. Say "I'm using the Strategy pattern here so we can swap pricing without touching the Lot" — not just "I made an interface." Vocabulary matters.
3
Lead with classes, not methods. Draw the entities first. Interviewers want to see that you can identify the nouns (domain model) before the verbs (behavior).
4
Bring up concurrency unprompted. Say "Since we can have concurrent entries, ParkingSpot.park() must be synchronized." This differentiates you — most candidates ignore threading entirely.
5
canFitIn() on Vehicle, not on Spot. A common mistake is putting the fit logic on ParkingSpot. But the vehicle knows its own constraints — putting it there keeps Spot dumb and extensible to new vehicle types.
6
Mention extensibility explicitly. "If we add a new vehicle type — say an E-Rickshaw — we just add a subclass and implement canFitIn(). No changes to ParkingLot or ParkingSpot." This is the Open/Closed Principle — say it.
7
ParkingLot is a Facade, not a God class. If you end up putting all logic in ParkingLot, flag it: "In production I'd split search, pricing, and payment into separate services." Shows architectural awareness.
8
The ticket is the source of truth. EntryTime lives on the ticket, not on the spot. The spot only knows if it's occupied — the ticket carries all audit information needed for billing.
