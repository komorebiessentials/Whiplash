// Netflix transcoding pipeline (simplified)
// A single 2-hour title → ~1,200 output files
class TranscodePipeline {
    void processTitle(String titleId) {
        // 1. Download source from S3 (original upload)
        File source = s3.get("originals/" + titleId);

        // 2. Split into 2-second segments (chunk)
        List<Segment> segs = split(source, 2); // ~3600 segs for 2hr

        // 3. Transcode each segment × 5 resolutions × 2 codecs in parallel
        for (Segment seg : segs) {
            kafka.publish("transcode-jobs", new Job(seg, RESOLUTIONS, CODECS));
        }
        // Workers: 3600 segs × 5 res × 2 codecs = 36,000 parallel jobs
    }
}
enum RESOLUTIONS { 240p, 480p, 720p, 1080p, 4K }
enum CODECS       { H264, H265 }  // H265 = 40% smaller, older devices need H264

// Cache key = (title_id, quality_level, chunk_number)
// Popular content pre-positioned to edges nightly (off-peak)

String getChunk(Request req) {
    String cacheKey = req.titleId + ":" + req.quality + ":" + req.chunkNum;
    byte[] chunk = edgeCache.get(cacheKey);
    if (chunk != null) return chunk; // ~98% cache hit rate

    // Cache miss: fetch from S3, cache for 7 days
    chunk = s3.get("segments/" + cacheKey);
    edgeCache.set(cacheKey, chunk, Duration.ofDays(7));
    return chunk;
}
// 184 Tbps × 98% CDN hit rate = 3.7 Tbps origin egress (manageable)

Self Check

Answer before revealing. These are real interview questions.

How does adaptive bitrate streaming (ABR) work? What protocol does Netflix use?Core Concept+

ABR works by encoding the same video at multiple quality levels and splitting each into short segments (2-4 seconds). The client player continuously measures download throughput and switches the quality level it requests next. If bandwidth is high, it upgrades; if bandwidth drops, it downgrades (trading quality for no buffering).

Netflix uses DASH (Dynamic Adaptive Streaming over HTTP) and HLS (for Apple devices). The master manifest (m3u8/mpd file) lists all available quality variants with their bandwidths. Client picks a variant; variant playlist lists individual segment URLs. All served over plain HTTP from CDN.

How does Netflix transcode a new movie so fast? How many output files does one title produce?Transcoding+

Source video is uploaded to S3, which triggers a Kafka event → transcoding workers. The pipeline: (1) Split 2hr movie into 2-second segments (~3600 segments). (2) Each segment is transcoded independently and in parallel across many machines. (3) For 5 resolutions × 2 codecs = 10 variants per segment: 3600 × 10 = 36,000 output files. Plus audio tracks, subtitle tracks, thumbnails.

Netflix calls this "parallel encoding" — a single title generates ~1,200 files total. Netflix also uses per-shot quality optimization (Encode Optimization): dark scenes get more bits, action scenes get more bits, static scenes get fewer. This gives better quality at the same bandwidth than naive encoding.

What is Open Connect? Why does Netflix operate its own CDN?CDN+

Open Connect is Netflix's proprietary CDN — hardware servers (called OCAs, Open Connect Appliances) that Netflix physically places inside ISP data centers around the world, for free. ISPs install them because it dramatically reduces their peering costs (Netflix is ~15% of global downstream traffic).

Why not use Akamai/CloudFront? (1) Economics: at Netflix's scale, operating own CDN is cheaper. (2) Latency: OCA inside your ISP = sub-5ms to your router. (3) Cache hit rate: Netflix pre-positions popular content nightly at each OCA based on regional viewing patterns. ~98% cache hit rate means origin servers barely receive traffic.

How does Netflix handle 20M watch progress writes per second?High Write+

Every 10 seconds while watching, the client sends a progress update. 200M active sessions / 10s = 20M writes/sec — far too much for a single DB. Solution: (1) Client writes to a stateless API server. (2) API server writes to Redis (in-memory, ~0.1ms). (3) A background job (or Kafka consumer) asynchronously flushes Redis to Cassandra every 30s. (4) On app open, read progress from Cassandra (source of truth).

Cassandra handles this well: partition key = user_id, clustering by title_id. Write path is O(1). The Redis buffer absorbs the spike — Cassandra only sees ~30x fewer writes after batching.

How does Netflix's recommendation system work at a high level?ML System+

Recommendations are NOT computed in real-time — they're pre-computed offline and served from cache. The pipeline: (1) Viewing history, ratings, and search events stream to Kafka. (2) Flink/Spark jobs aggregate and transform into user feature vectors. (3) Collaborative filtering + content-based models run nightly (or continuously) on the feature matrix. (4) Top-N recommendations per user written to Redis (user:{id}:recommendations). (5) GET /recommendations serves from Redis in sub-millisecond latency.

Key insight: personalized recommendations served this fast (sub-ms) is only possible because they're pre-computed. Real-time ML inference at this scale would be prohibitively expensive.
