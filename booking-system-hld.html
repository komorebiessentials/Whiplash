8. Key Design Decisions

Preventing Double-Booking — The Core Problem
Race condition: Two users request the same listing for Aug 1–5. Both read availability → both see "available" → both try to book → one must fail.

Best approach — UNIQUE Constraint: The UNIQUE(listing_id, date) constraint on the availability table is the key. Use an INSERT-based approach:

BEGIN;
INSERT INTO availability (listing_id, date, status, booking_id)
SELECT $listing_id, d::date, 'held', $booking_id
FROM generate_series($checkin, $checkout - 1, '1 day') d
WHERE NOT EXISTS (
  SELECT 1 FROM availability
  WHERE listing_id=$listing_id AND date=d::date
  AND status IN ('held','booked')
);
-- If any date conflicts: 0 rows inserted → ROLLBACK
COMMIT;
If the INSERT violates the unique constraint or returns 0 rows for any date, the entire transaction rolls back. No explicit locking needed; the DB handles it atomically.
Hold + Confirm Two-Phase Flow
Why two phases? Payment takes 1–3 seconds. If we mark dates as 'booked' immediately, then payment fails — dates are stuck until manual intervention.

Hold phase (15 min TTL):
• Mark availability rows as 'held' with hold_expires_at = now + 15 min
• Return holdId to client; client completes payment form

Confirm phase:
• Verify holdId is valid and not expired
• Charge payment (idempotent with idempotency key)
• On payment success: update availability to 'booked', booking to 'confirmed'
• On payment failure: release hold (set back to 'available')

Hold cleanup: Cron runs every 5 min: UPDATE availability SET status='available' WHERE status='held' AND hold_expires < NOW()
Search Architecture — Consistency vs Performance
Search (Elasticsearch) and booking (PostgreSQL) are completely decoupled.

• Search needs: full-text, geo-distance, faceted filters, sub-200ms → Elasticsearch
• Booking needs: ACID, row-level locking, exact availability → PostgreSQL

Eventual consistency on search: When a booking is confirmed, Kafka event → Elasticsearch index updated async (within 1–2s). Handle at display time: click a listing → real-time check against PostgreSQL.

Pricing engine: base_price × rule multipliers (weekend=1.2, peak_season=1.5, 7day_discount=0.9). Snapshot the pricing at booking time — so cancellation refund is based on what the guest actually paid.

9. Scaling & Bottlenecks

Availability Table (3.65B rows) Partition by listing_id range (hash partition). All date rows for a listing co-located on same shard → booking transaction stays on one node. Alternatively, partition by date (monthly) for easy archival.
Search at 58K QPS Elasticsearch cluster with dedicated search nodes. Cache GET /search responses in Redis for 30s TTL. CDN caches popular search results for city-level queries.
Availability Cache (Redis) Cache per-listing monthly availability as a bitset (1 bit per day = 31 bytes per listing per month). Read from Redis for calendar display. Invalidate on booking. Reduces PostgreSQL read load by 90%.
Kafka Fan-out booking.confirmed → Payment Service, Notification Service, Elasticsearch updater, Analytics pipeline. All async, decoupled, independently scalable.

10. Self-Check

+
Why do you need a hold phase? Why not just book the dates immediately?
Two-Phase
Payment processing takes 1–3 seconds. If you mark dates as 'booked' immediately and then payment fails, the dates are permanently blocked until manual cleanup. The hold phase creates a temporary reservation (15 min TTL) that gives the user time to complete payment. On success → promote to 'booked'. On failure or timeout → automatically release back to 'available'. This gives you a clean rollback path and avoids permanently blocking inventory on payment failures.
+
How does UNIQUE(listing_id, date) prevent double-booking?
Consistency
The constraint makes it physically impossible for two rows to exist with the same listing_id + date in the 'held' or 'booked' state. The INSERT-based booking approach tries to insert one row per date in the requested range. If any row would violate the unique constraint (another request already holds that date), the entire INSERT fails and the transaction rolls back. No application-level locking needed — the database enforces the invariant atomically. This is stronger than SELECT FOR UPDATE because it doesn't require explicit lock acquisition.
+
What happens if the hold expires before the user completes payment?
TTL & Expiry
A cron job runs every 5 minutes: UPDATE availability SET status='available' WHERE status='held' AND hold_expires < NOW(). The dates are released back to 'available' state. Lazy cleanup also happens: if another user tries to book a 'held' date whose TTL has passed, the system checks hold_expires and treats expired holds as available. If the original user tries to confirm after expiry, the confirm endpoint validates that holdId is still valid — if expired, it returns 409 "Hold expired" and the user must start over.
+
Why are search (Elasticsearch) and booking (PostgreSQL) in completely separate systems?
Architecture
They have fundamentally different requirements. Search needs: full-text search, geo-distance queries, faceted filters, relevance ranking — Elasticsearch is purpose-built for this at sub-200ms. Booking needs: ACID guarantees, row-level locking, exact availability, zero double-bookings — PostgreSQL with UNIQUE constraints handles this. Trying to do both in one system means compromising on both. The trade-off is eventual consistency in search (ES may lag PostgreSQL by 1–2s), handled by doing a real-time PostgreSQL check when a user actually clicks a listing.
+
How does a Redis distributed lock differ from SELECT FOR UPDATE in this context?
Locking
SELECT FOR UPDATE is a pessimistic row-level lock in PostgreSQL — it serializes all booking attempts for a listing by making each wait for the previous transaction to commit. For a popular listing during a flash sale, this creates a queue and risks deadlocks if lock order varies. A Redis distributed lock (SETNX on "lock:listing:{id}:{dates}" with 30s TTL) is faster to acquire (in-memory) and allows the system to immediately return "currently being booked, try again" rather than waiting. However, the UNIQUE constraint approach is usually preferred because it's simpler: let the DB enforce the invariant and handle contention at the INSERT level.

11. Interview Tips

🚫
Double-booking is THE question. Don't just say "use a lock." Explain the UNIQUE constraint + INSERT approach — it's atomic, elegant, and shows DB-level thinking rather than application-level synchronization.
⏱️
Explain why you need two-phase booking. "If I mark it booked immediately and payment fails, the dates are stuck forever. The hold-confirm pattern gives us a clean rollback path."
🔀
Separate search from booking. "These have fundamentally different requirements — search optimizes for speed and relevance, booking optimizes for consistency. They should use different databases."
💵
Snapshot pricing at booking time. Store the cancellation policy and pricing breakdown in the booking record. If the host changes prices later, the guest's refund should be based on what they originally paid.
🌎
Geo search nuance. Use PostGIS on PostgreSQL for exact geo queries, but Elasticsearch with geo_distance for search-time filtering. "Within 2km" queries need spatial indexing, not a table scan.
